流水线利用率

NVIDIA® Nsight™ Development Platform, Visual Studio Edition 4.7 用户指南
发送反馈

概述

CUDA 设备的每个流多处理器（SM）都包含许多专门执行特定任务的硬件单元。在芯片层面，这些单元提供执行流水线，warp 调度器将指令分派到这些流水线上。例如，纹理单元负责执行纹理读取并完成纹理过滤；Load/Store 单元负责从内存读取数据或将数据写回内存。理解这些流水线的利用率，并了解它们距离目标设备的峰值性能还有多远，是分析 kernel 执行效率的关键信息；同时也可以帮助识别由于某种类型流水线被过度使用而导致的性能瓶颈。

背景

Kepler GK110 白皮书 和 NVIDIA GeForce GTX 680 白皮书 都描述并对比了最新 CUDA 计算设备的架构细节。下表简要列出了每个 SM 中各类单元的数量以及对应流水线的每周期吞吐量：

Fermi (cc 2.0) Kepler (cc 3.x)
单元数量 吞吐量 单元数量 吞吐量
Load/Store 16 16 32 32
纹理（Texture） 4 4 16 16
算术运算 FP32 32 64 192 192
SFU 4 8 32 32

相对于这些架构吞吐量数值，流水线利用率指标报告的是运行时每条流水线观察到的利用率。高流水线利用率表明在 kernel 执行期间相应的计算资源被大量使用且经常处于繁忙状态。低值则说明该流水线使用频率不高、资源处于空闲。每条流水线的结果相互独立；将两条或多条流水线的利用率百分比相加并不会得到有意义的数值。由于流水线指标是 kernel 启动持续时间内的平均值，因此较低的值并不能完全排除该流水线在 kernel 执行某个时间点曾是瓶颈的可能。

算术指令由多种流水线执行，上表中以 32 位浮点指令和特殊函数运算两个例子作为示意。这些流水线提供的吞吐量差异很大，进而导致 CUDA C Programming Guide 中记录的 原生算术指令吞吐量 各不相同。对于本实验所支持的 CUDA 设备，这些算术吞吐量如下：

按定义，指令吞吐量描述了每个 SM 每周期可以处理多少操作。换句话说，吞吐量高的操作在发射时的代价比吞吐量低的操作要小。算术负载使用这一简化的成本模型——将发射的指令数按对应指令吞吐量的倒数加权。这样做可以评估每种算术指令对总体处理代价的占比有多大。这对于算术流水线利用率较高的 kernel 尤其有用，可以用来识别哪一类算术运算可能已经触及可用吞吐量的上限。

流水线利用率

显示在 kernel 执行期间 SM 上四条主要逻辑流水线的平均利用率。可用于判断某条流水线是否被过度使用，从而成为限制 kernel 性能的瓶颈；也有助于估计增加额外工作量后是否还能很好地扩展、还是会触及流水线上限。这里的"增加工作量"可以指：增加算术负载（例如提高某些计算的精度）、增加内存操作数量（包括引入寄存器溢出），或者通过增加每个 SM 上的 活跃 warp 数量来改善指令延迟隐藏。

报告的数值是 kernel 执行持续时间内的平均值。较低的利用率百分比并不能保证该流水线在 kernel 执行过程中从未发生过过载。

指标

Load / Store 覆盖所有发射的、会向目标设备内存系统发出请求的指令——不包括纹理操作。包含针对全局内存、本地内存、共享内存的加载和存储操作，以及任何原子操作。也包含寄存器溢出。计算能力 3.5 及以上的设备支持通过 只读数据缓存（LDG）加载全局内存；这类操作不计入 load/store 组，而是计入纹理流水线的利用率中。当 load/store 利用率较高时，请运行 内存统计，以获取所执行内存操作的类型、数量和效率的更多信息。

纹理（Texture） 覆盖所有发射的纹理读取指令；对于计算能力 3.5 及以上的设备，还包括通过 只读数据缓存（LDG）发起的全局内存加载。如果该指标较高，请运行 内存统计 — 纹理内存 实验，对所执行的纹理请求进行更详细的评估。

控制流 覆盖所有可能影响控制流的发射指令，例如分支指令（BRA,BRX）、跳转指令（JMP,JMX）、函数调用（CAL,JCAL）、循环控制指令（BRK,CONT）、返回指令（RET）、程序终止（EXIT）以及屏障同步（BAR）。关于这些具体指令的详细说明，请参阅 指令集参考。如果控制流利用率较高，请运行 分支统计 实验；这可以帮助理解控制流对整体 kernel 执行性能的影响。

算术运算 覆盖所有发射的浮点指令、整数指令、转换操作和移动指令。每一类对应的汇编指令详细列表请参阅 指令集参考。如果算术流水线利用率较高，请查看 算术负载 图表，以识别代价最高的指令类型。

算术负载

给出多种算术指令类别的估计代价分布。该成本模型以发射数乘以对应指令吞吐量的倒数作为加权。指令类别与本文档背景章节中算术吞吐量表的行一一对应。

算术指令由芯片上的多条流水线执行，它们可以并行工作。因此，算术负载的分布并不是 流水线利用率 图中算术流水线百分比的真正划分；但代价估计最高的指令类型，最有可能是导致片上流水线利用率最高的来源。

指标

下列指标描述中所有对具体汇编指令的引用，均指 CUDA 设备的原生指令集架构（ISA），详见 指令集参考。

FP32 所有 32 位浮点加法（FADD）、乘法（FMUL）、乘加（FMAD）指令的估计负载。

FP64 所有 64 位浮点加法（DADD）、乘法（DMUL）、乘加（DMAD）指令的估计负载。

FP32（特殊函数） 所有 32 位浮点倒数（RCP）、倒数平方根（RSQ）、以 2 为底的对数（LG2）、以 2 为底的指数（EX2）、正弦（SIN）、余弦（COS）指令的估计负载。

I32（Add） 所有 32 位整数加法（IADD）、扩展精度加法、减法、扩展精度减法、最小值（IMNMX）、最大值指令的估计负载。

I32（Mul） 所有 32 位整数乘法（IMUL）、乘加（IMAD）、扩展精度乘加、绝对差之和（ISAD）、置位计数（POPC）、前导零计数、最高非符号位（FLO）的估计负载。

I32（Shift） 所有 32 位整数左移（SHL）、右移（SHR）、漏斗移位（SHF）指令的估计负载。

I32（Bit） 所有 32 位整数位反转、位域提取（BFE）、位域插入（BFI）指令的估计负载。

逻辑运算 所有逻辑运算（LOP）的估计负载。

Shuffle 所有 warp shuffle（SHFL）指令的估计负载。

Conv（I8/I16 到 I32） 所有从 8 位、16 位整数到 32 位类型的类型转换的估计负载（I2I 的子集）。

Conv（与 FP64 互转） 所有与 64 位类型相互转换的类型转换的估计负载（I2F、F2I 和 F2F 的子集）。

Conv（其他全部） 所有其他类型转换的估计负载（I2I、I2F、F2I 和 F2F 中剩余的子集）。

分析

  • 如果 Load/Store 流水线利用率 较高……
    • ……运行 内存统计，查找那些每次请求事务数超过理想值的内存操作；改进这些非最优的内存访问可以降低 load/store 流水线的负担。
    • ……考虑能否通过改用不同的实现或不同的算法来减少所执行的内存操作总数。
    • ……若单个线程访问内存中连续的数值，请确保使用最宽的内存访问类型来完成操作。例如，每个线程需要读取 128 字节连续内存时，使用一次 128 字节宽的访问要优于执行四次 32 字节访问。为此可使用 内建向量类型。
    • ……如果 内存统计 — 本地内存 实验显示本地内存请求数较多，请开启 PTX 优化汇编器的代码生成统计信息打印（--ptxas-options=-v）以获取溢出字节数；如果该数值较高，可尝试通过修改 kernel 启动的 执行配置 或 launch bounds 来减少寄存器溢出。
  • 如果 纹理流水线利用率 较高……
    • ……运行 内存统计 — 纹理内存 实验，研究所发起的纹理读取的效率。
    • ……考虑能否通过改用不同的实现或不同的算法来减少所执行的纹理操作总数。同时检查 kernel 是否已经接近目标设备预期的峰值纹理性能。
  • 如果 控制流流水线利用率 较高……
    • ……运行 分支统计 实验。当 控制流效率 较低且控制流分歧较高时，尝试重新组织 kernel grid 中的线程，目标是最小化 warp 分歧。
  • 如果 算术流水线利用率 较高……
    • ……查看 算术负载，并优先优化代价最高的指令类型。在可行的情况下，优先使用更快、更专用的数学函数，避免使用较慢、更通用的版本。也可参考 CUDA C Best Practices Guide 中关于优化 算术指令 的通用建议。
    • ……运行 实际 FLOPS 和 实际 IOPS 实验。检查 kernel 是否已经接近目标设备的峰值性能，并研究是否可以减少所执行的算术指令数量。
