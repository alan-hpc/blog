摘要

本报告介绍了 DeepSeek-V4 系列，这是我们面向下一代大语言模型的最新研究成果，旨在突破超长上下文处理的效率瓶颈。DeepSeek-V4 系列包含两个变体：DeepSeek-V4-Pro（1.6T 总参数，49B 激活参数）和 DeepSeek-V4-Flash（284B 总参数，13B 激活参数），两者均原生支持 1M token 上下文窗口。

在架构层面，我们提出了三项核心创新：（1）流形约束超连接（mHC），通过将通道混合矩阵约束在 Birkhoff 多面体上，实现更稳健的层间信息流；（2）混合稀疏注意力，由压缩稀疏注意力（CSA）和重度压缩注意力（HCA）构成，仅需 V3.2 约 73% 的注意力 FLOPs 和 90% 的 KV Cache 开销；（3）将 Muon 优化器成功应用于超大规模语言模型训练，并针对分布式设置进行了系统性适配。

在系统层面，我们开发了 MegaMoE EP（基于波形重叠的专家并行流水线）、TileLang（面向神经网络内核的领域特定语言）、批不变推理内核，以及 MXFP4 量化感知训练框架。推理侧引入了异构 KV Cache 架构（状态缓存 + 分页缓存）和三种磁盘 KV Cache 策略。

评测结果表明，DeepSeek-V4-Pro-Max（最高推理模式）在开源模型中取得了新的最优性能，在代码竞赛（Codeforces 评分 3206，人类前 23 名）、长上下文检索（MRCR 1M 83.5%）、形式化数学（PutnamBench 120/120）等方面可与前沿闭源模型媲美，同时具备高度竞争力的 Agent 能力。

1. 引言

近年来，大语言模型（LLM）在推理、编程、数学等各类任务上取得了显著进展。随着智能体（Agent）系统、长文档分析、代码库级别理解等应用场景的涌现，对超长上下文（百万 token 级别）的高效处理成为下一代 LLM 的核心挑战。

现有的长上下文方案面临两个根本性困境：（1）计算开销：标准注意力的 FLOPs 随序列长度平方增长；（2）KV Cache 开销：所有 token 的 KV 需要驻留显存，百万 token 场景下超出单机 GPU 显存上限。虽然滑动窗口注意力（SWA）、线性注意力等方法缓解了部分问题，但往往以牺牲全局信息感知为代价。

DeepSeek-V4 系列提出了系统性解决方案：通过 CSA + HCA 的混合稀疏注意力架构，在保留完整全局信息感知的前提下，将注意力 FLOPs 降低约 27%，KV Cache 减少约 10%。结合 mHC 超连接对层间信息流的优化、Muon 优化器对矩阵参数训练的改进，以及 MegaMoE EP 等系统优化，DeepSeek-V4 系列在百万 token 场景下实现了效率与性能的同步提升。

2. 模型架构

2.1 流形约束超连接（Manifold-Constrained Hyper-Connections，mHC）

标准残差连接的形式为 $x_{l+1} = x_l + f_l(x_l)$，信息流动路径单一，限制了层间信息的灵活传递。Zhu 等（2025）提出的超连接（Hyper-Connections，HC）将残差扩展为 $n_{\text{hc}}$ 路通道，允许不同层的输出按可学习的权重矩阵混合传递。

DeepSeek-V4 在 HC 的基础上引入流形约束：要求通道混合矩阵 $B_l$ 必须属于 Birkhoff 多面体 $\mathcal{M}$（所有双随机矩阵的集合）：

$$\mathcal{M} = \left\{X \in \mathbb{R}^{n_{\text{hc}} \times n_{\text{hc}}} \;\middle|\; X_{ij} \ge 0,\; \sum_j X_{ij} = 1,\; \sum_i X_{ij} = 1\right\}$$

双随机约束保证了每路通道的权重之和为 1（行归一化）且每路通道被各层的贡献之和也为 1（列归一化），相当于层间信息流动的一种"保总量软排列"。其极点恰好是排列矩阵，因此 mHC 的通道混合可以理解为各置换矩阵的凸组合。

具体参数化方式为：令 $\tilde{A}_l$、$\tilde{C}_l$ 为可训练参数，则：

$$A_l = \sigma(\tilde{A}_l), \quad C_l = 2\,\sigma(\tilde{C}_l)$$

其中 $\sigma$ 为 sigmoid 函数，$A_l$ 为残差强度矩阵，$C_l$ 为输入缩放矩阵（系数 2 使期望值为 1）。随后通过 Sinkhorn-Knopp 迭代（交替行归一化和列归一化，共 20 次）将 $A_l$ 投影到 Birkhoff 多面体上，得到 $B_l$。

DeepSeek-V4 设置 $n_{\text{hc}} = 4$，mHC 为训练带来约 6.7% 的额外开销，但显著提升了模型性能和训练稳定性。

2.2 混合稀疏注意力（Hybrid Sparse Attention）

2.2.1 压缩稀疏注意力（Compressed Sparse Attention，CSA）

CSA 的目标是以低于全量注意力的代价捕获长程稀疏依赖。其核心设计包括两个组件：

Token 级压缩器：对输入序列应用压缩比为 $m=4$ 的 token 级压缩，将 $n$ 个 token 压缩为 $n/m$ 个压缩 token。压缩器通过可学习的线性变换实现，在降低后续注意力计算量的同时，尽量保留每组 token 的语义信息。

Lightning Indexer：在压缩 token 上使用 FP4 低精度 QK 内积快速计算 attention score，并选出每个 query 最相关的 top-$k$ 压缩 token。FP4 精度足以保证 top-k 排序的准确性，计算开销仅为全精度的 1/8。被选中的 top-$k$ token 随后参与完整精度的稀疏 MQA（多查询注意力）计算。

CSA 在每层同时包含一个滑动窗口注意力（SWA）分支（窗口大小 $n_{\text{win}}=128$），保证对近邻 token 的完整覆盖。因此 CSA 的 KV Cache 由两部分构成：固定大小的 SWA 窗口 KV + top-$k$ 压缩 token 的稀疏 KV，两者合计远小于全量 KV。

2.2.2 重度压缩注意力（Heavily Compressed Attention，HCA）

HCA 采用比 CSA 更激进的序列级压缩，压缩比为 $m'=128$：将连续 128 个 token 聚合为一个"摘要 KV 向量"，整个序列压缩为 $n/128$ 个摘要 token，然后对这 $n/128$ 个摘要 token 做完整（dense）MQA。

HCA 不使用 RoPE 位置编码（位置信息已隐含在压缩过程中），因此每个 query 可以无差别地关注所有 $n/128$ 个摘要 token，获得整个百万 token 序列的全局上下文摘要。HCA 的 KV Cache 大小为 $O(n/128)$，相比全量 KV 节省了 128 倍。

CSA 提供"精准定位"（通过稀疏 top-k），HCA 提供"全局鸟瞰"（通过密集压缩摘要），两者互补：CSA 覆盖局部精细依赖，HCA 覆盖长程粗粒度依赖，共同为每个 token 提供完整的上下文感知。

2.2.3 其他注意力实现细节

  • QK RMSNorm：对 Q 和 K 分别应用 RMSNorm，稳定注意力 logit 的量级，防止长序列下 attention score 出现极端值。
  • Partial RoPE：仅对 Q/K 向量的后 64 维应用 RoPE 位置编码，其余维度不加位置信息，在保留位置感知的同时提升模型对绝对位置的泛化能力。
  • Attention Sink：在 HCA 的压缩序列头部保留若干"汇聚 token"（attention sink），防止注意力权重分布退化。
  • 注意力头数：CSA 使用 16 个 query head，HCA 使用若干 query head（论文未披露确切数量）。

2.3 Muon 优化器

Muon（Jordan et al., 2024）是一种专为矩阵参数设计的优化器，核心思想是将梯度矩阵 $G$ 投影到其正交 Polar 因子（即对 $G$ 做 SVD，取正交部分 $U$，使 $G=U\Sigma V^T$），然后用 $U$ 代替归一化梯度做参数更新。这等价于以矩阵谱范数归一化梯度，保留了梯度的全局方向信息，而 Adam 的逐元素归一化则会破坏这一结构。

直接 SVD 计算代价高昂，Muon 用 Newton-Schulz 迭代数值逼近正交因子：

$$X_{k+1} = a\,X_k + b\,X_k(X_k^T X_k) + c\,X_k(X_k^T X_k)^2$$

其中系数 $(a,b,c)$ 使多项式在奇异值区间 $[0,1]$ 内快速收敛至 1（对应正交矩阵）。DeepSeek-V4 使用 10 次迭代（8 次固定系数 + 2 次精调系数），可将正交化误差压缩至 $10^{-7}$ 量级。

Muon 与 ZeRO 参数分片结合使用：每个 GPU 在本地梯度分片上执行 Newton-Schulz 迭代，通过 All-Reduce 聚合正交化后的更新。Muon 仅应用于线性层权重矩阵（注意力 QKV/Output 投影和 FFN 权重），嵌入层和 LayerNorm 仍使用 AdamW。据论文报告，Muon 在长上下文扩展阶段（128K→1M token）相比 AdamW 展现出更好的收敛稳定性。

2.4 DeepSeekMoE 架构

V4 沿用 DeepSeek 系列的 MoE（Mixture-of-Experts）架构，每个 FFN 层由多个专家网络构成，每个 token 只激活其中少数专家。V4-Pro 使用 256 个专家（8 个共享专家 + 248 个路由专家），每个 token 激活 top-8 路由专家 + 全部 8 个共享专家，共激活 16 个专家的 FFN，大幅降低推理时的实际计算量。

3. 支撑基础设施

3.1 MegaMoE EP：波形细粒度重叠流水线

MoE 的 Expert Parallel（EP）分组需要两次 All-to-All 通信：Dispatch（将 token 路由到对应专家 GPU）和 Combine（汇总专家输出）。传统 EP 中这两次通信是阻塞的，GPU 在通信期间空闲。

MegaMoE EP 将一批 token 切分为若干 wave（波），令不同 wave 的 Dispatch / 线性层 L1 / SwiGLU / 线性层 L2 / Combine 操作在时间上流水线重叠执行：当 Wave $i$ 在做 L1 GEMM 时，Wave $i+1$ 的 Dispatch 通信同步进行，Wave $i-1$ 的 SwiGLU 激活也在运行，使 All-to-All 通信完全隐藏在计算中。端到端测试表明，MegaMoE EP 相比朴素 EP 实现了 1.50–1.73× 的吞吐量提升（随 token 数和专家分布而异）。

3.2 TileLang DSL

TileLang（Wang et al., ICLR 2026）是 DeepSeek 内部开发的深度学习内核编程 DSL，用 Python 语法描述 tile 级别的 CUDA 操作，编译器自动生成优化的设备代码和主机调用接口。主要特性：

  • 主机代码自动生成：从 kernel 描述自动生成 Python/C++ 调用接口，无需手写 launch 逻辑。
  • Z3 SMT 整数分析：编译期静态检查 tile 参数合法性，提前发现越界、对齐错误。
  • IEEE-754 确定性：保证相同输入产生完全相同的浮点输出，便于分布式一致性验证。

V4 的多个核心内核（MegaMoE 专家 GEMM、Lightning Indexer、OPD 的 KL 散度计算）均使用 TileLang 实现。

3.3 批不变推理内核（Batch-Invariant Kernels）

推理时 prefill 阶段和 decode 阶段的 batch size 差异极大，单一内核难以同时优化两种场景。V4 采用双内核策略：Single-SM 内核（每序列一个 SM，适合 decode 小 batch）和 Multi-SM 内核（每序列跨多 SM，适合 prefill 大 batch），运行时动态选择。同时，V4 以 DeepGEMM 替代 cuBLAS 作为标准 GEMM 后端，DeepGEMM 专为 MoE 的 Fine-Grained Scaling GEMM 优化，在非规则 tile 形状下性能显著优于 cuBLAS。

3.4 FP4 量化感知训练（MXFP4 QAT）

V4 对 MoE 专家权重 和 CSA Lightning Indexer 的 QK 投影权重 应用 MXFP4（MicroScaling FP4）量化感知训练。MXFP4 在 4 位精度下引入 group-wise 共享指数，扩展了数值表达范围。

关键技巧是 FP4 与 FP8 的无损互转：FP8（E4M3）的动态范围足以覆盖 MXFP4 的所有可表示值，因此 FP4 权重可无损表示为 FP8，与现有 FP8 混合精度框架（FP32 主权重 + FP8 梯度）无缝兼容，无需修改反向传播流水线。推理时直接使用原生 FP4 权重，利用 Blackwell 架构 Tensor Core 的原生 FP4 GEMM 支持，降低显存占用和带宽需求。

3.5 训练基础设施

Muon ZeRO 混合：Muon 优化器与 ZeRO 参数分片深度集成，Newton-Schulz 正交化在本地分片上执行，通过 All-Reduce 聚合更新，与 ZeRO-3 的通信模式兼容。

长上下文 Context Parallel：百万 token 训练采用两阶段 Context Parallel（CP）：第一阶段在序列维度切分（序列并行），第二阶段在注意力头维度切分，充分利用 MegaMoE EP 的通信拓扑。

TorchFX 张量级 Checkpointing：利用 TorchFX 程序捕获框架，在张量粒度上精确控制梯度检查点（gradient checkpointing），仅保存反向传播必需的中间激活值，大幅降低峰值显存占用，同时减少不必要的重计算开销。

3.6 推理架构：异构 KV Cache 与磁盘卸载

异构 KV Cache：V4 的三流注意力对 KV Cache 的需求各不相同，采用分类管理：

  • 状态缓存（State Cache）：管理 SWA 的固定大小滑动窗口 KV（$n_{\text{win}}=128$），以环形缓冲区预分配，无动态分页开销。
  • 分页缓存（Paged Cache）：管理 CSA 稀疏全局 KV 和 HCA 压缩 KV，按需动态分配和释放，支持不同序列长度和路由结果。

磁盘 KV Cache：支持将 KV Cache 卸载到 NVMe SSD，提供三种策略：

  • Full 策略：全量缓存所有层的 SWA KV 到磁盘，最大化 GPU 显存节省，适合极长序列低频访问。
  • Periodic 策略：每隔 $k$ 层缓存一次，灵活平衡显存与 I/O 开销。
  • Zero 策略：不卸载 SWA KV，仅将已大幅压缩的 CSA/HCA KV 保留在显存，延迟最低，适合中等长度场景。

磁盘 KV Cache 通过异步预取和 ping-pong 缓冲区实现 I/O 与计算的重叠，避免 SSD 带宽成为推理瓶颈。

4. 预训练

4.1 数据与上下文扩展课程

V4 系列的预训练数据总量超过 32T tokens，覆盖多语言文本、代码、数学、科学论文等高质量语料。训练采用多阶段上下文扩展课程：

  • 主训练阶段：在 4K token 上下文长度下完成大部分预训练，建立核心语言理解和生成能力。
  • 中等上下文扩展：逐步引入 8K → 32K → 128K 长上下文数据（长文档、代码库、多轮对话等），配合 RoPE 频率参数调整渐进扩展。
  • 长上下文扩展至 1M：最终阶段使用百万 token 合成数据（超长文档拼接、大型代码仓库等），配合两阶段 CP 策略完成 1M token 原生支持。

4.2 训练稳定性技术

Anticipatory Routing（预期路由）：标准 token-choice 路由在做路由决策时，仅依赖当前 token 的隐藏状态，不感知其他 token 的路由偏好，易导致热门专家过载、训练损失震荡。Anticipatory Routing 引入对同批次其他 token 路由结果的轻量预期估计，使路由决策考虑全局负载，从而平滑训练曲线。

SwiGLU Clamping（SwiGLU 截断）：SwiGLU 激活函数 $f(x,y)=x\cdot\sigma(y)\cdot y$ 在 $y$ 极大时可产生异常激活值，引发数值不稳定（NaN/Inf）。V4 对 SwiGLU 输出施加软截断（soft clamp），将超出阈值的激活值平滑压缩，在正常范围内保持梯度不变。

论文在结论中坦承，Anticipatory Routing 和 SwiGLU Clamping 虽然实践中有效，但数学原理尚未完全清晰，属于"工程实验有效、理论待深入研究"的状态，未来将通过更系统的消融实验加以理解。

5. 后训练

5.1 专家训练（Specialist Training）

V4 的后训练分为两个主要阶段：专家训练和 On-Policy 蒸馏（OPD）。专家训练阶段为不同领域（数学、代码、创意写作、搜索等）分别训练专门的"领域专家"模型，训练方法为 SFT + GRPO + GRM 的组合：

  • SFT（Supervised Fine-Tuning）：在高质量领域数据上做监督微调，建立基础能力。
  • GRPO（Group Relative Policy Optimization）：通过对比同一 prompt 下多个生成结果的相对奖励优化策略，无需额外 critic 网络。
  • GRM（Generative Reward Model）：使用生成式 LLM 作为奖励模型，提供细粒度过程奖励，替代传统判别式打分器。

5.1.1 思维管理（Thinking Management）

V4 提供三种推理模式：Non-Think（快速直接响应）、Think High（中等深度推理链，8K 上下文）、Think Max（最深度推理，384K 上下文，更低长度惩罚）。不同模式在评测和产品中对应不同的计算预算。

在 agentic（多工具调用）场景中，V4 对 thinking trace 的管理策略做了区分：

  • 工具调用场景：完整保留所有轮次的 thinking trace（包括跨用户消息边界），模型可在多轮工具调用中积累推理历史（Cumulative Chain of Thought），支持长程复杂任务。
  • 普通对话场景：新用户消息到来时丢弃上一轮的 thinking trace，仅保留最终答案，保持上下文简洁，避免无效 thinking 占用 KV Cache。

Quick Instruction：chatbot 场景中通常需要在生成响应前先判断是否触发搜索、识别用户意图等辅助任务，传统做法需要额外小模型处理（额外 prefill）。Quick Instruction 将这些辅助任务编码为专用特殊 token（如 <|action|>、<|query|>、<|title|> 等），直接追加在已生成序列末尾，完全复用已有 KV Cache，多个辅助任务可并行生成，显著降低 TTFT（首 token 生成延迟），并消除维护独立小模型的工程开销。

5.1.2 On-Policy Distillation（OPD）

专家训练产生 10 余个领域专家模型，OPD 将所有专家知识蒸馏进统一学生模型 $\pi_\theta$。OPD 的目标函数为各专家的反向 KL 散度加权求和：

$$\mathcal{L}_{\text{OPD}}(\theta) = \sum_{i=1}^{N} w_i \cdot D_{\text{KL}}\!\left(\pi_\theta \,\|\, \pi_{E_i}\right)$$

其中 $N$ 为教师专家数量（V4 使用 10+），$w_i$ 为各专家的重要性权重，$\pi_{E_i}$ 为第 $i$ 个专家的输出分布。

V4 的 OPD 采用全词表 logit 蒸馏：完整计算所有 $|V|>100{,}000$ 词表位置的真实 KL 散度，而非以 token-level 优势估计近似（此类近似方差大、训练不稳定）。全词表蒸馏提供更稳定的梯度估计，但也带来显著的内存挑战（需要具体的工程设计缓解，见 §5.2）。

OPD 使用学生模型的在线（on-policy）生成轨迹作为训练样本：学生在各领域 prompt 上生成候选答案，教师为这些轨迹提供 logit 分布，学生据此优化。这确保了学生在自身分布下的对齐，避免了离线蒸馏中的分布偏移问题。

5.2 RL 与 OPD 基础设施

V4 的后训练基础设施在 DeepSeek-V3.2 分布式训练框架和推理引擎的基础上，引入以下重要改进：

5.2.1 FP4 量化集成

将 MXFP4 量化应用于 rollout 和所有推理前向传播（包括教师模型和参考模型），显著降低内存带宽消耗和采样延迟。训练步骤中，FP4 量化通过无损 FP4→FP8 反量化模拟，与现有 FP8 混合精度框架（FP32 主权重）无缝兼容，无需修改反向传播流水线。

5.2.2 高效教师调度（Efficient Teacher Scheduling）

全词表 OPD 面临的核心挑战：教师模型参数可能达到万亿级，多个教师模型不可能同时全量驻留显存；词表大小 $|V|>100{,}000$ 时，直接物化完整 logit 矩阵会超出显存。

V4 的解决方案：（1）所有教师权重卸载到集中式分布式存储，按需通过 ZeRO-like 参数分片加载；（2）前向传播时只缓存最后一层的隐藏状态（而非完整 logit），训练时通过对应预测头即时重建 logit，规避了显存瓶颈；（3）训练样本按教师 GPU 索引排序，确保每个独立教师头在设备内存中同一时刻只驻留一份，最大化内存复用；（4）所有参数和隐藏状态的加载/卸载在后台异步进行，不阻塞计算关键路径；（5）KL 散度计算使用专用 TileLang 内核加速并避免动态内存分配。

5.2.3 可抢占容错 Rollout

V4 的 GPU 集群实行全集群任务抢占调度，任何运行中的任务随时可能被中断。为保证 RL/OPD rollout 的正确性，V4 为每个生成请求实现了 token 粒度的 WAL（Write-Ahead Log）：每生成一个 token 立即追加到该请求的 WAL，同时保存 KV Cache 快照。

  • 抢占时：暂停推理引擎并持久化当前状态（WAL + KV Cache）。
  • 恢复时：直接从 WAL 继续解码，无需从头重新生成。

重要：重新生成会引入长度偏差——较短响应更易在抢占中存活，导致模型偏向生成短序列。token 粒度 WAL 从根本上消除了这一偏差，保证了 RL 训练的数学正确性。若遇到硬件故障，可通过 WAL 中的 token 重新 prefill 重建 KV Cache，完全恢复生成状态。

5.2.4 百万 Token 上下文 RL 扩展

1M token 序列的 RL/OPD 数据带来极大的内存和 I/O 压力。V4 将 rollout 数据结构分为两类：

  • 轻量元数据：可完整加载到内存，用于全局 shuffling 和 packing 布局计算。
  • 重量 per-token 字段：通过共享内存数据加载器延迟加载，在 mini-batch 粒度即用即释放，避免长序列数据全量驻留 CPU/GPU 内存。

每个 mini-batch 的数量根据计算吞吐量和 I/O 重叠情况动态确定，在实际运行中实现高效的计算-I/O 流水线。

5.2.5 DeepSeek Elastic Compute（DSec）沙箱平台

为支持 agentic AI 的 post-training 和评测（代码执行、工具调用、web 搜索等），V4 开发了生产级弹性计算沙箱平台 DSec。DSec 由三个 Rust 组件构成：

  • Apiserver（API 网关）：接收任务请求，分发给 Edge。
  • Edge（per-host 代理）：管理本机沙箱实例生命周期。
  • Watcher（集群监控）：协调全集群调度。

DSec 通过统一 Python SDK（libdsec）暴露四种执行基底：Function Call（预热容器池，最低冷启动延迟）、Container（Docker 兼容，基于 EROFS 的按需分层存储）、microVM（Firecracker，VM 级隔离，毫秒级启动）、fullVM（QEMU，支持任意 Guest OS）。生产环境中，单个 DSec 集群管理数十万并发沙箱实例，基于 DeepSeek 3FS 分布式文件系统横向扩展。

DSec 的核心设计原则：（1）快速镜像加载：通过 EROFS 只读层 + overlayfs 写时拷贝层，容器基础镜像按需从 3FS 加载，实现快速启动；（2）高密度并发：消除页缓存重复、减少 spinlock 争用，使单主机沙箱密度最大化；（3）抢占安全恢复：全局有序的 trajectory log 记录每个沙箱的命令调用及结果，支持客户端快速恢复、细粒度溯源和确定性重放。

5.4 真实场景评测

标准基准往往无法完整反映真实用户体验，V4 构建了针对 DeepSeek API 和 Chatbot 主要使用场景的专有内部评测框架。

中文写作（5.4.1）：以 Gemini-3.1-Pro 为基线，在 3170 条功能写作（商务文档、媒体文案、公文等）任务上，DS-V4-Pro 整体胜率 62.65% vs 34.10%；在 2837 条创意写作（小说、诗歌、散文等）的写作质量维度胜率 77.5%。主要优势来自 DS-V4-Pro 能主动满足隐性用户意图、提供自我核查，并生成深度连贯的长篇内容，而非浅显的 bullet point 列表。在复杂约束跟随和多轮写作场景中，Claude Opus 4.5 仍保持优势（52% vs 45.9%）。

搜索增强（5.4.2）：在 DeepSeek 产品的搜索场景中，Non-Think 模式使用 RAG，Think 模式使用 Agentic Search（模型主动迭代调用搜索和工具）。在 869 条主客观混合评测中，Agentic Search 的整体胜率 61.7%，远优于 RAG（18.3%），但成本仅略高（平均 16.2 次工具调用，prefill token 增加约 30%）。主要优势集中在单值信息检索和规划策略类任务，比较推荐类主观任务仍有提升空间。

白领专业任务（5.4.3）：构建涵盖金融、教育、法律、技术等 13 个行业的 30 项高认知需求中文专业任务，与 Claude Opus 4.6-Max 做人工盲测对比，从任务完成度、指令遵循、内容质量、格式美观四个维度评分。DS-V4-Pro-Max 的总体不败率达 63%（53% Win + 10% Tie），在任务完成和内容质量维度得分最高，但格式美观（幻灯片排版设计）仍有较大提升空间。

代码 Agent（5.4.4）：从约 200 个真实内部研发工作负载（PyTorch、CUDA、Rust、C++ 等多种技术栈，含功能开发、Bug 修复、重构、诊断等类型）中经质量筛选后保留 30 个任务。DS-V4-Pro-Max 通过率 67%，接近 Claude Opus 4.5（70%），大幅超越 Claude Sonnet 4.5（47%）。内部调研显示，52% 的 DeepSeek 研究人员已将 DS-V4-Pro 作为首选日常编程助手，最主要的批评是偶尔出现的细节错误、对模糊 prompt 的误解以及过度思考（over-thinking）。

6. 结论与局限性

本文介绍了 DeepSeek-V4 系列，这是一系列面向下一代大语言模型的预览版本，旨在突破超长上下文高效处理的效率瓶颈。通过整合 CSA 和 HCA 的混合注意力架构，DeepSeek-V4 系列在长序列效率方面实现了重大突破。架构创新配合广泛的基础设施优化，有效支持了百万 token 上下文的原生处理，并为未来的测试时计算扩展（Test-Time Scaling）、长程 agentic 任务以及在线学习等新兴范式建立了必要的技术基础。

评测结果表明，DeepSeek-V4-Pro-Max 在开源模型中重新定义了最优性能，在知识基准上显著超越已有开源模型，在推理性能上接近前沿闭源模型，并具备强竞争力的 Agent 能力。与此同时，DeepSeek-V4-Flash 在维持高度成本效益架构的同时，实现了与领先闭源模型可比的推理性能。我们相信，DeepSeek-V4 系列将为开源模型和 AI 社区开启百万 token 时代的全新可能。

在追求极致长上下文效率的过程中，DeepSeek-V4 系列采用了相对大胆的架构设计，为控制风险，保留了大量经初步验证有效的组件和技巧，这也使架构相对复杂。在未来的迭代中，我们将开展更全面、更系统的研究，将架构精炼为最本质的设计，在不牺牲性能的前提下追求更高的优雅性。与此同时，尽管 Anticipatory Routing 和 SwiGLU Clamping 已在实践中被证明有效，其底层原理仍有待深入理解。我们将积极研究训练稳定性的基础性问题，建立更完善的内部监控指标体系，力求在大规模训练中实现更有原则性、更可预测的稳定性。

此外，超越 MoE 和稀疏注意力架构之外，我们也将主动探索模型在新维度上的稀疏化可能——例如更稀疏的嵌入模块——以进一步提升计算和内存效率而不损害模型能力。我们还将持续研究低延迟的长上下文交互架构与系统技术，使百万 token 的长上下文部署和用户交互更加高效响应。同时，我们认识到长程多轮 agentic 任务的重要性和实践价值，将继续在这一方向深入探索。最后，我们还将推进多模态能力的研究，并持续改进数据策划与合成策略，在更广泛的任务场景和用户需求上提升模型的智能性、鲁棒性和实际可用性。
