DeepSeek V3 推理系统深度解析（三）：DeepGEMM Expert Compute，从 FP8 Grouped GEMM 到两套 MoE Layout

2026-05-06

原文来源 DeepSeek V3 推理系统深度解析（三）：DeepGEMM Expert Compute，从 FP8 Grouped GEMM 到两套 MoE Layout · Yang Min · 2026

DeepGEMM 的核心问题不是"FP8 GEMM 能不能运行"，而是MoE 的不规则 expert shape 如何让 FP8 GEMM 在 Hopper 上持续高效。标准 cuBLAS FP8 GEMM 假设固定的 M/N/K，但 MoE decode 的 token 分布让每个 expert 的 M 大小在运行时完全不可预测。DeepGEMM 用 JIT 编译 + grouped layout 绕开这个约束：不预编译，而是在第一次遇到特定 shape 时即时生成 CUTLASS kernel。不读这个设计决策，就不会理解为什么 JIT 在这里是必要的而不是可选的。

Prologue · 本篇问题与读法

这一篇只讨论 routed expert compute：DeepEP 已经把 token 送到目标 expert，下一步要用两次 expert GEMM 和一次 SwiGLU 把 token 算完，再交回 Combine。DeepGEMM 在这条链路里不是“又一个 GEMM 库”，而是一个专门服务 DeepSeek-V3/R1 FP8 MoE 形状的 expert compute 后端。

本文把问题拆成五层：第一层先确认 MoE FFN 的数学语义；第二层解释 prefill 与 decode 为什么要分成 contiguous / masked 两套 layout；第三层解释 FP8 block scale 和 scale promotion；第四层进入 Hopper SM90 kernel 的 TMA、WGMMA、barrier、scheduler；最后回到 vLLM，说明什么条件下这条路径会被选中，什么时候必须 fallback。

需要先划清边界：DeepGEMM 当前仓库已经扩展到 SM90/SM100、FP8/FP4/BF16、Mega MoE、indexer scoring 等更多 kernel family。本文聚焦的是 DeepSeek-V3/R1 推理主线里的 SM90 FP8 grouped GEMM ，也就是 part 3 的主题；Mega MoE 融合 kernel 和 Blackwell FP4 会在独立文章里展开。

读代码时先抓接口契约 ：DeepGEMM 不负责 routing、不负责普通 MoE 的 top-k 选择，也不替 Combine 做 slot-sum。它只吃已经整理好的 A / A_scale / B / B_scale / grouped_layout，并且要求 NT 权重布局、K 维 128 block quant、LHS scale 已经是 TMA-friendly 的 column-major 形态。任何额外 transpose / cast / permute 如果没有被前后 kernel 融掉，都会回到 HBM 成本里。

Routed expert FFN 的语义对象

$$ \begin{array}{lll} g & = & (t,k),\qquad e=\pi_{t,k},\qquad s=s_{t,k},\\[0.55em] X_g & = & \operatorname{Dispatch}(X_t,e),\\[0.55em] H_g & = & X_g W_{1,e}^T=[G_g,U_g],\\[0.55em] A_g & = & \operatorname{silu}(G_g)\odot U_g\cdot s,\\[0.55em] (\widetilde A_g,A_{sf,g}) & = & Q_{\mathrm{FP8}}(A_g),\\[0.55em] Z_g & = & \widetilde A_g W_{2,e}^T,\\[0.55em] Y_t & = & \sum_{k:\pi_{t,k}\ne -1} Z_{(t,k)} . \end{array} $$

这里故意把 routing weight s 放进 SwiGLU 输出一侧，而不是在 Combine 再乘一次。这样 Linear2 的输出已经是带权贡献，Combine 的语义就简化成 slot-sum。理解这一点很重要，否则很容易把 DeepGEMM / DeepEP / vLLM 里的 buffer 命名和高层 MoE 公式对不上。

1 · Expert compute 边界：不是普通 batched GEMM

1.1 DeepEP 输出什么，DeepGEMM 接什么

DeepEP 的 HT/LL dispatch 已经解决“token 到哪个 expert、跨 rank 怎么搬”的问题。到 DeepGEMM 入口时，核心对象不是原始 token 序列，而是按 expert 聚合后的 pooled token，以及两个辅助数组：每个 slot 属于哪个 expert、每个 expert 有多少真实 token。

如果把这一步当成普通 batched GEMM，会很快撞到两个问题。第一，每个 expert 的 M_e 不同，batch 维度并不规则；第二，prefill 和 decode 对 shape 的要求完全相反。Prefill 希望少 padding，decode 希望 shape 固定以便 CUDA Graph replay。

1.2 Grouped GEMM 只 group M 维

DeepGEMM 的 MoE grouped GEMM 与传统 grouped GEMM 的关键区别是：只在 M 维 group，N/K 固定。这样每个 expert 的权重形状相同，kernel 不需要为每个 group 维护完全不同的 GEMM descriptor；变化只发生在“这个 expert 有多少行 token”。

M-grouped GEMM 的约束

$$ \begin{array}{lll} A_e & \in & \mathbb{R}^{M_e\times K},\\[0.45em] B_e & \in & \mathbb{R}^{N\times K},\\[0.45em] C_e & = & A_e B_e^T,\qquad e\in\{0,\ldots,E-1\},\\[0.45em] \text{only } M_e \text{ varies} &;& N,K \text{ are fixed across experts.} \end{array} $$

这就是为什么 DeepGEMM 的 grouped layout 看起来像“把所有 expert 的 M 轴重新排布”，而不是像 cuBLAS batched GEMM 那样把每个矩阵当成完全独立的 batch item。

1.3 API contract：窄接口才是性能边界

DeepGEMM 的 Python 入口很短，但约束很硬。m_grouped_gemm_fp8_fp8_bf16_nt_contiguous 接收 (M_sum,K) 的 packed token；m_grouped_gemm_fp8_fp8_bf16_nt_masked 接收 (E,M_max,K) 的固定三维张量。两者共同要求是 FP8 E4M3 输入、BF16 输出、RHS 已经是转置语义的 (E,N,K) 权重，scale 是 block-wise 128 粒度。

入口激活 layout动态信息典型阶段不该在这里发生的事
m_grouped_gemm_fp8_fp8_bf16_nt_contiguous(M_sum,K)m_indices[M_sum]prefill / 训练 forward每 expert 单独 launch、小矩阵 padding 到统一 M_max
m_grouped_gemm_fp8_fp8_bf16_nt_masked(E,M_max,K)masked_m[E] + expected_mdecode / CUDA Graph每步重建 shape、CPU 参与重排、动态 capture

这里最容易漏掉的是 LHS scale。DeepGEMM 要求它以 column-major、16B TMA-aligned 的方式进入 kernel；如果上游传普通 contiguous scale，辅助函数会临时转置，这条慢路径虽然保证正确性，却会把性能账重新暴露出来。因此 vLLM 里的 quant kernel 会直接产出 DeepGEMM 需要的 scale stride，而不是先写普通 scale 再让 GEMM 入口修。

1.4 官方性能画像：小 M 更吃调度质量

DeepGEMM 报告在 H800 SXM5 上给出的数据有一个很稳定的信号：M=64/128 这类小 M dense shape 的相对收益很高，MoE grouped shape 的收益更像是稳态提升。原因并不神秘：小 M 时 launch、scheduler、tile occupancy、scale 读取和边界判断的占比更高，能否把这些开销 JIT 到编译期、能否让 CTA 数接近 SM 数，会直接改变吞吐。

场景代表形状观察到的信号这说明什么
Dense 小 MM=64/128, K=7168相比内部 CUTLASS 路径提升明显overhead 与 tile 并行度决定上限
Dense 大 MM=4096 级别更接近硬件上限，提升收敛主瓶颈回到 Tensor Core 吞吐
Grouped contiguous多 expert 大 M 拼接稳定优于泛化 grouped 路径layout 与 scheduler 更贴近 MoE
Grouped maskeddecode 小 M + 固定 shape带宽和 graph 约束一起出现需要为了 graph 接受 padding

换句话说，DeepGEMM 的胜负不只在 Tensor Core 峰值，而在“能不能把小而散的 expert 计算变成硬件愿意吃的连续 tile 流”。当 M 足够大时，很多 GEMM backend 都能接近上限；当 M 很小、expert 很多、top-k 很稀疏时，layout、JIT config 和 scheduler 才是主要差异。

2 · 两套 MoE Layout：prefill 紧凑，decode 固定2.1 Contiguous 与 Masked 是同一数学的两种系统形态DeepGEMM 提供 contiguous layout 和 masked layout。它们不是两个不同的 MoE 数学公式，而是两套不同的物理存储契约。 Draw.io 图表 · 可编辑 figures/F2.drawio 两套 grouped GEMM layout：contiguous 把所有 expert token 拼到一条 M 轴；masked 保留固定三维形状，用 token count 让 kernel 只算有效部分。 Contiguous 与 Masked 的 layout 对比$$ \begin{array}{lll} A_{\mathrm{contig}} & = & \operatorname{concat}_e\left(X_e,\operatorname{pad}_{a_M}(X_e)\right),\\[0.55em] m_{\mathrm{idx}}[m] & \in & \{0,\ldots,E-1,-1\},\\[0.55em] A_{\mathrm{masked}}[e,m,:] & = & \begin{cases} X_{e,m,:}, & m<n_e,\\ \mathrm{unused}, & m\ge n_e, \end{cases}\\[0.55em] n_e & = & \mathrm{expert\_num\_tokens}[e]. \end{array} $$Prefill token 多，动态 shape 能接受，最重要的是减少 padding 与无效计算；decode token 少，又处在每 step 重复执行的热路径，最重要的是固定 shape，让 CUDA Graph、persistent buffer 和后端选择都稳定。2.2 Contiguous layout：m_indices 让整 tile skip 成为 O(1) 判断Contiguous layout 会把每个 expert 的 token 段按 get_mk_alignment_for_contiguous_layout() 对齐。有效行记录 expert id，padding 行记录 -1。kernel scheduler 取到一个 M tile 后，不需要逐行扫描，只看 tile 首行对应的 grouped_layout。 Draw.io 图表 · 可编辑 figures/F3.drawio Contiguous layout 的关键不是把 token 拼起来，而是让 padding 行变成整 CTA 粒度的 skip 判断。 优化前，prefill 如果按 expert 分别 launch GEMM，会产生大量小 GEMM launch 和低 occupancy；如果做普通 padding batched GEMM，又会把每个 expert 补到统一 max_m。Contiguous layout 的优化路径是：把所有有效 token 拼接成一条大 M 轴，只在每个 expert 尾部对齐到 block 边界，padding 的成本最多落在尾 tile。Contiguous 的有效 tile 判断$$ \begin{array}{lll} m_0 & = & m_{\mathrm{offset}} + b_M\cdot B_M,\\[0.45em] \mathrm{valid}(b_M) & = & m_{\mathrm{idx}}[m_0] \ge 0,\\[0.45em] \mathrm{expert}(b_M) & = & \max\left(0,m_{\mathrm{idx}}[m_0]\right). \end{array} $$这也是 contiguous layout 为什么更适合 prefill：layout 准备、scatter/gather 与一次大 grouped kernel 的成本能被长序列摊薄。2.3 Masked layout：decode 用 padding 换 CUDA GraphMasked layout 把输入看成 (E,max_m,K)。每个 expert 都有固定槽位，真实 token 数存在 expert_num_tokens[e]。scheduler 遍历 group 时，根据 ceil_div(expert_num_tokens[e], BLOCK_M) 决定当前 expert 有多少 M blocks。 Draw.io 图表 · 可编辑 figures/F4.drawio Masked layout 故意保留固定三维张量：Graph replay 只更新 token count，kernel 形状不变。 优化前，decode 每步 top-k 后的 expert token 分布都是动态的，CPU 若要为每步重新组织 shape、重新 launch 小 GEMM，会把低延迟路径拖垮。Masked layout 的优化路径是：固定外壳，把动态信息收敛到一个小的 count/mask 张量，让 CUDA Graph 和 buffer address 稳定。2.4 DeepEP 与 DeepGEMM 的 layout 契约DeepEP low-latency dispatch 输出可以直接喂给 masked grouped GEMM；prefill 或训练 forward 更偏向 是：# baseline sketch: 每个 expert 形成一个独立小 GEMM for e in active_experts: x_e = gather(tokens, topk_ids == e) h_e = x_e @ w1[e].T a_e = silu(h_e[..., :I]) * h_e[..., I:] z_e = quant(a_e) @ w2[e].T 优化点在于：routing 语义仍然完整，但 GEMM 看到的是一条连续的 tile stream；padding 行用 -1 标记后可以被 scheduler 整 tile 跳过；scale 直接按 TMA 所需布局生成；Combine 只处理 GEMM 已经算完的 slot。3 · FP8 数值路径：block scale 与 scale promotion3.1 GEMM1、SwiGLU、Quant、GEMM2 的物理路径DeepSeek-V3 expert FFN 一般可以理解成两段 grouped GEMM：第一段从 hidden 到 gate/up，第二段从 intermediate 回到 hidden。中间的 SwiGLU + routing weight + FP8 quant 是边界最敏感的地方。它如果写回一个大 BF16 activation，再由第二个 GEMM 读回，就会把 HBM 往返暴露出来。在 DeepGEMM 的基线路径里，这些边界仍然是分段 kernel，但每段的 layout 和 scale 都按后续 GEMM 需要准备。到了 Mega MoE 文章，才会进一步讨论把这些边界折到同一个 persistent kernel 里。3.2 Scale promotion：为什么不是先反量化再 GEMMFP8 GEMM 的朴素理解是先反量化 A_fp8、B_fp8，再做高精度 GEMM。但这会产生两个问题：反量化后的中间 tile 要占更多寄存器/SMEM/HBM，scale 乘法也会和 Tensor Core mainloop 混在一起，难以保持流水。DeepGEMM 的做法是：Tensor Core 先对 FP8 tile 做 raw accumulation；每个 K-block 结束后，CUDA core 用对应的 A scale / B scale 做 promotion，再累加到 FP32 final accumulator。 Draw.io 图表 · 可编辑 figures/F5.drawio Scale promotion 的等价变换：不物化反量化矩阵，只在 K-block 边界把 raw accumulator 乘上 A/B scale。 Scale promotion 的计算等价式$$ C_{ij} = \sum_{b=0}^{K/128-1} s^A_{i,b}s^B_{j,b} \left( \sum_{r=0}^{127} \widetilde A_{i,128b+r}\widetilde B_{j,128b+r} \right). $$这比“先反量化再 GEMM”快，是因为被移除的不是数学乘法，而是反量化 tensor 的物化、读写和生命周期管理。Tensor Core 仍然做它最擅长的 FP8 MMA，CUDA core 在每个 K-block 后补上 scale 语义。如果把这一段落到代码层，可以把一轮 K block 看成三步：TMA 搬入 A_fp8、B_fp8 与 A_scale。WGMMA 先产生 raw accumulator。CUDA core 用 A_scale * B_scale promote 到 FP32 final accumulator。因此 DeepGEMM 并没有“省掉 scale”，而是把 scale 从数据路径前置反量化移到 accumulator 边界。前者会扩大中间数据生命周期，后者只扩大少量寄存器中的 accumulator 更新。3.3 SM90 与 SM100 的 scale 边界DeepGEMM README 里明确区分了两代实现的 scale 格式：SM90 路径使用 FP32 scale，SM100 路径使用 packed UE8M0。本文分析的是 V3/R1 的 SM90 FP8 grouped GEMM，所以公式里写成 FP32 scale；如果读当前仓库的 SM100/FP4 代码，看到 packed UE8M0、FP8xFP4 或 Mega MoE 相关实现，不要把它们反向套回 V3 的 Hopper 路径。3.4 Weight scale 与 activation scale 的方向不同另一个容易读错的点是：RHS 权重 scale 和 LHS activation scale 不是同一种存储方向。权重是离线准备的，vLLM 在加载 FP8 MoE weight 时会把 w13_scale / w2_scale 转成 DeepGEMM 需要的 scale layout；activation scale 是 runtime quant 产生的，必须尽量由 quant kernel 直接产出 column-major TMA-friendly stride。scale来源形状语义什么时候准备为什么重要A_scaledispatch 后 activation quant 或 SwiGLU quantM × ceil(K/128)，按 LHS token 行变化runtime，每 step / 每 batch要和 TMA load A 一起进 shared memoryB_scalecheckpoint / weight loader 后处理E × ceil(N/128) × ceil(K/128)load-time 或 warmup专家权重固定，适合提前变换这就是 DeepGEMM README 里强调“LHS scaling factor must be TMA-aligned and transposed”的原因。它不是格式洁癖，而是为了让 mainloop 里读取 scale 的地址模式与 TMA / shared memory pipeline 对齐。4 · Hopper SM90 Kernel：TMA 生产者、WGMMA 消费者与 scheduler4.1 五层结构：从 grid 到 WGMMADeepGEMM 的 SM90 FP8 kernel 可以从外到内读成五层：JIT config 先决定 block shape / stages / multicast；scheduler 把 persistent CTA 映射到 (m_block,n_block,group)；TMA warp group 把 A/B/scale 搬进 shared memory；math warp group 发 WGMMA；最后用 CUDA core 做 scale promotion 和输出。 Draw.io 图表 · 可编辑 figures/F6.drawio SM90 DeepGEMM kernel 的五层阅读法：shape config、scheduler、warp group、pipeline stage、WGMMA。 4.2 Warp specialization 与 setmaxnregHopper 上的核心设计是 warp specialization：TMA warp group 是 producer，math warp groups 是 consumers。它们通过 ClusterTransactionBarrier 和 full/empty phase 协作，TMA 负责把下一段 K tile 搬进 ring buffer，math 负责消费当前 tile。 Draw.io 图表 · 可编辑 figures/F7.drawio Warp specialization 时间线：TMA copy、WGMMA、scale promotion 和 empty/full barrier 交替推进。 setmaxnreg 的作用是把 register file 从“平均分配”改成“按角色分配”。TMA warp group 不需要大量 accumulator，寄存器可以降到 40 左右；math warp group 需要保存 WGMMA accumulator 和 scale promotion 中间量，寄存器可以提高到 232 左右。常见 SM90 路径的寄存器预算$$ R_{\mathrm{CTA}} = 128\cdot R_{\mathrm{TMA}} +2\cdot128\cdot R_{\mathrm{MMA}} = 128\cdot(40+2\cdot232) = 64,512 <65,536. $$这解释了为什么它能在单个 SM 的 register budget 内同时容纳一个 TMA warp group 和两个 math warp groups。优化前如果每个 warp 都按 math 角色配置，就会把 RF 压爆或者降低 occupancy；优化后 producer 少拿寄存器，consumer 多拿寄存器，资源与职责对齐。4.3 TMA descriptor、swizzle 与 multicastTMA descriptor 把 GMEM tensor 的形状、stride、box size、swizzle 等信息变成硬件可直接使用的 copy 描述符。DeepGEMM host 侧为 A、B、D、scale 准备 descriptor，kernel 内部通过 cute::prefetch_tma_descriptor 和 tma::copy 发起异步拷贝。TMA multicast 是另一层优化：当多个 CTA 需要相同 A tile 或 B tile 时，可以把一次 global load multicast 到 cluster 内多个 CTA 的 shared memory。SM90 上这件事还要受 peer CTA 是否存在、grouped layout 是否同 expert 等条件约束；contiguous layout 如果两个相邻 M tile 属于不同 expert，就不能盲目 multicast B。4.4 Scheduler：waves、非对齐 block 与 rasterizationDeepGEMM 的 get_best_configs 思路可以压缩成四个问题：这个 shape 需要多少 waves；shared memory 能放多少 stages；TMA multicast 是否值得；block rasterization 如何让 L2 更容易复用。JIT 的价值在这里非常具体：一旦 shape 和 block 成为编译时常量，很多分支、loop bound 和 descriptor 形态都能静态化。 Draw.io 图表 · 可编辑 figures/F8.drawio Shape-dependent config：DeepGEMM 会为了贴近 SM 数选择非 2 的幂 block，并用 swizzle/rasterization 提升 L2 局部性。 非对齐 BLOCK_N 的并行度账本$$ \begin{array}{lll} B_N=128 &:& \left\lceil 256/128 \right\rceil\cdot\left\lceil 7168/128 \right\rceil=112,\\[0.55em] B_N=112 &:& \left\lceil 256/128 \right\rceil\cdot\left\lceil 7168/112 \right\rceil=128. \end{array} $$H100/H800 是 132 SM 级别，112 个 CTA 会留下明显并行度空洞；128 个 CTA 更接近硬件规模。这里的“非 2 的幂 block”不是怪癖，而是用 tile 数去匹配 persistent kernel 的 wave 结构。代码里的 scheduler 还做了两件重要但不显眼的事。第一，contiguous layout 通过 grouped_layout + m_block_idx * BLOCK_M 找到当前 M tile 属于哪个 expert，由此把 RHS 指针切到对应 expert；第二，masked layout 会按 masked_m[e] 逐 expert 计算 ceil_div(M_e,BLOCK_M)，所以 padding 行不会参与 tile 数统计。两者都把“动态 expert token 数”压缩成了 scheduler 的 tile 索引问题。4.5 JIT config：为什么 shape 要进入编译期DeepGEMM 的 JIT key 包含
