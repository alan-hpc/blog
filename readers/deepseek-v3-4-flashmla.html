2026-05-04

原文来源 DeepSeek V3 推理系统深度解析（四）：MLA Prefill/Decode 分流，FlashMLA 与 Latent KV Kernel · Yang Min · 2026

FlashMLA 解决的问题是：MLA 的 latent KV 压缩节省了 KV cache 存储，但把 decode 时的 attention 计算推向了一个标准 FlashAttention 无法高效处理的形状。标准 MHA decode 的 Q 形状是 [B, 1, H, d]，逐 head 计算；MLA decode 的 Q 需要先把 latent 解压还原，再按压缩后的 KV 做 attention，tile 形状和 warp 分配策略都和 MHA 不同。FlashMLA 针对这个形状重写了 tile scheduler，让 decode 阶段的内存带宽利用率不因 latent 解压而下降。

0 · 本篇问题

这一篇先把 attention family 的变化讲顺：MHA/GQA/MQA 都是在“完整 K/V 空间”里改变 KV head 共享方式，而 MLA 进一步把历史 cache 压到 latent 空间；真正让 decode 热路径 变轻的是后面的 K/V absorb，把历史侧恢复 K/V 的线性层搬到当前 query 与 output 两侧。弄清这条线以后，再看 vLLM 里的 prefill/decoding 分流，以及 FlashMLA 的 MQA transpose、persistent task division、warp-group seesaw、资源占用与吸收边界。

MLA 最早在 DeepSeek-V2 论文里作为模块契约提出，V3/R1 继续沿用这条 cache 思路。先看论文原图：它把输入 $h_t$ 分成 query latent $c_t^Q$、KV latent $c_t^{KV}$ 和 RoPE 分支；斜线阴影标出的不是完整 K/V，而是推理阶段真正需要缓存的 latent KV。这个图回答“MLA 在模型层想减少什么”。

再看 MLA Module 的全局图：一条 hidden state 进入 MLA 后，会分成 query latent、KV latent 与 RoPE 分支；prefill 可以把这些分支恢复成 per-head Q/K/V，而 decode 会尽量让历史 KV 留在 latent cache 里。这张图先建立“MLA 是一个 模块契约，不只是 FlashMLA kernel”的全局概念。

1 · MLA 与 Absorb

把视角从 kernel 拉回到 cache：decode 每生成一个新 token，都要用当前 token 的 query 去扫描所有历史 token 的 K/V。于是优化可以问两个很具体的问题：

  1. 每个历史 token 在 cache 里到底存几条 K/V stream？
  2. 这些历史状态在每一步 decode 时，是否还要恢复成完整 K/V？

MHA、GQA、MQA 回答的是第一个问题：减少或共享完整 K/V stream。MLA 回答的是更深一层： 不要在 cache 里存完整 K/V，而是存一个能恢复 K/V 的 latent 状态。Absorb 再往前一步： 让图中 kv_b_proj 的 K/V 上投影不再对每个历史 token 执行，decode 热路径上连 “恢复历史完整 K/V”这件事也不要做。

1.1 完整 K/V：MHA/GQA/MQA

普通 attention 的语义很朴素：hidden state $h$ 经过三组投影得到 $Q,K,V$，再算 score、softmax 和输出。MHA/GQA/MQA 都没有改变这个语义，它们只是在 K/V head 的数量上 做共享。

形态历史 cache 里的一条 token它解决什么它没有解决什么
MHA$H_Q$ 组完整 K/V每个 Q head 都有独立 K/V，表达最直接KV cache 最大
GQA$H_{KV}$ 组完整 K/V，且 $1多个 Q heads 共享一组 K/V，cache 变小存的仍是完整 K/V
MQA1 组完整 K/V所有 Q heads 共享同一条 K/V stream存的仍是完整 K/V，只是 stream 数最少

所以从 MHA 到 GQA/MQA，本质是把 cache 里的“完整 K/V stream 数量”降下来：

MHA/GQA/MQA：cache 缩小来自 K/V head 数

$$ \mathrm{KVCacheBytes} \propto T\cdot H_{KV}\cdot(d_k+d_v). $$

这里 $T$ 是历史长度，$H_{KV}$ 是需要保存的 K/V head 数。MHA 里 $H_{KV}=H_Q$； GQA 让 $H_{KV}$ 变小；MQA 让 $H_{KV}=1$。但是无论怎么共享，cache 里存的对象仍然是 已经展开好的 $K,V$。

这一点很重要，因为 MLA 不是继续减少 $H_{KV}$，而是换了 cache 里的对象。

普通 attention 的 baseline 账本

$$ Q=hW_q,\qquad K=hW_k,\qquad V=hW_v, $$$$ QK^T=(hW_q)(hW_k)^T=hW_qW_k^Th^T, $$$$ P=\operatorname{softmax}(QK^T),\qquad O=PV=P(hW_v),\qquad Y=OW_o=PhW_vW_o. $$

这张账本后面还会用到：score 侧有 $K^T$，output 侧有 $VW_o$。只有线性层能被移动， softmax 语义不能动。

1.2 Latent cache：MLA 存什么

MLA 把问题换了一个层次。历史 token 不再直接写入完整 $K,V$，而是先投到低维 $c_{KV}$；需要参与 attention 时，再用两个上投影矩阵恢复出 $K,V$。

可以把它理解成一个“cache 合约”：

cache 写入时cache 里留下什么如果需要完整 K/V
hidden state $h$ 进 MLA$c_{KV}=hW_{d,KV}$，以及 RoPE 分支 $k^R$$K=c_{KV}W_{u,k}$，$V=c_{KV}W_{u,v}$

这一步的意义是容量：cache 不再跟完整 K/V 的维度绑定，而是跟 latent 维度绑定。对长上下文 推理来说，历史 cache 是反复被读的最大状态之一，先把它压小是 MLA 的第一层收益。

但 naive MLA 还有一个问题：decode 时历史 token 很多，当前 token 很少。如果每一步都 先把所有历史 $c_{KV}$ 恢复成完整 $K,V$，就会把“cache 变小”的收益又花在恢复计算和 中间张量物化上。

MLA naive 形态：cache 小了，但 compute 还没省

$$ c_{KV}=hW_{d,KV},\qquad Q=hW_q, $$$$ K=c_{KV}W_{u,k},\qquad V=c_{KV}W_{u,v},\qquad Y=\operatorname{softmax}(QK^T)VW_o. $$

所以到这里要分清两件事：

  • MLA latent cache 解决“历史状态存得太宽”。
  • Absorb 解决“decode 每步都恢复历史 K/V”。

1.3 Absorb 前后：公式与图中位置

Prefill 和 decode 的边界在这里出现。Prefill 里 prompt 长度为 $T$，query 和 key 都是长序列； 恢复 per-head K/V 后交给 MHA/FlashAttention，工程路径很成熟。Decode 则完全不同：当前 query 往往只有 1 个 token，但历史 cache 有 $T$ 个 token。Absorb 的全部收益都来自这个 不对称性：把原来发生在历史侧、会乘上 $T$ 的上投影，搬到当前 query 或 output projection 这一侧。

先写不吸收的 decode 路径。这里 $C_{KV}$ 是所有历史 token 的 latent cache，$K^R_{\mathrm{hist}}$ 是 RoPE 分支；kv_b_proj 的两个上投影会先把历史侧恢复成完整 noPE $K/V$：

吸收前：历史侧恢复完整 K/V

$$ \begin{array}{lll} K^C_{\mathrm{hist}} &=& C_{KV}W_{UK},\\[0.35em] V^C_{\mathrm{hist}} &=& C_{KV}W_{UV},\\[0.35em] S &=& Q^C(K^C_{\mathrm{hist}})^T + Q^R(K^R_{\mathrm{hist}})^T,\\[0.35em] Y &=& \operatorname{softmax}(S)V^C_{\mathrm{hist}}W_O . \end{array} $$

吸收后，kv_b_proj 不再对每个历史 token 物化完整 $K/V$。K-side 把 $W_{UK}^T$ 搬到 query noPE 路径；V-side 把 $W_{UV}$ 合进 output projection。RoPE 分支不动：

吸收后：score / PV 都直接读 latent cache

$$ \begin{array}{lll} \widetilde Q^C &=& Q^C W_{UK}^T,\\[0.35em] S &=& \widetilde Q^C C_{KV}^T + Q^R(K^R_{\mathrm{hist}})^T,\\[0.35em] O_{\mathrm{latent}} &=& \operatorname{softmax}(S)C_{KV},\\[0.35em] Y &=& O_{\mathrm{latent}}(W_{UV}W_O). \end{array} $$

F2 比抽象定位图更适合作为 absorb 解释图。左边是 MLA decode 源流程，右边是 TRTLLM-MLA / FlashMLA 视角：它直接把“从哪吸收、吸收到哪、cache 里到底还剩什么” 摆在同一张图里。

图中位置具体含义对应公式
右侧 kv_cachecache 里拼的是 KV_C_Normed 与 K_PE，也就是 $[C_{KV};K^R]$，不是完整 per-head $K/V$FlashMLA 直接读 $C_{KV}$ 和 $K^R_{\mathrm{hist}}$
右侧 W_UK_TK-side absorb 的目标位置：当前 token 的 noPE query 先乘 $W_{UK}^T$，再与 RoPE query 拼成 FlashMLA 的 Q$\widetilde Q^C=Q^CW_{UK}^T$
左侧顶端 W_UV / W_UV Out TransposeV-side absorb 的目标位置：FlashMLA 先输出 latent 维度，attention 外侧再恢复到模型输出维度$Y=O_{\mathrm{latent}}(W_{UV}W_O)$
K_PE / q_pe RoPE 分支位置编码不是共享线性上投影，不能同样吸收；它仍然拼回 score 输入$Q^R(K^R_{\mathrm{hist}})^T$ 保留在 score 中

两个矩阵都吸收后，decode 阶段不需要把 cache 恢复成完整 K/V。换句话说，latent cache 可以直接作为 FlashMLA 看到的 K/V 载体，只是 query 和 output 两侧已经提前接住了 原本属于 $W_{UK}$ 与 $W_{UV}$ 的线性变换。

第 1 节的结论可以压成三句话：

  • MHA/GQA/MQA：还是完整 K/V，只是 K/V stream 数不同。
  • MLA：cache 改存 latent，但 naive 路径仍可能恢复完整 K/V。
  • MLA + absorb：decode 热路径直接读 latent cache，完整历史 K/V 不再物化。

只有最后一步成立，FlashMLA 才能把 decode attention 写成一个直接读 latent KV cache 的 kernel。RoPE 分支没有被吸收，它仍然作为额外维度拼回 score；被 FlashMLA 直接消费的 latent cache 主要对应 noPE 的 K/V 主体。

2 · Prefill / Decode 分流

2.1 Prefill 路径

MLA prefill 源图里的关键不是某个算子名字，而是阶段选择：prefill 的 prompt 长度大，full attention 的 $T\times T$ 结构明显，恢复 per-head $K,V$ 后交给 已有 MHA/FlashAttention 路径更自然。cache 写入时仍然只留下 latent 状态，为后续 decode 服务。

Prefill 阶段的 MLA 张量展开

$$ c_Q,c_{KV},k^R = hW_{qkv,a},\qquad Q=[q^C;q^R]\in\mathbb{R}^{B\times T\times128\times192}, $$$$ K=[k^C;k^R]\in\mathbb{R}^{B\times T\times128\times192}, \qquad V=v^C\in\mathbb{R}^{B\times T\times128\times128}. $$

调研文档里的实现 trace 以 V2-Lite 为例，$\mathrm{head\_num}=16$、$\mathrm{seq\_len}=68$； 换到 V3/R1 主模型时，head 数扩大到 128，但路径语义不变：prefill 面向长 prompt 吞吐，恢复 per-head Q/K/V 后交给 full attention；cache 写入仍保留 $c_{KV}+k^R$。

2.2 Decode 路径

Decode 的输入只有当前 token，历史全在 cache 里。MLA decode 源图 把当前 query 先经过 q_b_proj 得到 $q^C,q^R$，再用 $W_{UK}^T$ 把 noPE query 投到 latent KV 的维度；cache 侧直接读取 $[c_{KV};k^R]$，形成 MQA 视角下的 FlashMLA 输入。实现 trace 里的 decoding 例子假设 batch=1、历史 cache sequence length=68；F2 右侧虚线框中的部分就是后续 FlashMLA kernel 要做的工作。

这段需要先抓住两个边界。第一，矩阵吸收之后，cache 已经可以直接作为 attention 的 K/V 操作数，decode 不再先恢复完整的 per-head $K,V$。第二，每个历史 token 的 $c_{KV}$ 会被 128 个 query heads 共享，所以 KV 侧 head 数是 1。换句话说， FlashMLA 不是普通 MHA，而是一个 KV 共享的 MQA：多条 query head rows 同时看同一条 latent KV stream。

Decode 阶段的 MQA 视角

$$ Q_{MQA}=[q^C W_{UK}^T;q^R]\in\mathbb{R}^{B\times128\times576}, \qquad KV_{cache}=[c_{KV};k^R]\in\mathbb{R}^{T\times1\times576}. $$

到这里，FlashMLA 的输入形状就可以写成三条张量契约：

张量形状语义例子
q$[B,1,H,D_{qk}]$$[2,1,128,576]$
k$[B,T,1,D_{qk}]$$[2,2048,1,576]$
v$[B,T,1,D_v]$$[2,2048,1,512]$

这里最容易忽略的是 q 的 transpose。常规 MHA 是一个 Q head 对应一个 K/V head； MQA 则是多个 Q heads 对应一个 K/V head。FlashMLA 会把 $q$ 从 $[B,1,H,D_{qk}]$ 转成 $[B,H,1,D_{qk}]$。在例子里就是：

MQA transpose：把 head 轴改写成 query-row 轴

$$ q_{raw}\in\mathbb{R}^{2\times1\times128\times576} \quad\Longrightarrow\quad q_{mqa}\in\mathbb{R}^{2\times128\times1\times576}. $$

这个转换不改变 score 的数学含义，只改变 kernel 看到的并行形状：128 个 query heads 被解释成 $S_q=128$ 行 query，而 $k,v$ 仍然保持 $H_{KV}=1$。因此它很像 seq_len=128 的 extend attention，只是这 128 行不是 128 个不同 token，而是同一个 decode token 的 128 个 heads。后面的 block_m=64、两个 Q tiles、persistent task division，都是从这个形状改写自然推出来的。

3 · FlashMLA Kernel

3.1 任务划分

源文这里分三步讲：persistent kernel、Q 的划分、K/V 的划分。H100/H800 有 132 个 SM，所以 FlashMLA 直接启动 132 个 resident CTAs，让一个 block 基本接管一个 SM 上的 shared memory、register 与 warp resources。示例 grid 写成 $[2,1,66]$，乘起来 正好是 132：第一维对应 Q 侧两个 $block_m=64$ 的 query tiles，第三维把这些常驻 CTA 摊到所有 SM 上。

Q 侧的来源就是前面的 MQA transpose。一个 batch 的当前 decode token 现在看起来是 $[128,576]$，因为 128 个 heads 已经被改写成 query-row 维度。block_m=64 意味着 一个 CTA 一次加载 $[64,576]$ 的 Q tile，所以同一个 decode token 的 Q 会被两个 CTA 覆盖。

K/V 侧按 sequence 维度切，block_n=64。K 和 V 实际来自同一段 latent cache buffer， 这里只要看 K 的切法即可：每个 batch 的历史序列被切成若干 64-token tiles，然后所有 batch 的 tile 数累加成全局 work units。metadata 记录每个 CTA 要处理的 batch idx 范围与 token range。来到某个 sequence 尾部时，如果剩余 tile 不足以填满一个 SM， scheduler 会把下一个 batch 的 tile 也塞进同一个 CTA。结果就是：一个 CTA 可能处理 多个 batch，同一个 batch 也可能被多个 CTA 计算出 partial outputs。

3.2 Kernel 主循环

进入 kernel 后，每个 CTA 一次处理某个 batch 的一段 partial result。它先加载一次 $Q\in\mathbb{R}^{64\times576}$，然后沿着 metadata 给出的 K/V tiles 循环： 每轮加载 $K,V$ 的一个 tile，计算 $QK^T$，更新 online softmax 的 scale，再对已有 $O$ 做 rescale 并累加新的 $PV$。循环结束时，这个 CTA 得到的不是全局最终输出，而是 当前 split 的 partial $O$、row max 与 row sum。

把这个过程拆开看，计算图里其实有三个状态在循环里穿针引线。第一，$Q$ tile 只加载一次并留在 shared memory；第二，K/V tile 按 iter 0, iter 1, iter 2, ... 不断流过同一个 CTA；第三，online softmax 不只产出 $P$，还维护 row max 与 row sum， 让新一轮 score 进来时可以先把旧的 $O$ rescale 到同一归一化尺度，再累加新的 $PV$。 所以 block 内的主循环不是“GEMM1 -> softmax -> GEMM2”三步这么简单，而是：

Block 内 online attention loop

$$ S_i=QK_i^T,\qquad (P_i,m_i,\ell_i)=\mathrm{online\_softmax}(S_i,m_{i-1},\ell_{i-1}), $$$$ O_i=\alpha_i O_{i-1}+P_iV_i . $$

这也解释了源文里“最后需要一个额外 kernel”的原因。任务划分阶段为了负载均衡，可能 把同一个 batch 的 KV 范围分给多个 CTA；这些 CTA 分别写回 partial result。后处理 kernel 再用同一套 online-softmax 合并规则，把多个 split 的 $O$、max、sum 规约成 最终 decode 输出。

因此 FlashMLA 的速度不是“单个 block 算完一切”，而是两个层次配合：block 内用 online softmax 保证分块 attention 数学等价；block 间用 split-reduce 把负载均衡造成的 partial outputs 合并回来。

3.3 Seesaw 调度

对照 FlashMLA main 的 splitkv_mla.cuh，这里不能理解成“WG1 负责加载，WG0 负责 计算”。一个 CTA 仍是 256 threads，也就是 8 个 warps / 2 个 warp groups；TMA copy 由特定 thread/lane 发起，Q 先被搬到 sQ，K/V 用 sK0/sK1 两个 shared buffers 承接。真正的 seesaw 是以两个 KV blocks 为一组：WG0 维护左半输出 $O_L$，WG1 维护右半输出 $O_R$，同时通过 shared memory 交换 softmax 后的 $P$ 与 rescale 系数。

更贴近实现的主循环可以写成：

  1. CTA 预装当前 batch 的 $Q$，并预装 $K_0/K_1$ 到 sK0/sK1；每个 $64\times576$ K block 又拆成 9 个 $64\times64$ TMA tiles，GEMM 会边等边算。
  2. WG0 计算 $P_0=QK_0^T$，WG1 计算 $P_1=QK_1^T$。这是两个 KV blocks 的并行入口， 不是单纯一个 WG 算、另一个 WG 只搬运。
  3. WG0 对 $P_0$ 做 online softmax，更新 shared sM，写出 sScale0，rescale $O_L$ 并计算 $O_L \mathrel{+}= P_0V_{0L}$。
  4. WG1 等 sScale0，再把 $P_1$ 合进同一个 running max，写出 sScale1，rescale $O_R$ 并计算 $O_R \mathrel{+}= P_1V_{1R}$。
  5. WG0 等 sScale1 后，把 $P_0$ 乘上新的 scale 并写入 sP0；WG1 读取 sP0， 补上 $O_R \mathrel{+}= P_0V_{0R}$。
  6. WG1 把 $P_1$ 写入 sP1；WG0 读取 sP1，补上 $O_L \mathrel{+}= P_1V_{1L}$。
  7. 当 K0/K1 不再被当前 GEMM 使用时，对应 warp group 立刻发起下一轮 K/V TMA copy，把内存延迟藏在当前 PV/QK 计算后面。

这样一轮结束后，$O_L$ 与 $O_R$ 合起来等价于对 $K_0,K_1$ 两个 KV blocks 做了一次 标准 online attention update。需要注意的是，每轮在 shared memory 里同步的是 running max sM、rescale 系数 sScale0/sScale1 和 softmax 后的 sP0/sP1；row sum rL 主要留在各自 warp group 的 register 里，直到 tile loop 结束才用 sL_reduction_wksp 与 __syncthreads 做跨 WG 归约。因此“每轮同步 row max 与 row sum”这个说法不够准确。

流程会以两个 KV blocks 为步长重复，直到当前 batch 的 tile range 完成并写回 global；如果 metadata 让这个 CTA 接着处理下一个 batch，kernel 会提前发起下一批 Q 的 TMA copy，再进入同样流程。NamedBarrier 负责 sScale/sP 的生产者-消费者 handoff，__syncthreads 主要出现在 barrier 初始化、最终 row-sum 归约、 store/batch 边界这些 CTA-wide 阶段。

Ping-Pong / Cooperative 的边界。 这里容易和 CUTLASS Hopper 的两个 persistent schedule 名词混起来。Cooperative kernel 的核心是两个 consumer warp groups 协作同一个 output tile，通常把 tile 切成几部分来降低单个 WG 的 register pressure；Ping-Pong kernel 则让两个 consumer WGs 处理不同 output tiles，一个 WG 做 epilogue 时，另一个 WG 继续跑 MMA，用 ordered barrier 维持 mainloop / epilogue 的顺序。

FlashMLA 的 seesaw 不是把其中一个名字直接照搬过来。它像 Cooperative：两个 WGs 分摊同一个逻辑输出 $O=[O_L\mid O_R]$，解决 $64\times512$ accumulator 带来的 register 硬约束；它也像 Ping-Pong：TMA copy、$QK^T$、softmax、$PV$ 和下一块 K/V copy 被穿插起来，尽量不让 Tensor Core 等数据。但它又不能像普通 GEMM Ping-Pong 那样把两个 WGs 完全交给不同 output tiles，因为 attention 的 online-softmax 状态 sM/rL 与同一个 $O$ accumulator 必须保持同一条 row 语义。

源码里的 warpgroup_cooperative_qkt_gemm、warpgroup_cooperative_pv_gemm_localP 和 warpgroup_cooperative_pv_gemm_remoteP 也要这样读：这里的 cooperative 是 128-thread warp group 内一起发 WGMMA、一起维护 tile fragment，不等同于 CUTLASS Cooperative schedule 这个整核调度名称。FlashMLA 真正的整核调度名称仍然是 seesaw： 用 sScale0Ready/sScale1Ready/sP0Ready/rO1sP0sV0RIssued 这些 handoff 点，把同一个 online attention update 拆成左右半输出和两个 KV blocks 的交错执行。

再细一点说，cooperative 不是指每一个 $PV$ half-GEMM 都由两个 WGs 一起发射。 P0V0L 与 P1V1R 是各自 WG 的 local-P half GEMM，不需要等对方的 $P$；P0V0R 与 P1V1L 才是把 softmax 后的 $P$ 写入 shared memory，再由另一个 WG 消费的 remote-P half GEMM。

这里确实要先同步，但不是在 GEMM2 开头做一次粗粒度的 CTA-wide 同步，而是在 producer-consumer 边界用 NamedBarrier 卡住读写顺序：WG1 读 sP0 前必须等 WG0 save_rPb_to_sP(sP0)、fence_view_async_shared() 之后发出的 sP0Ready； WG0 读 sP1 前没有单独的 sP1Ready，而是等 WG1 完成 sP0 remote half-PV 发射后 发出的 rO1sP0sV0RIssued。这个点发生在 WG1 保存并 fence sP1 之后，因此也保证 WG0 看到的 sP1 已经可被 async proxy 消费。所以图里的 cooperative 框只应该罩住 save sP → barrier → remote half-PV 这段 shared-P handoff；local half-PV 仍然留在 各自 warp group 内。

3.4 Hopper 数据通路

这里还要保留源文的 Hopper 约束。Hopper WGMMA 通过 4 个 warps 操作 SM 上的 Tensor Cores；WGMMA 要求操作数 B 必须在 shared memory，操作数 A 可以在 shared memory 或 register，输出 accumulator 总是在 register。在 FlashMLA 的 GEMM1 中， Q/K 都从 shared memory 进入 WGMMA；在 GEMM2 中，$P$ 在 register，$V$ 在 shared memory，$O$ 仍在 register。

两组 warp group 不是固定的“一个搬运、一个计算”，而是围绕两个 KV blocks 分摊 GEMM1、softmax 和 GEMM2：WG0 维护 $K_0/V_0$ 相关的 $P_0$ 与 $O_L$，WG1 维护 $K_1/V_1$ 相关的 $P_1$ 与 $O_R$，再用 shared memory 做交叉补项。double buffer、 数据加载和计算 overlap、以及写回策略都服务于同一个目标：不要让 WGMMA/Tensor Core 因等待 K/V 或搬运 O 而空转。

因此，内核即便 compute-bound，也不能忽略 memory latency。FlashMLA 对 $64\times576$ 的 K block 不做一次大搬运，而是拆成 9 个 $64\times64$ TMA copy。 第一个 copy 完成后，第一段 GEMM 就能启动；后面的 copy 与 GEMM 继续流水推进。

4.2 Register 账本

register 侧要分 GEMM1 和 GEMM2 两笔账看。GEMM1 计算的是 $QK^T\rightarrow P$， 使用的 MMA tile 可以写成 $m\times n\times k=64\times64\times16$。因为 $D_{qk}=576$，所以 K 维会被拆成 $576/16=36$ 个循环；在每个循环里，$Q$ tile 和 $K$ tile 都从 shared memory 进入 WGMMA，真正占 register 的主要是输出 accumulator $P$。

因此每个 GEMM1 的 $P$ tile 需要 $64\times64=4096$ 个 FP32 accumulator。当前实现里 WG0/WG1 分别为 K0/K1 维护自己的 P0/P1，每个 tile 落在一个 warp group 的 128 个线程上，平均就是 $4096/128=32$ registers/thread，这个压力并不夸张。

GEMM2 才是问题的核心。它计算的是 $PV\rightarrow O$，其中 $P$ 是 $64\times64$， $V$ 是 $64\times512$，输出 $O$ 是 $64\times512$。$V$ 仍然从 shared memory 进入 WGMMA，但 $O$ accumulator 必须留在 register。

4.3 吸收边界

到这里容易产生一个误解：既然 K/V absorb 能把历史 cache 保持在 latent，为什么不在所有 阶段都吸收？边界其实来自 decode 的非对称性。decode 时当前 query 很少，历史 KV 很长； 把上投影从“每个历史 token”搬到“当前 query / output 外侧”，收益非常直接。prefill 时 query token 也很长，完整 $T\times T$ attention 本身就是主计算，恢复 per-head K/V 后交给成熟的 MHA/FlashAttention 路径反而更自然。

先看不吸收的 decode 热路径。cache 虽然存的是 $c_{KV}$，但每一步仍然会把所有历史 token 恢复成完整 K/V：

不吸收：每步都恢复历史 K/V

$$ K_{\mathrm{hist}}=C_{KV}W_{u,k},\qquad V_{\mathrm{hist}}=C_{KV}W_{u,v}, $$$$ Y=\operatorname{softmax}(QK_{\mathrm{hist}}^T)V_{\mathrm{hist}}W_o. $$

这里的浪费不是数学错，而是位置错：$C_{KV}$ 有 $s_k$ 行，decode 每一步都围着历史 cache 做上投影；当前 token 只有 $s_q$ 行，通常 $s_q\ll s_k$。吸收以后，K 侧把 $W_{u,k}^T$ 合进当前 query，V 侧把 $W_{u,v}$ 合进 output projection：

吸收：历史 cache 不再恢复完整 K/V

$$ Q' = QW_{u,k}^T,\qquad O_{\mathrm{latent}}=\operatorname{softmax}(Q'C_{KV}^T)C_{KV}, $$$$ Y=O_{\mathrm{latent}}(W_{u,v}W_o). $$

这个改写给出一条实用边界：

场景是否吸收原因
Prefill / long prompt通常不吸收，恢复 per-head K/V$s_q\approx s_k=T$，full attention 吞吐由成熟 MHA/FlashAttention 路径吃掉；cache 写入仍保持 latent 即可
Decode / single token 或小 $s_q$吸收$s_q\ll s_k$，把历史侧上投影搬到当前 query/output 侧，避免反复恢复完整历史 K/V
RoPE 分支不能简单吸收位置编码不是纯粹的共享线性上投影，真实 MLA 仍要把 $q^R/k^R$ 拼回 score 输入
需要显式完整 K/V 的调试或通用 kernel不吸收或只做部分吸收数学上能移动线性层，不代表现有 kernel/layout 接受 latent KV operand
FlashMLA dense decode吸收后进场kernel 的契约就是 MQA 视角下的 latent K/V：$k=[B,T,1,D_{qk}]$，$v=[B,T,1,D_v]$

因此，吸收不是“MLA 永远应该这样算”，而是 decode 热路径上的位置选择：当历史很长、当前 query 很少，移动线性层能把重复工作从 $s_k$ 侧挪到 $s_q$ 侧；当 prefill 需要处理长 prompt，全序列 full attention 的工程路径更重要。

最后用 roofline 直觉解释为什么吸收后的 FlashMLA 仍可能不是单纯 memory-bound。设 query head 数为 $h_q$，每个 request 的 query token 数为 $s_q$，KV 长度为 $s_k$，并且 $s_k \gg h_qs_q$。MLA decode 的算术强度近似为：

FlashMLA decode 的算术强度

$$ \frac{\mathrm{FLOPs}}{\mathrm{Bytes}} \approx h_qs_q\frac{d_k+d_v}{d_k} \approx 2h_qs_q . $$

H800 SXM5 的实际峰值按官方博客中的节流频率估算约为 865 TFLOPs，带宽约为 3.35 TB/s；compute-bound 的阈值约为 $\frac12\cdot\frac{865}{3.35}\approx128$。V3/R1 decode 不做 Tensor Parallel， $h_q=128$，即使 $s_q=1$ 也已经踩在 compute-bound 边界上。
