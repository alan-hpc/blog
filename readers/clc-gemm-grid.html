前置篇：Wave Quantization 问题

本篇是 GEMM 教程系列第三部分（前两部分聚焦单个 threadblock 的 WGMMA、流水线与 warp 特化），转而从整个 grid 的视角审视 GEMM。在这个尺度上，主要有两类优化：一是利用 threadblock swizzling 和 cluster 最大化 L2 cache 命中率；二是更好地将工作分配到各 threadblock，以充分利用 GPU 的计算资源并实现良好的负载均衡。本篇聚焦后者。

具体来说，我们将讨论一种名为 Stream-K 的分块策略，它解决了Wave Quantization问题——当工作 tile 的总数无法被 SM 数量整除时就会出现这一问题。Stream-K 在 M、N 较小但 K 较大导致标准 tile 分块无法充满 GPU 时同样适用。

NVIDIA GPU 由若干个 SM（流式多处理器）组成，每个 SM 有独立的共享内存、寄存器堆、Tensor Core 等，相互独立运行。理想的工作负载能最大化 SM 间的并行性——均匀分配工作，使所有 SM 在整个 kernel 期间保持忙碌。若部分 SM 比其他 SM 更早完成任务，就会出现负载不均衡。

考虑一个可以被分解为等大小工作单元的计算（如 GEMM 中每个工作单元计算一个 bM×bN 的输出 tile），这些工作单元被分配给各 threadblock，每个 threadblock 在可用的 SM 上完成其任务。若工作单元总数超过可用 SM 数，则工作单元将分多个 wave 处理——每个 wave 对应所有可用 SM 各完成一个工作单元。

Wave Quantization发生在工作单元总数不能被 SM 数整除时。例如，假设有 10 个工作单元和 4 个 SM，执行时间线如下：

当工作单元总数相对于 SM 数量较少时，Wave Quantization会严重拖累性能。例如，在拥有 114 个 SM 的 H100 PCIe GPU 上，115 个工作单元所需的 wave 数与 228 个完全相同——都是 2 个 wave！换句话说，从第 114 个增加到第 115 个工作单元，设备利用率几乎减半。

持久化 Kernel（Persistent Kernels）

为解决Wave Quantization，需要更好的分块与调度方案。传统 kernel 的 grid 大小依赖于问题维度，每个 CTA 处理一个工作单元。在持久化 kernel 中，grid 大小是固定值，通常等于可用 SM 数，使每个 CTA 独占一个 SM。可以用以下 CUDA 代码获取 SM 数：

int num_SMs;
cudaGetDeviceAttribute(&num_SMs, cudaDevAttrMultiProcessorCount, device_id);
dim3 dimGrid(num_SMs);

每个 CTA 持续驻留在其 SM 上，循环处理多个工作单元直到所有工作完成。这种设计给程序员提供了对调度的更大控制权，可以通过tile scheduler（本质上是一个迭代器，告知每个 CTA 下一个工作单元在哪里以及何时停止）来实现更复杂的策略，例如 Stream-K。

分块调度策略

为逐步引出 Stream-K，先来了解几种更简单但存在不足的Wave Quantization处理方案。以下以一个假想的 GPU「Hipparchus H10」（仅有 4 个 SM）为例说明。

数据并行（Data Parallel）

最基础的方案是在 M 和 N 方向均匀切分 tile，以轮询（round-robin）方式分配。这与非持久化的工作 tile 网格 kernel 本质相同，唯一区别是保证了顺序。工作单元之间无依赖，因此称为数据并行工作调度。

直接的改进是增加工作单元数量（减小 tile 大小），从而提高利用率。

然而，减小 tile 大小会带来代价：算术强度（Arithmetic Intensity）下降——每次全局内存访问对应的计算量减少；每个 warp 的指令数减少，降低了流水线延迟隐藏机会；还有 MMA atom 对 tile 大小的约束（如 H100 要求 128×128×16 的 WGMMA atom）。这些权衡没有显而易见的最优解，通常需要结合 CUTLASS Profiler 进行实验。

Split-K

除了在 M、N 方向切分，还可以沿 K 方向切分——在 K 较大时最为有效。Split-K 将每个 tile 沿 K 方向切成固定数量的片段：

Split-K 引入了新的超参数——切分数，存在以下权衡：

  • 切分数增大 → wave 数增多 → Wave Quantization效应减小，SM 利用率可能提升；
  • 切分数增大 → K 方向 tile 变小 → 全局内存访问与计算的比率上升；
  • 切分数增大 → 每个 CTA 的指令数减少 → 延迟隐藏机会降低；
  • 引入了同步与规约开销，Split-MN 中不存在这一成本。

Stream-K

上述策略改善了Wave Quantization，但并未消除它。回到 9 个工作 tile、4 个 SM 的例子，理想情况是每个 SM 运行 2.25 个 wave——这正是 Stream-K 的动机。

Stream-K 为每个 SM 分配一个持久化 CTA，每个 CTA 被分配分数个工作 tile（tile 在 K 方向被切分）。与 Split-K 类似，对于被切分的 tile，协作处理的 CTA 通过全局内存 workspace 的轮转规约合并结果。

与之前策略相比，Stream-K 的优势：

  • 消除量化：无 wave 概念，每个 CTA 处理 2.25 个工作 tile，计算量约为原来的 2.25/3；
  • 保留大 tile 优势：大多数原始 128×128×128 tile 由单个 CTA 完整处理，保持高算术强度；
  • 同步开销低**：早期片段的计算可远早于最终片段，epilogue CTA 无需长时间等待；
  • 代价：需额外全局内存传输以共享部分 tile 数据。

混合 Stream-K（Hybrid Stream-K）

Stream-K 有一个隐藏代价：缓存性能下降。在数据并行调度中，同一行或列的多个输出 tile 同时被处理，对操作数矩阵的请求在时间上对齐，容易命中 L2 cache。

而 Stream-K 引入了相位偏移（skew）：由于每个 SM 从大小不同的部分 tile 开始，它们在同一时刻处理不同的 K 偏移，导致 cache 命中率大幅下降。

解决方法是将持久化 kernel 与普通数据并行 kernel 混合调度：尽量使用数据并行，只保留足够处理Wave Quantization尾部的工作量给 Stream-K。具体来说，将 1 至 2 个完整 wave 的计算分配给 Stream-K 阶段，剩余工作（数量可被 SM 数整除）采用数据并行。

Tile 调度器抽象

由于工作分块与调度的问题在很大程度上与每个 CTA 内部的内存和计算操作是独立的，GEMM 实现（如 CUTLASS）通常将其封装在一个称为 tile scheduler 的抽象中——本质上是一个迭代器，告知每个 CTA 下一个工作单元在哪里以及何时停止。

Tile scheduler 的核心职责：

  • 决定 grid 大小：非持久化 kernel 中与问题大小相关；持久化 kernel 中固定为 SM 数量。
  • 提供工作迭代接口：
for (auto worktile = scheduler.get_initial_tile();
     scheduler.is_valid(worktile);
     worktile = scheduler.get_next_tile(worktile)) {
    auto [m_block, n_block, k_block_start, k_block_stop] = worktile.get_block_coord();
    for (k_block = k_block_start; k_block < k_block_stop; ++k_block) {
        // mainloop
    }
    // epilogue
}

对于 Stream-K 和 Split-K，scheduler 还需处理以下额外职责：分配并管理全局内存 workspace（用于存储部分结果和同步 barrier）；告知每个 CTA 当前 tile 是完整输出还是部分输出；指定哪个 CTA 负责执行 epilogue（最终规约并写入输出张量）。

作者在 GitHub 示例代码中提供了三种调度器：一个简单的非持久化调度器；一个数据并行持久化调度器；以及一个混合 Stream-K 调度器（包含 CUTLASS 部分但非全部优化）。

结论（持久化 Kernel 篇）

本篇讨论了Wave Quantization及其对 GEMM 性能的影响，介绍了持久化 kernel 的概念，并系统梳理了数据并行、Split-K、Stream-K 和混合 Stream-K 四种分块策略。Stream-K 通过消除 wave 边界实现了更均匀的负载分配，是 Hopper 时代 GEMM 内核调度的重要进步。

通过 CUTLASS GEMM API 使用 Stream-K

以下基于 CUTLASS 3.x GEMM API（示例 48）介绍如何启用 Stream-K 调度器。首先指定 GemmKernel 使用 StreamKScheduler：

using GemmKernel = cutlass::gemm::kernel::GemmUniversal<
    Shape<int,int,int>,
    CollectiveMainloop,
    CollectiveEpilogue,
    cutlass::gemm::StreamKScheduler
>;

Stream-K 只支持特定的 mainloop 和 epilogue schedule，需使用 TmaWarpSpecializedCooperative。通过 scheduler arguments 设置分解模式：

using DecompositionMode = typename cutlass::gemm::kernel::detail::
    PersistentTileSchedulerSm90StreamKParams::DecompositionMode;

typename Gemm::GemmKernel::TileScheduler::Arguments scheduler_args;
scheduler_args = { splits, swizzle, raster, DecompositionMode::StreamK };

CUTLASS scheduler 支持四种分解模式：DataParallel、SplitK（需指定切分数）、StreamK、Heuristic（默认，由 CUTLASS 自动选择最优）。

CUTLASS 实现细节

调度方案：CUTLASS 实现了上述混合调度，将最多两个 wave 的工作分配给 Stream-K 阶段，其余采用数据并行，改善 L2 cache 性能。

规约方式：默认使用轮转（turnstile）规约——CTA 0 先写入全局内存 workspace，CTA 1 等待后累加规约，……最后一个 CTA 从 workspace 规约到累加器并执行 epilogue。可选非确定性模式（ReductionMode::Nondeterministic），中间 CTA 原子规约，无需相互等待（但浮点加法不具结合律，结果可能有数值差异）。

Threadblock 光栅化（Rasterization）：持久化 kernel 的优势之一是可以控制 tile 的启动顺序，使同行或同列的 tile 接近同时被处理，提高 L2 命中率。CUTLASS 提供 RasterOrderOptions::AlongM、AlongN、Heuristic（自动选短边方向）。

Swizzling：在光栅化基础上进一步考虑两个维度的邻近性。max_swizzle_size 参数可设为 1（无 swizzle）、2、4 或 8，表示在光栅方向行进多少个 tile 后再向另一维度推进。

Cluster 与 multicast：Hopper 引入的 threadblock cluster 允许同一 GPC 内的 CTA 共享 SMEM，TMA load 可以 multicast 到 cluster 内所有 CTA。CUTLASS 通过以 cluster 为单位（而非单个 tile）构建调度表来优雅处理这一特性，在 Stream-K 阶段也会将 cluster 分组，使每组在同一时刻处理相同的 K 偏移，避免相位偏移导致的 cache 失效。

动机

考虑矩阵乘法（GEMM）问题 $C = AB$，其中 $A \in \mathbb{R}^{M \times K}$，$B \in \mathbb{R}^{K \times N}$，$C \in \mathbb{R}^{M \times N}$。$C$ 的计算通过将问题形状 $(M, N, K)$ 按 tile 形状 $(\text{bM}, \text{bN}, \text{bK})$ 进行切分来并行化，每个 $\text{bM} \times \text{bN}$ 的输出 tile 按如下方式计算：

$$C^{[i,j]} = \sum_k A^{[i,k]} B^{[k,j]}$$

每个工作 tile $C^{[i,j]}$ 必须分配给某个处理单元——具体来说，是 CUDA 执行模型中的一个 CTA 或一组 CTA（cluster）。Tile 调度问题就是确定如何将工作 tile 集合最优地分配到各处理单元上。

本文讨论 Cluster Launch Control（CLC）——NVIDIA Blackwell GPU 上一个硬件支持的特性，用于实现最优的 tile 调度，尤其是在负载均衡方面。为提供背景，我们先梳理几种常见的调度策略及 CLC 所针对的不足，然后深入讲解在 CuTe DSL kernel 中使用 CLC 的实现细节，最后给出 GEMM kernel 的性能对比。

单 Tile 调度（Single Tile Scheduling）

最朴素的 tile 调度方案是启动一个形状为 $(M/\text{bM},\ N/\text{bN})$ 的 cluster 网格，将每个工作 tile 分配给唯一的 cluster。这对负载均衡有利：网格中的 cluster 数量多于 SM 组数，当每个 cluster 退出时，硬件调度器会将排队中的 cluster 分发到空闲的 SM 组上。

然而，这种策略整体上往往不够优，因为每个 cluster 要支付固定的启动开销——流水线初始化、描述符设置等——而这些开销只被单个 tile 摊销。此外，单 tile 调度无法跨工作 tile 进行重叠以隐藏延迟，例如无法将一个工作 tile 的 epilogue 与另一个的 mainloop 重叠执行。

静态持久化 Tile 调度（Static Persistent Tile Scheduling）

另一种方案是使用持久化 tile 调度方案。我们在此简要回顾持久化 tile 调度的概念，详细说明请参阅 Colfax 的前一篇文章。

在持久化方案中，我们启动恰好等于 GPU 可并发调度数量的 cluster 网格。一旦某个 cluster 被启动，它将"持久"驻留在 GPU 上，计算一批工作 tile。例如，假设有 148 个 SM，cluster 大小为 2，则可在 GPU 上并发启动 74 个 cluster。对于包含 512 个工作 tile 的 GEMM kernel，可以对工作 tile 做线性排序，让每个 cluster 计算编号为其索引的每第 74 个工作 tile。

持久化 tile 调度的主要优点是可以将一个 tile 的 epilogue 与下一个 tile 的 mainloop 重叠执行；同时避免了启动新 cluster 的延迟。然而，静态持久化 tile 调度会导致负载不均衡问题。

例如，考虑分组 GEMM（grouped GEMM），它计算一组 GEMM：

$$C_i = A_i B_i, \quad i = 0, 1, \dots, \text{num\_problems} - 1$$

动态持久化 Tile 调度的标准实现

为实现动态持久化 tile 调度，需保证两条性质：

  1. 每个 tile 最终都会被某个 cluster 处理；
  2. 没有任何 tile 被超过一个 cluster 处理。

一种标准策略是维护一个全局原子计数器（即信号量锁），跟踪下一个未分配的 tile。当 cluster 完成当前 tile 后，对该计数器执行原子 fetch-and-increment 操作来认领下一个 tile 的索引。每个 cluster 持续申请工作，直到返回的 tile 索引大于等于总 tile 数为止，从而保证性质 (1)。由于原子操作具有线性化特性，每个 cluster 获得唯一的 tile 索引，从而保证性质 (2)。这一策略例如在 quack tile scheduler 中有所实现。

quack 的实现位于 _fetch_next_work_idx（PersistenceMode.DYNAMIC 分支）。每个 cluster 的 scheduler warp 由 lane 0 负责执行原子操作，结果通过 shuffle_sync 广播给 warp 内所有 32 条 lane：

# quack/tile_scheduler.py  L393–L409  (PersistenceMode.DYNAMIC)
elif const_expr(params.persistence_mode == PersistenceMode.DYNAMIC):
    next_work_linear_idx = Int32(0)
    if cute.arch.lane_idx() == 0:
        # 固定 M：atomic_inc 在到达 limit 时自动归零
        next_work_linear_idx = num_persistent_clusters + utils.atomic_inc_i32(
            cute.size(params.problem_shape_ncluster_mnl) - 1,   # limit = 总 tile 数 - 1
            params.tile_count_semaphore,                         # 全局内存中的计数器
        )
        # varlen_m（序列长度可变）：改用 atomic_add，kernel 结束后手动归零
        # next_work_linear_idx = num_persistent_clusters + utils.atomic_add_i32(1, ...)
    return cute.arch.shuffle_sync(next_work_linear_idx, 0)      # 广播给全 warp

atomic_inc 与 atomic_add 的区别：固定问题尺寸时，atomic_inc_i32(limit, ptr) 在计数器达到 limit 后自动循环归零，不需要 kernel 外部显式清零；而可变 M（varlen）场景因为 tile 总数在运行时才确定，改用 atomic_add_i32 并要求 kernel 结束后将信号量手动复位。lane 0 独占写全局内存、warp 内用寄存器广播——这是最小化全局内存往返次数的经典 warp-level 协作模式。

这种方案简单且与体系结构无关，但并非没有缺陷：所有 cluster 必须反复对同一个全局计数器执行原子操作，这在 cluster 之间引入了一定程度的串行化，并需要反复往返全局内存。此外，全局计数器在每次 kernel 启动前都必须清零（atomic_inc 版本自动处理，atomic_add 版本需要手动）。

幸运的是，Blackwell 提供了动态持久化调度的硬件实现，即 Cluster Launch Control（CLC）。它简化了软件层面动态持久化调度的实现，并提供了若干其他优势，将在下文描述。

Blackwell 的 Cluster Launch Control（CLC）

CLC 是从 Blackwell 架构起支持的动态持久化 tile 调度的硬件实现。它的启动网格与单 tile 调度器相同（即基于问题的工作 tile 数量——见 walkthrough 中对 __compute_grid 的讨论），但第一波活跃的 cluster 会反复尝试"抢夺"尚未启动的 cluster 的工作——取消它们的启动并获取其 tile 坐标，由自己来完成这些工作。因此，第一波 cluster 可能会持续存活并完成所有工作，网格中其他 cluster 可能永远不会被启动。另一方面，CLC 也具有灵活性，可以动态允许 cluster 在未完成所有 tile 的情况下退出，并在之后启动新的 cluster 继续处理（详见"CLC 与并发 kernel 及抢占"一节）。

为什么需要 CLC：静态调度的长尾问题

SM 被抢占有哪些情况？

  • 多 kernel 并发（最常见）：应用同时向多个 CUDA stream 提交 kernel，GPU 调度器可能把同一个 SM 先后分配给不同 kernel，正在运行的低优先级 kernel 就被"切走"了
  • 高优先级 kernel 插入：用 cudaStreamCreateWithPriority 设置了更高优先级的 kernel，Pascal 架构之后支持指令级抢占，不需要等当前 thread block 跑完就能切换
  • MPS 多进程：多个进程共享同一张 GPU 时，不同进程的 kernel 会竞争 SM 资源

不过 CLC 更核心的价值不只是对抗抢占——即使没有抢占，只要 tile 间计算量本来就不均等（比如 Attention causal mask 下靠后的 tile 几乎是空 tile），静态分配下早做完的 SM 也会白白空等。这正是 FA4 引入 CLC 的主要动机。

在 Hopper 的 Static Persistent Scheduling 模型中，每个 output tile 在 kernel launch 时就被静态映射到固定的 SM：persistent CTA 数 = SM 数，每个 SM 负责自己的那份 tile 列表，全程不变。这套模型在 workload 均衡时表现良好，但一旦某个 SM 被更高优先级的 kernel 或上下文切换"抢占"，它手上的 tile 不会自动迁移给空闲的 SM——整个 grid 的完成时间由最慢的那个 SM 决定。Tensor Core 越快，这条"长尾"越显著。

怎么理解 CLC：不是把"软件任务队列 + 全局原子计数器"搬进硬件。更直接的理解是——grid 本身就是任务列表；还没被硬件调度器发射的 cluster，就是还没人领的活；try_cancel 就是一个 SM 把别人还没开始做的活抢过来自己做。失败了就说明所有活都已经有人领了，当前 cluster 正常退出即可。

PTX 指令：try_cancel 与 query_cancel

PTX 层面上用于 CLC 逻辑的主要指令有两组。

第一组是 clusterlaunchcontrol.try_cancel，它发出一个原子请求，尝试取消一个尚未启动的 cluster，并获得编码后的响应数据。然后可以用 clusterlaunchcontrol.query_cancel 对该数据进行解码，以判断取消是否成功，并在成功时获取被取消 cluster 的 tile 坐标（用于"抢夺"其工作）。

clusterlaunchcontrol.try_cancel 的语法如下：

clusterlaunchcontrol.try_cancel.async{.space}.completion_mechanism{.multicast::cluster::all}.b128 [addr], [mbar];

.completion_mechanism = { .mbarrier::complete_tx::bytes };
.space = { .shared::cta };

此指令可以与 TMA 做类比：

  • 与 TMA 类似，只有一个线程应发起 try_cancel 操作。但 TMA multicast 中每个参与的 CTA 各发出一条 TMA 指令，而 try_cancel 整个 cluster 中只能有一个线程发起。多个线程提交 try_cancel 会导致多个 cluster 被取消。
  • 与 TMA 类似，该操作会将一些数据异步写入 SMEM（由 [addr] 指定的地址）。若使用 multicast，数据必须广播给 cluster 中的所有 CTA；而 TMA 可以选择只广播给 cluster 中的一个子集。对于 cluster 内有多个 CTA 的情况（cluster_shape > 1×1×1，如 2-CTA cluster），若 try_cancel 不使用 multicast，则只有发出指令的 warp 拿到了响应；它需要自行从 SMEM 读取响应、计算工作 tile 坐标，再写回 SMEM，让 cluster 内其他 CTA 也能读到。
  • 与 TMA 类似，使用事务屏障（transaction barrier）来跟踪 try_cancel 操作的完成状态。但任何 try_cancel 操作始终传输 16 字节。
  • 作为 cluster 范围的操作，在带 multicast 的 try_cancel 中，需确保 cluster 中其他 CTA 均未退出，以避免未定义行为。

clusterlaunchcontrol.query_cancel 的语法如下：

clusterlaunchcontrol.query_cancel.is_canceled.pred.b128 pred, try_cancel_response;

clusterlaunchcontrol.query_cancel.get_first_ctaid.v4.b32.b128 {xdim, ydim, zdim, _}, try_cancel_response;

clusterlaunchcontrol.query_cancel.get_first_ctaid{::dimension}.b32.b128 reg, try_cancel_response;

::dimension = { ::x, ::y, ::z };

使用方式如下：

  • 观察到 try_cancel 完成后，可以对其返回的 16 字节数据发出 query_cancel 系列指令。PTX 文档将该数据描述为"opaque（不透明）"，编程指南称其为"encoded（编码）"，这意味着 query_cancel 是获取有用信息的唯一方式。
  • .is_canceled 返回一个谓词，表示所请求的取消是否成功。注意：若 .is_canceled 返回 false，则发出除 .is_canceled 之外的 query_cancel 指令会导致未定义行为，因此应始终先检查 .is_canceled。
  • 进一步注意：如果 CTA 已观察到 try_cancel 失败，再次发出 try_cancel 同样会导致未定义行为。因此，在此之后，该 CTA 不能再使用 CLC，只能在处理完当前工作队列后退出。
  • try_cancel 失败通常并不表示错误，而是调度逻辑的一部分——最常见的失败原因是网格中已没有更多未启动的 cluster。
  • .get_first_ctaid 可用于获取被取消 cluster 的第一个 CTA 在网格中的坐标，使用 .v4 可获取全部三个维度（第四个元素内容未指定），也可用 ::dimension 指定某个特定维度。

CLC 实现 Walkthrough（CuTeDSL 示例）

Blackwell CuTeDSL 示例 dense_gemm_persistent_dynamic.py 实现了一个标准的稠密 GEMM，CLC 的 try-cancel 由每个 cluster 中的单个调度 warp 执行，调度 warp 与 cluster 中其他 warp 之间的通信通过 CLC pipeline 处理。每个 CTA 启动的 warp 编号（在 __init__ 方法中定义）如下：

self.epilogue_warp_id = (0, 1, 2, 3)
self.mma_warp_id = 4
self.tma_warp_id = 5
self.sched_warp_id = 6

_compute_grid：网格大小计算

在 __call__ 方法中，通过 _compute_grid 确定 kernel 启动参数中的 grid 变量：

def __call__(...):
    ...
    # Compute grid size
    self.tile_sched_params, grid = self._compute_grid(
        c, self.cta_tile_shape_mnk, self.cluster_shape_mn
    )
    self.kernel(...).launch(
        grid=grid,
        block=[self.threads_per_cta, 1, 1],
        cluster=(*self.cluster_shape_mn, 1),
        stream=stream,
    )

def _compute_grid(
    c: cute.Tensor,
    cta_tile_shape_mnk: Tuple[int, int, int],
    cluster_shape_mn: Tuple[int, int],
) -> Tuple[utils.ClcDynamicPersistentTileSchedulerParams, Tuple[int, int, int]]:
    c_shape = cute.slice_(cta_tile_shape_mnk, (None, None, 0))
    gc = cute.zipped_divide(c, tiler=c_shape)
    num_ctas_mnl = gc[(0, (None, None, None))].shape
    cluster_shape_mnl = (*cluster_shape_mn, 1)

    tile_sched_params = utils.ClcDynamicPersistentTileSchedulerParams(
        num_ctas_mnl, cluster_shape_mnl
    )
    # will round up to a whole number of clusters
    grid = utils.ClcDynamicPersistentTileScheduler.get_grid_shape(tile_sched_params)
    return tile_sched_params, grid

cta_tile_shape_mnk 从 MMA tiler 推导，统一支持 1CTA 和 2CTA MMA 模式：

self.cta_tile_shape_mnk = (
    self.mma_tiler[0] // cute.size(tiled_mma.thr_id.shape),
    self.mma_tiler[1],
    self.mma_tiler[2],
)

网格计算将 C 按 CTA tiler 切分以确定初步网格形状，再将其向上取整到 cluster 形状的整数倍以满足网格的 cluster 可整除性要求。这一计算与单 tile 调度器完全相同，尤其是不涉及 SM 数量。

CLC Pipeline 初始化

与其他标准 GEMM pipeline 一同，CLC pipeline 在 kernel 调用的开头附近创建：

# Initialize clc_pipeline (barrier) and states
clc_pipeline_producer_group = pipeline.CooperativeGroup(pipeline.Agent.Thread)
cluster_size = cute.size(self.cluster_shape_mn)
# 每个 CTA 有 4 个 epilogue warp + 1 个 MMA warp + 1 个 TMA warp
# 每个 cluster 有 1 个 scheduler warp
num_clc_consumer_threads = 32 * (
    1 + cluster_size * (1 + len(self.epilogue_warp_id) + 1)
)
clc_pipeline_consumer_group = pipeline.CooperativeGroup(
    pipeline.Agent.Thread, num_clc_consumer_threads
)
clc_pipeline = pipeline.PipelineClcFetchAsync.create(
    barrier_storage=storage.clc_mbar_ptr.data_ptr(),
    num_stages=self.num_clc_stage,  # 本例中为 1
    producer_group=clc_pipeline_producer_group,
    consumer_group=clc_pipeline_consumer_group,
    tx_count=self.num_clc_response_bytes,  # 16
    cta_layout_vmnk=cluster_layout_vmnk,
    defer_sync=True,
)

关于 num_clc_consumer_threads 的计算：cluster 中所有 CTA 的 TMA、MMA 和 epilogue warp 都需要知道正确的工作 tile 坐标（以及取消是否成功），因此有 cluster_size * (1 + len(self.epilogue_warp_id) + 1)。调度 warp 本身也是消费者（它需要知道自己的取消请求是否失败作为退出信号），所以再额外加 1。注意，由于所有 CTA 启动相同数量的 warp，cluster 中非 leader CTA 也会启动一个"调度 warp"，但这些 warp 不执行任何工作，既不是 CLC pipeline 的生产者，也不是消费者。

在 CLC pipeline 创建的稍上方，可以看到为 CLC 操作和通信分配的共享内存：

class SharedStorage:
    # ... (TMA load、acc、TMEM 的 mbarrier 存储)
    clc_mbar_ptr: cute.struct.MemRange[cutlass.Int64, 2]  # 一空一满两个 mbarrier（只有一个 stage）
    clc_response: cute.struct.MemRange[cutlass.Int32, 4]  # 每个 stage 存储 try_cancel 响应的 16 字节

调度 Warp 代码块

if warp_idx == self.sched_warp_id and is_first_cta_in_cluster:

    clc_producer_state = pipeline.make_pipeline_state(
        pipeline.PipelineUserType.ProducerConsumer, self.num_clc_stage
    )

    while work_tile.is_valid_tile:
        clc_pipeline.producer_acquire(clc_producer_state)
        mbarrier_addr = clc_pipeline.producer_get_barrier(clc_producer_state)
        tile_sched.advance_to_next_work(mbarrier_addr)  # 发出 try_cancel
        clc_producer_state.advance()

        # 调度 warp 下面也充当消费者
        clc_pipeline.consumer_wait(clc_consumer_state)
        work_tile = tile_sched.get_current_work()  # 发出 query_cancel
        clc_pipeline.consumer_release(clc_consumer_state)
        clc_consumer_state.advance()
    clc_pipeline.producer_tail(clc_producer_state)
  • 第 1 行确认只有每个 cluster 的第一个 CTA 执行此代码块。
  • 第 3-5 行，pipeline state 以 PipelineUserType.ProducerConsumer 定义，起始 phase bit 被翻转，因此调度 warp 不会在 producer_acquire 处初始等待，可以立即开始获取工作 tile。

进一步查看 PipelineClcFetchAsync 的 producer_acquire 方法（位于 sm100.py）：

调度 warp 中前 cluster_size 个线程各自负责向 cluster 中的不同 CTA 发信号（线程 i 向 CTA i 发信号）。consumer_mask = 0 允许所有消费者在 release 时向 cluster 中的第一个 CTA 发信号。

调度 warp 代码块中的 tile_sched.advance_to_next_work(mbarrier_addr) 会导致 try_cancel 被发出——它调用来自单个选定线程的 issue_clc_query，最终对应 PTX 指令 clusterlaunchcontrol.try_cancel。

消费者端：get_current_work

调度 warp 的消费者部分，与所有其他消费者 warp（TMA、MMA、epilogue warp）共用相同的代码：

clc_pipeline.consumer_wait(clc_consumer_state)
work_tile = tile_sched.get_current_work()  # 发出 query_cancel
clc_pipeline.consumer_release(clc_consumer_state)
clc_consumer_state.advance()

get_current_work 本质上是 work_tile_info_from_clc_response 的封装（均在 dynamic_persistent_tile_scheduler.py 中）：

def work_tile_info_from_clc_response(
    self, result_addr: cute.Pointer, *, loc=None, ip=None
) -> WorkTileInfo:
    m_idx, n_idx, l_idx, vld = cute.arch.clc_response(result_addr, loc=loc, ip=ip)
    cute.arch.fence_proxy(
        "async.shared",
        space="cta",
    )
    cta_idx_in_cluster, cta_idy_in_cluster, _ = self.cta_id_in_cluster
    cur_tile_coord = (m_idx + cta_idx_in_cluster, n_idx + cta_idy_in_cluster, l_idx)
    return WorkTileInfo(cur_tile_coord, vld)

第 4 行的 clc_response 对应 PTX 指令 clusterlaunchcontrol.query_cancel，对响应数据进行解码。由于 query_cancel 得到的是被取消 cluster 的第一个 CTA 在网格中的坐标，需要加上当前 CTA 在其 cluster 内的坐标偏移，才能正确得到其 tile 坐标。

第 5-8 行的 shared async proxy fence 看起来不寻常。在标准 GEMM kernel 中，这类 fence 仅在 TMA store 之前出现。而这里唯一相关的 async-proxy 操作是 try_cancel 将响应数据写入 SMEM，fence 是在响应数据解码之后才调用的。因此，这个 fence 实际上是在保护：防止下一次迭代的 try_cancel 在当前迭代还未读取完 SMEM 之前就覆盖该位置的数据。另外注意 clc_response 调用之前没有 proxy fence——类比 TMA load，try_cancel 的响应数据传输完成后很可能也隐式执行了 proxy fence。

多阶段 CLC Pipeline

虽然本示例不支持，但可以通过为 CLC pipeline 设置多个 stage 来排队多个工作 tile（例如，CUTLASS C++ kernel 中深度为 3）。这在某些情况下有助于隐藏调度延迟，尤其是当某些工作 tile 完成极快时（例如，变长 attention 中某些工作 tile 可能甚至是空的）。

然而，深层 CLC pipeline 也有一个隐患——通过为不同 SM 排队不等量的工作负载，动态负载均衡效果会变差。实际上，stage 数越多，CLC 的行为就越像静态持久化调度。此外，对于 wave 数很少且负载不均衡的问题，即使只有 1 个 stage，我们可能也希望阻止调度 warp 在 MMA mainloop 结束之前就发出 try_cancel——否则被分配了大 K tile 的 cluster 可能立即获取另一个大 K tile，最终导致与静态持久化调度器相似的高度不均衡的工作分配。

CLC 与并发 Kernel 及抢占

根据编程指南，除了 kernel 中没有更多未启动 cluster 之外，try_cancel 失败的另一个原因可能是：在第一个 kernel 已开始执行之后，又启动了一个更高优先级的第二个 kernel。在观察到 try_cancel 失败后，第一个 kernel 的 CTA 将退出，为第二个 kernel 腾出 GPU 资源。待高优先级 kernel 执行完毕后，若第一个 kernel 还未执行完其完整网格，则会启动新的 cluster 继续完成剩余工作。

允许这种"抢占"是 CLC 比静态持久化调度更灵活的另一个体现——静态持久化调度在 kernel 启动后无法动态重新分配资源。

持久化调度器优于单 tile 调度器的结果并不意外，因为前者能够将 epilogue 与 MMA mainloop 重叠执行。K 较小时，epilogue 占单个工作 tile 运行时间的比例相对较大；K 较大时，epilogue 的占比较小，因此单 tile 调度因缺乏 epilogue 重叠而损失的效率相对较少。对于小问题规模，各调度器之间几乎没有差异，因为此时工作量不足一个完整的 cluster wave。

然而，CLC 与静态持久化之间观察到的性能差异显得更为难以解释，CLC 在较大工作负载时总体上表现稍差。为深入理解，我们从 Nsight Compute PM sampling 比较两者的 tensor pipe 吞吐量图（x 轴为时间，y 轴为利用率百分比）。

对于问题形状 (16384, 16384, 2048)，CLC 的 tensor pipe 吞吐量图在结尾无明显下降，而静态持久化在结尾存在逐渐下降的过程——说明部分 SM 比其他 SM 先完成并进入空闲，因此CLC 能更好地利用所有 GPU 资源。

对于 (32768, 32768, 2048)，情况恰好相反：静态调度的结尾下降反而更不明显，而 CLC 的 tensor pipe 吞吐量始终偏低。与此相关的一个指标是：NCU 报告 CLC 的 L2 命中率仅为 35%，而静态持久化为 52%。造成这一差异的原因尚不清楚。注意这两个 kernel 均未使用工作 tile swizzling；而对于 (16384, 16384, 2048)，NCU 显示两个 kernel 的 L2 命中率均约为 60%。

上述实验表明，即使对于均衡工作负载，也应同时保留静态调度和 CLC 以用于调优。另外需要注意的是，这些示例 kernel 未包含工作 tile swizzling、block scaling 或复杂 epilogue（如 bias、activation）等特性，加入这些特性后对比结论可能会发生变化。

鉴于 CLC 不存在 tensor pipe 吞吐量结尾下降，我们还统计了每个 SM 计算的 tile 数。静态持久化调度器下，各 SM 计算的 tile 数最多相差 1；而 CLC 并非如此。例如，对于问题形状 (16384, 16384, 2048)，各 SM 处理的 tile 数在 54 到 59 之间，其频率分布直方图（以 SM pair 为单位，因使用 2CTA MMA）如下：

结论（CLC 篇）

本文探讨了 CLC——Blackwell GPU 上引入的动态持久化调度的硬件实现。CLC 综合了单 tile 调度和静态持久化调度这两种传统方案各自的优势。

我们考察了 CLC 所需的低级 PTX 指令 try_cancel 和 query_cancel，并逐步讲解了 CuTeDSL 实现（示例 dense_gemm_persistent_dynamic.py）——每个 cluster 使用单个 scheduler warp 尝试 steal 工作 tile，并通过 CLC pipeline 将结果传递给其他 warp。

对于不均衡工作负载，CLC 明显优于静态持久化；但我们也发现，即使在均衡工作负载下，CLC 与静态持久化之间仍存在微小差异，两者并没有明显的绝对赢家——建议在实际调优时同时保留两种方案。

延伸：FlashAttention-4 的 CLC 实现（PR #2218）

原始来源 Dao-AILab/flash-attention PR #2218 · Add CLC work-stealing tile scheduler to FA4 状态：已合入主线 · 对应 SM100/SM110（B200/B100）

背景：为什么 Attention 更需要动态调度

GEMM 的工作 tile 计算代价大致均等，而 Attention 面临天然的负载不均衡：causal masking 让靠近序列末尾的 tile 计算量远少于前端 tile；document mask、变长序列（varlen）、GQA 的不同 head 分组更会让不同 tile 的实际工作量差异悬殊。静态 grid-stride 分配下，完成快的 SM 会空转等待慢的 SM，拖长整体延迟。CLC 的 work-stealing 机制正好解决这一问题。

Work Distribution 可视化工具

PR 附带了一个基于 printf 日志重建的 Work Distribution Viewer，用于直观展示 CLC work-stealing 的执行轨迹：每一行对应一个 SM，每个格子对应一个 tile，颜色区分"初始分配"、"被 steal"、"已完成/无效"状态。Histogram 展示各 SM 完成的 tile 数分布，可快速发现负载不均衡。

改动前：静态 Grid-Stride 调度

改动前，每个 CTA 的工作 tile 由以下公式静态决定：起始 tile 索引 = blockIdx.x，每次步进 gridDim.x（SM 数）。L2 cache swizzle 和 LPT（Longest-Processing-Time-first，逆序排布使长 tile 优先）作为启发式优化内嵌在坐标映射中，但本质仍是静态分配：

# 改动前（静态）
tile_scheduler = TileSchedulerCls()          # 每次调用都创建新实例
work_tile = tile_scheduler.initial_work_tile_info()
while work_tile.is_valid_tile:
    # prefetch_next_work() → no-op（静态模式下什么都不做）
    tile_scheduler.prefetch_next_work()
    tile_scheduler.advance_to_next_work()    # tile_idx += gridDim.x
    work_tile = tile_scheduler.get_current_work()

问题：所有 CTA 以固定步长遍历 tile，快 SM 在完成本轮 tile 后只能等待，无法去处理慢 SM 积压的工作。

改动后：CLC 动态 Work-Stealing

PR 引入了三个核心抽象：

  • SchedulingMode 枚举：NONE / STATIC / DYNAMIC / CLC，通过环境变量 FA_CLC=1 启用 CLC 路径。
  • ClcState：封装硬件调度器 ClcDynamicPersistentTileScheduler、异步 pipeline PipelineClcFetchAsync 以及 producer/consumer barrier state，与 GEMM CLC 示例结构相同。
  • TileSchedulerProtocol：统一接口，prefetch_next_work() 和 producer_tail() 在静态模式为 no-op，CLC 模式才真正工作。
# 改动后（CLC）——Kernel 初始化阶段
clc = ClcState.create(
    hw_scheduler=ClcDynamicPersistentTileScheduler.create(...),
    pipeline=PipelineClcFetchAsync.create(
        barrier_storage=clc_mbar_ptr,
        num_stages=self.sched_stages,   # 通常为 1
        producer_group=...,             # scheduler warp
        consumer_group=...,             # 所有 compute warp
    ), ...
)
tile_scheduler = SingleTileLPTScheduler.create(tile_sched_params, clc=clc)

# Scheduler warp（独立运行，持续向硬件请求新 tile）
def clc_scheduler_warp(self, tile_scheduler):
    work_tile = tile_scheduler.initial_work_tile_info()
    while work_tile.is_valid_tile:
        tile_scheduler.prefetch_next_work()   # → producer_acquire + try_cancel PTX
        work_tile = tile_scheduler.advance_to_next_work()
    tile_scheduler.producer_tail()

# Compute warp（消费者，等待 scheduler warp 填充结果）
def advance_to_next_work(self):
    if const_expr(self.params.scheduling_mode == SchedulingMode.CLC):
        self.clc.consumer_wait()      # 等待 mbarrier
        work = self.get_current_work()   # query_cancel 解码坐标
        self.clc.consumer_release()
        return work
    # 静态路径：tile_idx += gridDim.x

坐标映射：CLC 返回值 → Attention tile 坐标

CLC 硬件返回的是原始 grid 坐标 (block, head, batch_split)，FlashAttention 需要通过 clc_work_to_coords() 将其映射为语义坐标：

def clc_work_to_coords(self, work) -> WorkTileInfo:
    block_idx = work.tile_idx[0]
    if const_expr(self.params.cluster_shape_m > 1):
        block_idx = block_idx // self.params.cluster_shape_m  # 2-CTA 模式
    if const_expr(self.params.lpt):
        block_idx = self.params.num_block - 1 - block_idx    # LPT 逆序
    # split_kv 解包 ...
    return WorkTileInfo(..., work.is_valid_tile)

性能结果

PR 作者预期在负载最不均衡的 workload 上收益最大，实测结果与预期基本吻合。由于当时 LPT（Longest-Processing-Time-first）调度尚未启用，Alibi 和 Causal 的收益相近；Document mask 看到了明显提升。

  • 收益明显：Alibi masking、document mask、causal masking + 非均匀序列长度；
  • 持平或略退：稠密 MHA（无 mask/sparsity），尤其 hdim=64 非 GQA 路径，作者认为规律性太强不像随机误差，原因待查。

合入时遗留问题

  • 遗留问题 1 — 竞争条件：128×128 测试中，部分 consumer warp 在 scheduler warp 填充响应前就读取了 SMEM。作者推测：num_tiles < num_sms 时只需初始 work，query CLC 后在 warp 15 的 consumer_wait 后打印显示"no more work / all invalid"，但其他 consumer warp 没有正确 sync，导致它们读到了未初始化的随机响应数据。Racecheck 能检测到错误但无法定位根源。
  • 遗留问题 2 — 寄存器溢出：no-op 稠密 MHA 下 NCU 显示异常大量 register spill，PTXas 编译（含 13.1 版本）显示 SASS 无溢出，作者认为可能是 PTXas 的边界情况，但不完全确定；ptxas patch 也未能彻底解决。
  • 遗留问题 3 — 抽象泄漏：调度逻辑散落在多处，作者希望找到更好的封装方式后再正式 land（事实上 PR 已合入，后续会持续改进）。
  • 后续计划：统一 API + 环境变量开关（FA_CLC=1），对所有 Flex 场景默认启用；bwd 路径集成（当前仅 fwd）。

测试文件 test_clc_fuzz.py 覆盖了大量 adversarial 场景：Q/K 序列长度不匹配（如 1 vs 4096）、非对齐 tile 边界（质数、奇数尺寸）、GQA 多种 head 比、2-CTA 配置、minimal 问题（1×1×1）、causal masking + 不均衡形状、大规模（batch=32、head=8、seqlen=4096）等。

与 GEMM CLC 的关键区别：GEMM 的所有 tile 计算代价均等，CLC 的 work-stealing 仅针对 Wave Quantization；而 Attention 中即使 Wave Quantization 效应很弱，tile 间的内在计算量差异（causal mask 导致的有效序列长度不同）也足以让动态调度带来收益。这使 CLC 在 Attention kernel 中的适用价值比在 GEMM 中更广泛。
