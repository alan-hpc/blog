原文来源 DeepSeek V3 推理系统深度解析（七）：DSA 稀疏注意力，从 MLA Indexer 到 Sparse FlashMLA 接口 · Yang Min · 2026

DSA（Dynamic Sparse Attention）的难点不在于稀疏 attention 本身，而在于如何在不损失长上下文精度的前提下动态决定哪些 token 对参与全量 attention。静态稀疏（如固定 sliding window）会丢掉重要的远距离依赖；动态稀疏需要在 attention 计算之前就知道哪些 token 重要，但"重要性"本身就是 attention 的输出——这是一个鸡和蛋的问题。DSA 的解法是用低精度的 indexer 做第一遍筛选，再用全精度 attention 做精查；理解这个两段式设计，才能明白为什么 FP4 indexer 在这里不是精度妥协而是精度解耦。

DSA（Dynamic Sparse Attention）和 MTP 是两个正交问题：MTP 让 decode 少跑几轮（见上一篇），DSA 让每一轮 attention 少读 KV。本篇按 DSA C14.6 原始流程图 的三段结构重构：

DSA 优化的是 attention 的访问范围

$$ \begin{array}{lll} T_{\mathrm{decode}} &\approx& N_{\mathrm{step}}\cdot \bigl(T_{\mathrm{attn}}(N_{\mathrm{kv}})+T_{\mathrm{moe}}+T_{\mathrm{comm}}+T_{\mathrm{sample}}\bigr), \\[0.45em] \mathrm{DSA} &:& N_{\mathrm{kv}}\downarrow \quad \text{从 full history 降到 top-k=2048},\\[0.35em] && \text{多了 indexer 与 gather 的成本}. \end{array} $$

1 · 起点 · 从 MLA 到 DSA 的演化动机

1.1 MLA 已经压缩 KV，但 dense decode 仍读完整历史

MLA（Multi-Latent Attention，本系列 part 4）已经把 K/V 压缩成 latent KV cache，KV 体积 问题已经解决；FlashMLA 也已让 decode attention 直接消费 latent KV，kernel 效率 问题也解决了。

但长上下文场景 下 dense MLA decoding 仍然要扫完整历史：

  • 上下文 64K，每一步 attention 必须扫 64K 个 history token 的 latent KV
  • 上下文 128K，扫描量翻倍
  • 上下文 256K+，attention 已不再是 softmax-bound 而是 KV-read-bound

DSA 的目标：让 main attention 只看一个由训练得到的 top-k 历史 token 集合 ，而不是完整历史。

Dense MLA vs Sparse MLA 的访问范围

$$ \begin{array}{lll} \mathrm{Dense} &:& O_t=\operatorname{softmax}(Q_tK_{\le t}^{\top})V_{\le t}, \\[0.45em] \mathrm{Sparse} &:& S_t=\operatorname{TopK}(I_{t,:},2048),\quad O_t=\operatorname{softmax}(Q_tK_{S_t}^{\top})V_{S_t}. \end{array} $$

1.2 DSA 的成本不只是「多一个 indexer」

DSA 的成本由四部分组成：

成本项说明何时占比大
Indexer projectionweights_proj/wq_b/wk 三条线性token 数大时显著
Indexer score 计算每个 query 对每个 history token 打分上下文越长越贵
TopK Selector(2048)从 N 个 history 选 top-2048N » 2048 时受益最大
KV gather按 indices 抓取 KV pageindices 散乱时受益小

短上下文上 DSA 反而比 dense MLA 慢 ——history 才几千 token，indexer 的额外开销超过省下的 attention 成本。长上下文（64K+）才是 DSA 的主战场 。
