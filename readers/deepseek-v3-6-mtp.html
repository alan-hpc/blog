MTP（Multi-Token Prediction）的价值不只是"一次预测多个 token"，更在于它把 speculative decode 的 draft 生成成本几乎降到了零。传统 speculative decode 需要额外的 draft 模型，带来额外的内存占用和 KV cache 开销；MTP 把 draft head 内嵌到主模型的最后几层，复用主模型的 hidden state，代价只是额外的 linear + LM head，不额外引入 attention 层。理解 MTP 的关键在于读懂它为什么能以这么低的成本生成有效 draft。

Prologue · 本篇问题与读法

本篇从 MTP 技术设计材料重构 MTP 投机采样 这一条线（与 DSA 稀疏注意力分开成两篇）。MTP 的优化对象是decode loop 的次数 ——一次 target/main forward 验证多个 draft token，若接受前缀足够长，后面若干轮就不用跑。

MTP 减少的是 decode loop 次数

$$ \begin{array}{lll} T_{\mathrm{decode}} &\approx& N_{\mathrm{step}}\cdot \bigl(T_{\mathrm{attn}}(N_{\mathrm{kv}})+T_{\mathrm{moe}}+T_{\mathrm{comm}}+T_{\mathrm{sample}}\bigr), \\[0.45em] \mathrm{MTP} &:& N_{\mathrm{step}}\downarrow,\quad \text{每一轮成本不变，但轮数减少}. \end{array} $$

需要先划清边界：DSA 稀疏注意力（少读 KV）与 MTP（少跑 step）是两个正交问题，单独用一篇 DSA 稀疏注意力 讲。本篇只关心 MTP。

1 · 训练目标 · 从 next-token 到 multi-token prediction

1.1 DeepSeek-V3 把 MTP 放在训练目标里

DeepSeek-V3 Technical Report 把 MTP 放在训练目标里：模型在 next-token loss 之外，额外预测后续 K 个 future token 。这并不等于推理时能白拿 K 个 token；它只是让模型具备提出 draft 的能力。

从 next-token 到 multi-token prediction

$$ \begin{array}{lll} L_{\mathrm{next}} &=&-\sum_t \log p_\theta(x_{t+1}\mid x_{\le t}), \\[0.45em] L_{\mathrm{MTP}} &=&-\sum_t\sum_{i=1}^{K} \lambda_i \log p_{\theta,i}(x_{t+i}\mid x_{\le t}). \end{array} $$

这段目标对推理系统的含义是：MTP path 可以给出 draft_tokens，但每个 draft 都必须由 target/main path 验证。MTP 的收益来自"被验证后接受的连续前缀" ，不是来自 draft 的数量本身。

1.2 DeepSeek 原生 MTP 结构

DeepSeek 原生方案的关键设计：

  • 共享 embedding + output head ：每个 MTP module 不重复维护词表 lookup 和 output projection
  • 每个 MTP module 独立 KV cache ：因为 module $i$ 的 token 错位是 $+i$，与主模型的 attention pattern 不同
  • 隐状态拼接 hint ：MTP module 不是孤立预测，输入里带上前一层（main 或上一个 MTP）的 hidden state

2 · 推理流程 · vanilla vs eagle 的取舍

2.1 vanilla 模式 · 多 MTP module 并行

设计材料里给的 vanilla 例子非常具体。假设 prompt 是 ABCD，主模型在 context phase 从 ABCD 采样得到 E，MTP1/MTP2/MTP3 分别拿不同的错位输入，输出三个 draft token F/G/H：

路径输入 token使用 hidden输出 draftKV 特征
mainABCD主模型 hiddenEmain KV: ABCD
MTP1BCDEH_A,H_B,H_C,H_DFMTP1 独立 KV
MTP2CDEFH_B,H_C,H_D,h_EGMTP2 独立 KV
MTP3DEFGH_C,H_D,h_E,h_FHMTP3 独立 KV

到了 generation phase，主模型输入 E + FGH，输出一串 golden token。材料里的例子是 target 输出 F G X Y：

  • F 与 F 一致 → 接受
  • G 与 G 一致 → 接受
  • H 与 X 不一致 → 拒绝；后面的 Y 也不能跳过去接受
  • X 作为 bonus token（target 在 G 位置之后的下一个采样）也提交

最终一次 target forward 推进 EFGX = 4 个 token，其中 H 之后的 KV 必须从有效上下文里拿掉。

vanilla 模式的关键工程代价：

  • 多 future position → 多套 MTP module + 多份 KV cache
  • 接受率不高时仍要补齐每个 module 的输入 —— 即使 0 接受，下一轮 K 个 module 都要 K 个 input token，要从前序 decode 步骤里找到对应的 token id 和 hidden state
  • decode 阶段 bookkeeping 沉重 —— 引擎需要为每个 module 维护「最近 K 个 token 的 id + hidden state」的滚动窗口

设计材料原话：“如果接收的少甚至没有接收，也需要给每个 MTP 模块输入 K 个 token，这会造成复杂度，需要从之前的 decode 步骤中找到之前的 token id 和 hidden states。”

2.2 vanilla 的 KV 驱逐：每个 MTP module 单独算账

vanilla 模式下，每个 MTP module 都有自己的 KV cache 。当 target 接受 FG、拒绝 HXY... 时：

  • main KV：ABCDEFG 提交，H 之后的 KV invalid
  • MTP1 KV：原本是 BCDE，第二轮 prefix 变成 BCDE + FGX（注意是 main 的接受序列）
  • MTP2 KV：类似处理
  • MTP3 KV：类似处理

每个 MTP module 都要单独维护 cache_len，且与 main 的 accepted prefix 同步 。这是 vanilla 模式下接口最沉重的部分。

2.3 eagle 模式 · 单 MTP module 自回归

eagle 模式把同一份 MTP module 自回归复用 K 次 。设计材料的关键描述：

  • “It reuses the same MTP module and repeats multiple times to predict draft tokens”
  • “MTP module reuses the same KV cache when predicting multiple draft tokens”

推理阶段的核心差异：

  • Prefill ：第一次 MTP forward 输入 prompt，得到第 1 个 draft token；之后每次只输入刚生成的 1 个 draft token
  • Decode ：第一次 MTP 调用输入「prefill 阶段生成的 1 个 token + 接受后的 N 个 token」；后续迭代只输入 1 个 token
  • 状态简化 ：只维护一份 MTP weight、一份 MTP KV cache、一条 draft stream

2.4 vanilla vs eagle 的工程取舍对比

维度vanillaeagle
MTP module 个数K 个1 个（参数复用）
MTP KV cache 份数K 份独立1 份
推理阶段 bookkeeping每 module 维护「最近 K token + hidden」滚动窗口只需维护接受序列
拒绝时的输入补齐必须给每 module 补 K 个输入第一次输入 N 接受 token，后续只 1 个
实现复杂度高低
多 module 并行理论上可，工程上贵串行 K 次（自回归）
多模型权重情况需所有 K 套权重只用第 1 套权重即可

MTP 技术设计材料明确推荐先实现 eagle 模式 ：复杂度可控，KV bookkeeping 简单，对单 GPU/小集群友好。多 MTP 模块的 vanilla 模式留作后续扩展。

2.5 接受率决定上限，不是 draft length 决定上限

令 $A$ 表示一轮验证后被接受的连续 draft prefix 长度，则一次 target forward 实际推进 $1+A$ 个 token，其中额外的 1 是 target 输出的 bonus token。MTP 的理想收益和 $\mathbb{E}[A]$ 相关，而不是和最大 draft 长度 $K$ 直接相等。

MTP 的一阶收益模型

$$ \begin{array}{lll} \mathrm{advance} &=&1+A, \\[0.35em] \mathrm{speedup} &\lesssim& \dfrac{1+\mathbb{E}[A]} {1+\rho_{\mathrm{mtp}}+\rho_{\mathrm{verify}}+\rho_{\mathrm{rollback}}}. \end{array} $$

这里 $\rho_{\mathrm{mtp}}$ 是 draft path 开销，$\rho_{\mathrm{verify}}$ 是 target 验证与采样开销，$\rho_{\mathrm{rollback}}$ 是 KV 回滚与状态更新开销。接受率低时 draft 做了很多但推进不多；接受率高时才真正减少 decode loop 。

3 · 验证策略 · 三种 acceptance + relaxed

3.1 Acceptance-Rejection Sampling

设计材料给出的三种接受策略（按由严到松）：

策略形式优点风险
argmaxtarget top-1 token == draft token严格、不引入分布偏移接受率最低
thresholdtarget 在 draft token 上的概率 ≥ τ接受率可调（默认 0.6）选错 τ 会引入偏移
topkdraft 落在 target top-k 集合里接受率较高top-k 越大风险越大

连续前缀接受规则

$$ \begin{array}{lll} a_i &=&\mathbf{1}\{\mathrm{Accept}(d_i,\ p_\theta(\cdot\mid x_{\le t+i-1}))\}, \\[0.35em] A &=&\max\{m:\ a_1=a_2=\cdots=a_m=1\}. \end{array} $$

必须是连续前缀 。第一个 draft 被拒绝后，后面的 draft 即使概率也高，也不能跳过拒绝点继续提交，因为它们依赖的条件上下文已经变了。

3.2 Relaxed Acceptance · 提高接受率但有质量风险

设计材料中的实战边界："目前只用在 reasoning 模型前边的 thinking 阶段"——thinking 阶段的精度对最终回答影响小，可以激进一点；但回答阶段必须用严格的 argmax 或 threshold。

3.3 验证代码骨架

int LlamaBatch::EagleVerification( const std::vector<int>& draft_tokens, const Tensor& verification_logits, MTPAcceptPolicy policy, float threshold) { int accepted_count = 0; for (int i = 0; i < (int)draft_tokens.size(); ++i) { auto position_logits = verification_logits.slice(i, 1); auto position_probs = Softmax(position_logits); bool accept = false; switch (policy) { case MTPAcceptPolicy::ARGMAX: accept = (ArgMax(position_probs) == draft_tokens[i]); break; case MTPAcceptPolicy::THRESHOLD: accept = (position_probs[draft_tokens[i]] >= threshold); break; case MTPAcceptPolicy::TOPK: { auto topk_tokens = TopK(position_probs, 5); accept = std::find(topk_tokens.begin(), topk_tokens.end(), draft_tokens[i]) != topk_tokens.end(); break; } } if (accept) accepted_count++; else break; // 第一个拒绝即停 } return accepted_count; }

注意 else break ——一旦拒绝立即停止，体现"连续前缀"语义。

4 · Serving 实现 · 状态机、接口、KV 驱逐

4.1 引擎状态机：PREFILL → SCORE → VERIFICATION

MTP 技术设计材料里把 engine 状态拆成 4 个 phase：

PREFILL → 常规 prefill 阶段（与无 MTP 相同） PREFILL_COMPLETED → prefill 完成、等待开始 MTP forward SCORE → main model + MTP forward, 得到 1 + K draft token VERIFICATION → main model 验证, 接受/拒绝 + KV 驱逐

执行流程：

  1. 新进来的序列都是 PREFILL
  2. 走过 prefill 后变成 PREFILL_COMPLETED，此时执行第一次 MTP forward 得到 1 + K 个 token，进入 SCORE
  3. SCORE 阶段输入 1 + K 个 draft token 给 main model，得到 1 + K 个 golden token，进入 VERIFICATION
  4. VERIFICATION 三步：
    • token 比对 或其他采样方式得到 accepted prefix
    • 驱逐未接受 token 的 KV
    • 用 accepted tokens 作为输入，再做一次 MTP forward 得到下一轮 1 + K draft token，回到 SCORE

4.2 Sequence 字段扩展

设计材料给出 Sequence 结构需要增加的 MTP 字段：

enum MTPEaglePhase { PREFILL, // 常规模式 PREFILL_COMPLETED, // prefill 完成，等待开始 MTP SCORE, // main model 推理，draft model 得到 golden token VERIFICATION // 验证阶段 }; struct Sequence { int input_length; // 原有：本轮输入长度 mutable int cache_len; // 原有：已缓存长度 mutable std::vector<int> tokens; // 原有：已计算 tokens // MTP Eagle 扩展字段 bool mtp_eagle_enabled = false; // 是否启用 MTP Eagle int mtp_draft_count = 3; // 每轮草稿数 K MTPEaglePhase mtp_phase = PREFILL; std::vector<int> mtp_draft_tokens; // 草稿 token 序列 std::vector<float*> mtp_draft_hiddens; // 对应的 hidden states int mtp_accepted_count = 0; // 上轮接受的数量 };

调度策略上的关键决定：刚跑完 Prefill、还没有进行 MTP forward 的序列，与正常 decode 序列一样插入最前的 decode 序列里 ——不为 MTP 单独拉一条调度路径，最大限度复用现有 batch。

4.3 控制参数（引擎级别）

引擎级别（kernel launch 时确定）的 MTP 控制参数：

mtp_eagle_enable: true # 是否启用 eagle 模式 mtp_draft_length: 3 # K，每轮 draft token 数 mtp_accept_policy: argmax # also: threshold / topk mtp_accept_threshold: 0.6 # threshold 策略的概率阈值

为什么是引擎级别而不是请求级别：MTP 涉及 weights load、状态机、KV 池子分配，这些资源都需要在 kernel 启动时确定 。请求级别的开关会让 GPU 一边跑 MTP 一边跑非 MTP 序列，状态机和 batch 编排都很复杂。

4.4 KV Cache 管理 · 按 token 驱逐 + prefix cache 不能进入

4.4.1 KV 驱逐目标

验证阶段主模型已经为 1 + K 个 token 写了 KV。若只接受前缀，拒绝后缀的 KV 不能被之后 attention 读取 。最直接的做法是物理清零，但 hot path 更合理的做法是逻辑屏蔽 ：更新 cache_len、h_context_length 或 attention kernel 的 max_k_len。

设计材料原话："通过 h_context_length 控制访问范围即可，attention kernel 通过 max_k_len 控制访问范围 - 不会读取被驱逐的 token 数据"。物理 buffer 残留无效 KV 不影响正确性，只要 attention 的有效长度不读它。

max_k_len ─────►├──── valid history (committed) ────┤├── invalid (dropped) ──┤ ↑ attention 不会越过这条线

4.4.2 prefix cache 边界 · BlockTrie 不能看见 draft

设计材料里特别强调三条规则：

  1. prefix-caching 只处理 seq.prompt ：

    • 匹配：block_trie_->match(seq) 仅在 !seq.prompt.empty() && seq.blocks.empty() 时执行
    • 缓存：block_trie_->cache(seq) 只处理 prompt 范围
    • 清空：prefill 后立即 seq.prompt.clear()

  2. Decode 阶段完全不涉及 BlockTrie ：

    • Decode tokens 存在 seq.tokens 里
    • MTP draft tokens 存在 mtp_draft_tokens 里
    • 两者都不进入 BlockTrie

  3. 无需处理块中部分 token 的清理 ：通过 h_context_length 控制访问范围即可

这不是实现细节，而是正确性边界 ：被拒绝的 draft token 如果进入 prefix cache，将来可能被别的请求复用，造成非常隐蔽的错误（用户问相同问题但答案不同，且无法复现）。物理 buffer 残留无效 KV 是无害的，但语义层泄露被拒绝的 draft 是灾难性的。

4.5 MTPDecoder 是 decoder 旁边的第二条路径

设计材料里把 MTP weights 放到独立的 MTPDecoderLayerWeight，并在 unified_decoder_、dynamic_decode_ 旁边新增 MTPDecoder。这样主模型 decoder 不需要被改成一堆 MTP-specific 分支；MTP path 只在需要 draft 时被 engine 调用。

LlamaV2 ├── unified_decoder_ ← main model ├── dynamic_decode_ ← sampler └── MTPDecoder ← NEW: MTP path └── MTPDecoderLayerWeight

边界很重要：主模型负责验证、MTPDecoder 负责提案、dynamic_decode 负责采样策略 。三者混在一个 kernel 或一个类里，后面很难做 fallback、灰度、回滚和性能定位。

4.6 与现有 LlamaBatch::Forward 的兼容

设计材料强调 LlamaBatch::Forward 流程要适配但兼容老逻辑。三处改动：

改动内容兼容老逻辑的方法
序列分类 + 准备阶段h_input_length_buf_, input_d_ptrs按 mtp_phase 分组，原 decode 序列走原路径
Mini-batch 处理把 SCORE / VERIFICATION 拆成不同 mini-batch不开 MTP 时这些 mini-batch 为空
MTP Eagle 后处理函数验证 + 驱逐 + 下一轮 MTP forward通过 mtp_eagle_enabled 短路

5 · 性能基线 · TRT-LLM 上的 DeepSeek-R1-FP4 nextn 收益曲线

参考曲线给出的几条信号：

  1. nextn=0（无 MTP baseline） TPS 约 155，nextn=2 时 TPS 翻到 ~265，1.7× 速比
  2. nextn 继续加大到 7 收益变缓，TPS 仅再涨到 ~309（约 2× baseline）
  3. Overlap scheduler 和 Relaxed Acceptance 都能进一步提高吞吐 ——前者把 MTP 的 idle 时间藏起来，后者直接提高接受率（牺牲一点质量）
  4. Acceptance Rate 从 ~17% 涨到 ~46% ——但仍远低于 100%，说明大部分 draft 仍被拒绝

这条曲线的实战含义 ：

  • nextn 不是越大越好，K=2 或 3 已经能拿到大部分收益
  • K 越大 draft 计算成本越高，收益边际递减
  • Relaxed Acceptance 在 thinking 阶段值得开，回答阶段慎用

6 · 上线边界 · 系统视角看 MTP

6.1 MTP 让 scheduler 看到 variable-rate streams

普通 decode 默认每轮每个请求推进一个 token。MTP 之后，每个请求推进的 token 数 = 1 + accepted_count ，而 accepted count 动态变化。连续 batching 看到的是 variable-rate streams：

  • 有的请求本轮推进 4 个 token
  • 有的只推进 1 个
  • 有的 draft 全部拒绝

最稳妥的做法是在请求层维护 accepted count，把下一轮输入重新归入稳定的 graph bucket，而不是让每个 kernel 都面对动态分支 。MTP 的收益来自少跑主循环，不能让调度碎片把收益吃掉。

6.2 5 项常见误读

误读更准确的说法
MTP 速度提升取决于 draft 长度 K取决于平均接受长度 $\mathbb{E}[A]$；K=3 已能拿大部分收益
eagle 模式接受率低于 vanilla实测接近，eagle 工程复杂度低很多更值
Relaxed acceptance 总是开就行仅 thinking 阶段开；回答阶段必须严格
拒绝的 draft 不影响主模型KV 必须按 token 驱逐，否则正确性出错
MTP 是采样器的小改动是整个引擎状态机 的扩展（PREFILL→SCORE→VERIFICATION）

6.3 上线前 6 项 sanity check

实战里 MTP 上线前必查：

  1. prefix cache 不会被 draft 污染（4.4.2 边界）—— 用相同 prompt 跑 100 次看输出是否一致
  2. 拒绝后 KV 长度正确收回 —— attention kernel 的 max_k_len 必须每轮更新
  3. 每个 module 的 KV pool size 足够（vanilla）/ 单 MTP module KV 足够（eagle）
  4. draft 失败率监控 —— accept rate < 30% 时考虑关 MTP 或调 nextn
  5. CUDA Graph bucket 包含变长 input —— input_length 从 1 跳到 1+K 的两种 bucket 都要 capture
  6. 采样器 logits 同步 verification 输入 —— verification_logits 形状必须与 draft_tokens 对齐
