① Tensor Core 四代演进总览

过去十年 GPU 架构的所有大动作几乎都围绕一件事：把 Tensor Core 喂饱 。每一代的设计都是上一代的瓶颈所驱动的——这条线索贯穿全文。原 NVIDIA 官方一图说明这条主线：

发射粒度影响什么？ 它不是算力大小，而是“哪一级软件实体负责提交这次 MMA”。粒度变小会改变三件事：math consumer 要绑住多少 warp、同步边界停在 warpgroup 还是 mbarrier、累加器继续占 RF 还是搬到 TMEM。NVIDIA PTX ISA 明确列出 tcgen05 各指令的 issue granularity：mma/cp/shift/commit 在 cta_group::1 下可由单线程发起，ld/st 则仍按 warp 访问 TMEM 的 1/4。

② Blackwell 执行层级

回顾：Hopper 之前如何把 Tensor Core 喂饱

从 Volta mma 到 Ampere multi-stage 再到 Hopper Persistent Kernel + Warp Specialization

要理解 Blackwell 为什么要“颠覆性”地重设编程范式，必须先看清 Hopper 这一代是怎么把流水做到极致的——以及在那之上还有哪些软件代价无法消除 。

1.1 Volta / Ampere 时代：Tensor Core 的“被动喂饭”

Before · Volta 之前
FP32 SIMT 算 GEMM。每个 thread 各算一格，对 RF 容量、L1 hit rate 极敏感，大矩阵 30% 已经是天花板。
Problem
纯 SIMT 单元无法摊平稠密 GEMM 中海量的 MAC，每个 cycle 只发射一条 fma 。
How · Volta
引入 Tensor Core，每条 mma.sync.m8n8k4 指令完成 16×16 块 fma，密度提高 8× 。但仍是 thread-level 同步指令。
After
FP16 GEMM 能跑到 60–70% 峰值，但访存仍是同步 ：load → sync → mma → load → sync → mma。

Ampere 加了一件关键事：异步加载 。cp.async 让 GMEM→SMEM 不再阻塞计算线程，配合 multi-stage 软件流水，访存与 Tensor Core 计算可以重叠。NVIDIA 在 AI Open Day 上的总览图直接展示了这一切：

1.2 Hopper 的两个改进：Persistent Kernel + Warp Specialization

Ampere multi-stage 解决了"访存挡住计算"的问题，但还没解决tile 边界的 prolog / epilog 。在传统 Block-per-tile 模型下，每个 block 都要走一次 "prolog→compute→epilog"，prolog/epilog 时段 Tensor Core 完全闲置。当 Tensor Core 越来越快，这部分占比越来越大。

❌ Block-per-tile 的三大问题
  1. 每个 block 都有 prolog/epilog ：分配 SMEM、生成 swizzle 模式、初始化累加器、最后写回 GMEM——这些都不用 Tensor Core，但占 cycle。
  2. SM 反复“上下班” ：block 启动/退出本身有开销，几百到上千 cycle。
  3. 负载不均 ：当某个 SM 跑得快时，它得空等其它 SM。

Hopper 的解法是 Persistent Kernel ：把 grid size 钉死等于 SM 数，每个 CTA 长期占据一个 SM ，依次处理多个 output tile。prolog 在 kernel 启动时只做一次 ，epilog 也只做一次。tile 之间的边界变成"数据切换"而不是"block 切换"。

但仅 Persistent 还不够——单个 CTA 内部还要把 TMA producer 和 wgmma consumer 错开。Hopper 由此推出 Warp Specialization ：把一个 warpgroup 专门做 TMA 加载，另两个 warpgroup 交替做 wgmma 计算（"ping-pong"），中间用 mbarrier 协调。NVIDIA 官方时间线最直观：

DEEP DIVE 为什么 mbarrier 不只是个普通 semaphore

普通信号量需要 producer 主动 arrive()。但 TMA 的“传完了”事件是由硬件 DMA 引擎 发出的，不是软件能 hook 的。tx-count 字段让硬件可以在 cp.async.bulk 完成时自动累加字节数 ，软件只需声明"我期望 16 KB"，剩下完全靠 DMA fabric 自己跑完。同理 Blackwell tcgen05.mma 完成时 umma_arrive 也是硬件原子 地把 mma 完成 + SMEM 释放 + TMEM 写入这三件事串成一条 arrive。

1.6 但 Hopper 仍然把哪些事钉死了？

Hopper 已经把 GPU 流水推到非常极致。但有三个根本限制 它没动——这三件事正是 Blackwell 要拆掉的：

How · NV-HBI

DEEP DIVE 为什么不学 AMD MI300 暴露 chiplet

AMD MI300 的 chiplet 拓扑对软件可见——不同 die 上的 SM 通信延迟不同，需要程序显式调度。NVIDIA 的判断是：编程模型的稳定性比“暴露硬件”换来的微小性能提升更重要 。所以 NV-HBI 的设计目标是“对软件零感知”——跨 die 的 cluster mbarrier 与同 die 看起来一样，仅在 L2 partition 编排上做了内部优化。这是 Blackwell 整套软件栈（CUTLASS / CuTe / TMA）几乎不变就能用的根本原因。

2.1.1 NV-HBI 跨 die 一致性 · 硬件如何把双 die 拼成 monolith

上面 deep-dive 框说 NV-HBI “对软件零感知”，但硬件层面是怎么做到“零感知”的 没展开。本节按硬件 → 一致性协议 → 性能特征三层拆开。

第二层 · 跨 die 一致性协议（推断版本，NVIDIA 未完全公开）

类比 CPU 多 socket 服务器的 NUMA + cache coherence。NV-HBI 大致走类似 MOESI 风格协议，但针对 GPU 的 throughput 工作负载做了简化：

  1. L1 / SMEM 不跨 die ：每 SM 的 L1 cache 与 SMEM 只服务本 SM，没有跨 die 一致性问题；
  2. L2 partition 自治 + 跨 die snoop ：每个 die 有 ~32 MB L2（B200 总 65 MB）；当某 SM 访问的 cache line owner 在另一个 die 的 L2 时，本 die L2 通过 NV-HBI snoop 拿数据；
  3. HBM 物理位置 vs 逻辑寻址解耦 ：4 个 HBM3e stack 物理上在 2 个 die 上分布，但软件看到统一 192 GB 地址空间；HBM controller 跨 die 做 routing；
  4. Atomic 跨 die ：通过 NV-HBI 的 atomic message 实现；延迟比同 die atomic 高 ~30 cyc 但保证跨 die 一致；
  5. Cluster mbarrier 跨 die ：cluster network 通过 NV-HBI 路由跨 die 的 mbarrier arrive 信号——这就是为什么 Layer 4 中的 2-CTA UMMA 即使 leader 与 peer CTA 在不同 die 也能正常工作。

2.3.1 它解决了什么问题？

Before · Hopper RF 累加器
wgmma 把累加器写在每 thread 的 RF 中。一次 64×256 累加占 64 reg/thread，warpgroup 4×128 thread 共占 32 KB RF。
Problem
(1) RF 总共 256 KB，被累加器吃掉一大半；(2) 累加器钉在同 warpgroup ，无法让别的 warp 接管 epilogue；(3) RF 寻址按 thread-private，复杂的 thread-value layout 难以高效处理。
How · TMEM
把累加器从 RF 搬到 TMEM。专属总线、专属寻址、与 SMEM 带宽叠加 而非竞争。
After
RF 释放给真正的 SIMT 计算（地址、scale、softmax）；累加器 ownership 与 warp 解绑——任意 warp 可读 TMEM；warpgroup 可以切角色。
✅ TMEM 三大设计取舍
  1. 容量恰好 = RF (256 KB) ：让 Hopper 上原本“够装一个 warpgroup 累加器”的 workload 直接迁移过来不变。
  2. 形状 128 lanes × 512 cols ：128 = warpgroup 的 thread 数；512 col × 32-bit 在 FP16 下正好够 128×256 累加块。这是为 wgmma 类 tile 量身订做的尺寸。
  3. 仅 Tensor Core 可访问 (SIMT 不可触) ：避免给 TMEM 加 cache line / coalesce / bank 仲裁等通用机制——只对 5th-gen TC 暴露，硬件可以走专门的、规则的、宽得多的总线。

2.3.2 sub-partition 限制是怎么回事？

TMEM 物理上分成 4 个 sub-partition，每个 32 lanes × 512 cols。一个 warp 只能访问属于自己 sub-partition 的 32 lanes 。这意味着：

  • 读取整个 128×N 累加块 需要 4 个 warp（即一个 warpgroup）协作，每 warp 拿 32 行；
  • scale factor / 元数据 如果要让所有 lane 都看到，必须多播复制 4 份（见 Layer 11）；
  • TMEM 不是“shared memory plus” ——它的访问模式是高度规则化的，不是任意指针。
⚠️ 这是 Blackwell 编程上最容易踩的坑：写 epilogue 时如果忘了让 4 个 warp 都参与 tcgen05.ld，就只能拿到 1/4 的累加器。

2.3.3 带宽：与 SMEM "相加"而非"竞争"

论文实测 TMEM 提供 16 TB/s 读 + 8 TB/s 写。关键是这不和 SMEM 带宽竞争 ——TMEM 走独立的总线连到 Tensor Core，与 SMEM→Tensor Core 通路并行。所以一个典型 mma 可以同时：

  • 从 SMEM 读 B (3.8 TB/s)
  • 从 TMEM 读 A (16 TB/s 上限)
  • 把累加器写回 TMEM (8 TB/s)

三条总线同时跑，等效带宽是 SMEM-only 路径的 ~2.1×。这就是论文报道"FP8 实测 3851 TFLOPS @ 96.3% 峰值"的物理基础——没有 TMEM 这块带宽，5th-gen Tensor Core 算力提了也只能闲着 。

3.2 单线程异步 MMA：颠覆性的“小事”

表中最不起眼但最关键的是 tcgen05.mma 的"单线程发射"。听起来像微优化，实际上是整个 Blackwell 编程模型重塑的支点 。

Before · Hopper wgmma
必须 4 个 warp（128 thread）同时活着才能发射；累加器在每 thread 的 RF 中；要写 wgmma.fence + commit_group + wait_group 三段同步。
Problem
(1) 4 warp 任一没准备好就卡住整个发射；(2) RF 累加器导致 warpgroup 必须存活到 epilogue；(3) 三段同步软件易写错、栈追踪难。
How · tcgen05.mma
1 个 thread 发射一条指令；累加器写到 TMEM；硬件自动把完成事件 arrive 到 mbarrier。其它 31 个 thread 完全自由。
After
1 个 dispatcher warp 持续喂 Tensor Core；剩下 warp 可以做 softmax、做 epilogue、做下一轮 TMA wait。论文实测延迟恒为 11 cyc ，与 N 几乎无关。
DEEP DIVE “延迟恒为 11 cyc”是什么意思

Hopper wgmma m64nNk16 的实测延迟与 N 强相关：N=64/128/256 分别是 32/64/128 cyc——典型时序流水（temporal pipelining） 特征：tile 越宽、串行 cycle 越多。Blackwell tcgen05.mma m{64,128,256}n{64,128,256}k16 测得延迟全在 11.0–11.4 cyc 之间，与 tile 宽度几乎解耦 。论文据此推断 Blackwell 5th-gen Tensor Core 是spatial array 设计 而非 temporal pipelining——硬件把 tile 在空间上完全展开，所以更宽的 tile 不付出时间代价。这意味着 Blackwell 上选择更大的 MMA tile 是“白吃午餐”。

  3. 看 PTX ：grep tcgen05，没有就是 kernel 没真正升级到 Blackwell。

关键新增：TC completion tracker + TC retire → mbar wired 路径 。这两件事让 Tensor Core 的完成事件不再需要软件 commit/wait，而是由硬件 atomically 通知 mbar。

第二层 · 单条 tcgen05.mma 的硬件路径（10 步）

  1. Issue ：1 个线程发射 tcgen05.mma，PTX 携带 instruction descriptor + A/B descriptor + D 的 TMEM 地址 + idesc + (optional) mbar 地址；
  2. Decode ：warp scheduler 把指令分发到 Tensor Core 子单元；descriptor 在硬件中解码出精度、shape、accumulate flag；
  3. Operand fetch ：根据 A descriptor 从 SMEM（或 TMEM）通过专用宽路径加载 A operand；从 SMEM 加载 B；不占用 LSU；
  4. Spatial array compute ：5th-gen Tensor Core spatial array 一次展开整个 tile（m{64,128} × n{8..256} × k16）做 MAC，单条 ~11 cyc；
  5. Accumulate（如果 idesc.enable_input_d=true） ：spatial array 内部 += 当前 TMEM D 的值；
  6. TMEM write ：算完的累加器通过 TMEM write port 原子地 写回 D 的 TMEM 地址；写入完成才算 mma 真正 retire；
  7. Retire signal ：TC completion tracker 标记此 mma 完成；如果 mma 关联了 mbar，硬件自动触发下一步；
  8. mbar atomic update ：TC retire 信号通过专用 wired 路径写入 SMEM 中 mbar 的 arrive_cnt 字段（atomic +1）；
  9. Phase 翻转 ：mbar engine 检测到 arrive_cnt 达到 expected_cnt，原子翻转 phase bit；
  10. Consumer 唤醒 ：所有 wait_barrier 的 warp 看到 phase 翻转，解除阻塞继续执行。

第三层 · 两种 retire 模式

tcgen05.mma 支持两种 retire 通知方式，对应不同 mbar 用法：

模式PTX 语法语义典型用法
单 mma 直接 arrive mbartcgen05.mma ... [mbar];
(指令直接带 mbar)		这条 mma 完成后单独 +1 mbar不常用，开销过大
commit-group 模式多条 tcgen05.mma ... ;
+ tcgen05.commit [mbar];		一次 commit 把截至此刻发射的所有 mma 打包成一个 retire 事件 → +1 mbar常用 ，DeepGEMM/CUTLASS 默认

commit-group 模式的优势：mainloop 里发射 N 条 mma 算一个 K-tile 的所有 sub-mma，最后一条 commit 通知 mbar；mbar 的 expected_cnt = 1（不是 N）。消费者只需 wait 1 次 ，无关 mma 个数。这就是 §3.3 mainloop 伪码里 umma_arrive(mma_mbar) 在循环外的原因。

第四层 · 与 Hopper wgmma 的硬件路径对比

Hopper wgmma 路径
(1) wgmma.fence 显式 fence SMEM；(2) wgmma 异步发射；(3) wgmma.commit_group 软件标记一组；(4) wgmma.wait_group 0 软件 poll 完成位；(5) 累加器在 RF 里 ，由该 warpgroup 自己 spin-wait 。
Hopper 路径的硬件代价
(1) commit_group / wait_group 是软件指令 ，每条占 1 cycle warp scheduler；(2) RF accumulator 写入由 wgmma 内部完成，但warpgroup 必须保持活跃 等 wait_group 解除；(3) 跨 warpgroup 跟踪需要 SMEM 软件 fence。
Blackwell tcgen05.mma 路径
(1) 无 fence ：硬件保证 SMEM read 在 mma 内完成；(2) 无 commit_group/wait_group ：tcgen05.commit + mbar 替代；(3) accumulator 在 TMEM，发射 warp 写完即解放 ；(4) 完成事件由 TC tracker 直接 wired 到 mbar，0 软件 polling 。
定量收益
每个 K-tile mainloop 同步代码：Hopper ~6 条软件指令（fence + commit + wait + reset_phase）；Blackwell 0 条（commit 算半条，因为它本身就是 mma 路径的一部分）。一个 deep mainloop（10 stage）累计省 ~60 cycle / iteration。

第五层 · 跨 CTA 的 retire 多播（2-CTA UMMA 时）

当用 tcgen05.mma.cta_group::2 时，retire 事件需要通知两个 CTA ——leader 和 peer 都要知道 mma 完成。硬件路径：

// Leader CTA 发射
tcgen05.mma.cta_group::2 [d_tmem], a_desc, b_desc, idesc;
// 由 leader 单线程发射，peer CTA 不参与发射但其 TMEM 也被写

// Leader CTA 的 commit (multicast 到两个 mbar)
umma_arrive_multicast_2x1SM(mma_mbar, mcast_mask);
// 展开 PTX:
//   tcgen05.commit.cta_group::2.mbarrier::arrive
//                  .multicast::cluster::all.b64 [mbar];
// 行为: TC tracker 检测到 cta_group::2 标志,
//       atomic 触发两个 CTA SMEM 中 mbar 的 phase 翻转

关键 trick：peer CTA 的 mbar 地址 = leader CTA mbar 地址清掉 bit 24（因为 cluster 内 CTA pair 的 SMEM 地址只差 bit 24）。硬件 multicast unit 利用这点把一个 atomic update 拆成两个 atomic update 同时发到两个 SMEM bank。

✅ “TC retire → mbar atomic” 是 Blackwell 异步范式的关键 wired 路径

把它和 §6.1.1 的 CLC 对比：CLC 让 work distribution 异步（CTA 不等 atomic counter），TC retire 让 mma 完成异步（warp 不等 wgmma.wait_group） 。两者本质都是“把曾经吃 SM cycle 的事 wired 到专用硬件 + mbar 通知”。这就是“全异步”在硬件层面的具体实现：每种异步引擎都有专用 retire 路径直连 mbar engine，软件只需要 wait_barrier 一种同步原语。

💡 实战调试技巧： 如果发现 tcgen05.mma 完成事件没触发 mbar 翻转——99% 是因为忘了写 tcgen05.commit，TC tracker 不知道何时算“一组结束”。Hopper wgmma 的 commit_group 是“开发者主动声明组的边界”；Blackwell tcgen05.commit 是“主动通知 TC tracker 立即 retire 截至此刻的所有 mma”——语义近似但缺一不可。

3.7 TMEM / tcgen05 编程的 6 大新手陷阱（社区高频踩坑）

本节汇总 NVIDIA Developer Forums、CUTLASS GitHub issues、Modular blog、gau-nernst 个人博客等社区资料中出现频率最高的 6 个陷阱——每个都编译过、不报错、运行结果错 ，是真正难调的 bug。写 Blackwell kernel 前过一遍这份清单能省好几天。

陷阱 ① · alloc / dealloc 当成单线程操作

新手会这么写
if (threadIdx.x == 0) {
    tcgen05::alloc(num_cols, &tmem_base);
}
__syncthreads();
为什么错
tcgen05.alloc 是 warp-collective 操作——整 warp 必须共同执行才能完成 atomic 申请并把 base ptr 写入 SMEM。单 thread 调用 = 其它 31 thread 没参与 = 申请挂起或 PTX 报错。
正确写法
// 整 warp 都进入这个分支
if (warp_id == 0) {
    tcgen05::alloc(num_cols, &tmem_base);
}
__syncthreads();
// dealloc 也必须由同一个 warp 完成
if (warp_id == 0) tcgen05::dealloc(tmem_base, num_cols);
硬件原因
TMEM 是有限资源，alloc 内部走 cluster-scope atomic counter；让 warp 而非 thread 触发，硬件可批量做仲裁。同 warp dealloc 是因为 alloc handle 存在 warp-private 状态机里。

陷阱 ② · 单 warp 取 epilogue 累加器，只拿 1/4

新手会这么写
// 想偷懒只用 1 个 warp 做 epilogue
if (warp_id == 0) {
    tcgen05::ld<...>(rC, tmem_base);
    // 后处理 + store
}
为什么错
每个 warp 仅能访问 TMEM 同 sub-partition 的 32 lanes 。warp 0 拿 lane 0–31，warp 1 拿 lane 32–63...单 warp 调 tcgen05.ld 只能拿到累加器前 1/4 行（编译过、不报错、剩 3/4 是垃圾值）。
正确写法
// 必须 4-warp warpgroup 协作
auto tiled_t2r = make_tmem_copy(SM100_TMEM_LOAD_32dp32b1x, tmem_acc);
// 4 warp (128 thread) 同时进入
copy(tiled_t2r, tmem_src, rmem_dst);
// CuTe 内部按 warp_id 分配 lane 范围, 一次拿完 128 lane
调试线索
epilogue 输出只有前 1/4 行正确，剩下都是 0 / NaN / 旧值——99% 是这个陷阱。

陷阱 ③ · 把 cute::gemm 当作单线程函数调用

新手会这么写
看到“tcgen05.mma 单线程发射”就以为：
// 想让一个 thread 单独调 cute::gemm
if (threadIdx.x == 0) {
    cute::gemm(tiled_mma, tCrA, tCrB, tCtAcc);
}
为什么错
cute::gemm 内部展开为多个 sub-mma 发射 + 同步 。它假设整 warp 进入这个函数（用 elect_one_sync 内部选 leader）。单 thread 进入 → elect_one_sync 找不到其它 active thread → 整 warp 死锁 。
正确写法
// 三层 elect 嵌套, leader CTA + leader warp + leader thread
if (cta_in_cluster_v == 0
        && cute::elect_one_warp()
        && cute::elect_one_thread()) {
    cute::gemm(tiled_mma, tCrA, tCrB, tCtAcc);
}
调试线索
CUDA-GDB 看 SASS 卡在 BAR.SYNC 或 NANOSLEEP——典型的 elect_one_sync 找不到 leader。

陷阱 ④ · 忘了 tcgen05.commit, mbar 永远不翻转

新手会这么写
看到 wgmma 用 commit_group / wait_group 三段式，以为 Blackwell 也是；或反过来，看到 Blackwell “自动 arrive mbar” 描述以为 mma 后什么都不用写：
tcgen05.mma [acc], a, b, idesc;
tcgen05.mma [acc], a, b, idesc;
// 期待 mbar 自动 arrive — 错!
wait_barrier(mbar, phase);  // 永远死锁
为什么错
tcgen05.mma 完成事件由 TC tracker 累积 但不主动 arrive mbar ——必须发 tcgen05.commit [mbar] 才会把“截至此刻所有 mma”打包成一次 mbar arrive 事件。漏写 commit = mbar 永远等不到 = consumer warpgroup 死锁。
正确写法
for (int kb = 0; kb < nK; ++kb) {
    tcgen05.mma [acc], a(kb), b(kb), idesc;
}
tcgen05.commit [mbar];   // ← 关键! 把 N 条 mma 打包通知 mbar
// consumer:
wait_barrier(mbar, phase);  // 此时正常解锁
调试线索
nsight-compute 看 wait_barrier 长期 stalled、mma TC active% 高但 epilogue 不开始——99% 漏 commit。

陷阱 ⑤ · 复用 Hopper wgmma 路径，自以为升级了 Blackwell

常见情况
把 Hopper 的 CUTLASS kernel 在 sm_100a 编译过了，看 nvcc 没报错就以为“升级完成”。但 wgmma 在 Blackwell 已被官方 deprecated 。
为什么没拿到收益
wgmma 在 sm_100a 上仍能编译，但走的是Ampere/Hopper 兼容路径 ，不调 5th-gen Tensor Core spatial array。结果：(1) 实测 throughput 与 H100 几乎一致（< 5% 提升）；(2) 不能用 FP4/FP6；(3) 不能用 block-scaling；(4) 不能用 2-CTA UMMA。等于花 B200 的钱跑 H100 的 kernel 。
怎么验证 kernel 真在用 Blackwell 路径
(1) 看 SASS ：cuobjdump --dump-sass，HGMMA / QGMMA 是老 Hopper 路径，HMMA / QMMA / OMMA 才是真正 5th-gen；(2) 看 atom ：用 SM100_MMA_* 而非 SM90_MMA_*；(3) 看 PTX ：grep tcgen05，没有就是没用上。
迁移路径
把所有 SM90_MMA_* atom 换成 SM100_MMA_*，让 CUTLASS 重选 collective builder——后者会自动配合 TMA + TMEM。不能机械复制旧 kernel 改架构 flag 。

Layer 4

CTA Pair · 把两块 TMEM 拼成一块、让 SMEM 流量砍半

2-SM UMMA：cluster_layout_vmnk + TMA multicast + leader CTA 协调

单 SM 的 TMEM 容量 256 KB——已经能装 128×256 的 FP16 累加器（占半个 TMEM）。想让 tile 再大就必须跨 SM 。Blackwell 的解法：2-CTA UMMA ，让两个相邻 CTA 联合发射一条 MMA，操作数沿 M 维拆分，B 通过 multicast 共享。NVIDIA 官方图：

4.1 它解决了什么问题？

Before
两个独立的 1-CTA UMMA：CTA0 算 128×256 + CTA1 算 128×256。各自完整加载 16×256 的 B ，SMEM 流量 = 2 × |B|。
Problem
FLOPs 一样、SMEM 流量翻倍——arithmetic intensity 减半 。GEMM 已经是 BW-bound 时这意味着 throughput 损失。
How · 2-CTA UMMA
一条 tcgen05.mma.cta_group::2，A 沿 M 切，B 通过 TMA multicast 共享。leader CTA (偶数 ID) 单线程发射。
After
FLOPs 不变，SMEM 流量 = 1 × |B|，砍半 。等价 arithmetic intensity 翻倍，对 BW-bound GEMM 直接换来 1.5–2× throughput。

4.2 怎么做到的？三个机制配合

4.2.1 cluster_layout_vmnk · 把 cluster 切成"V × M × N × K"

CuTe 用一个 4 维 layout 描述 cluster——其中 V 是peer ID（0 或 1，区分一对 CTA 中谁是哪个）：

// cluster_shape = (4, 4, 1)，pair-UMMA 的 AtomThrID = _2
auto cluster_layout_vmnk = tiled_divide(
    make_layout(cluster_shape),
    make_tile(typename TiledMMA::AtomThrID{}));
// 得到形如 ((2), 2, 4, 1) 的 4D layout：
//   V=2 表示每个 MMA 由 2 个 CTA 协作；
//   M=2 表示沿 M 维有 2 个 MMA-pair；
//   N=4 表示沿 N 维 4 个；K=1。

auto mma_v = blockIdx.x % size<0>(cluster_layout_vmnk);   // peer ID
ThrMMA cta_mma = tiled_mma.get_slice(mma_v);              // 按 peer 切片
Tensor tCgA   = cta_mma.partition_A(gA);                  // 仅本 CTA 的那一半

4.2.2 TMA multicast · 让 B 加载一次广播到两 SMEM

CTA0 和 CTA1 都需要同一份 B。Blackwell 用 16-bit bitmask 编码"哪些 CTA 接收"，TMA 从 GMEM 读一次，硬件 fabric 自动复制到所有目标 SMEM。bitmask 例：

// 4×4 cluster，CTA 0 (peer=0)：A 与同行 CTA 共享，B 与同列共享
tma_bitmask_a = 0x1111;  // 同行
tma_bitmask_b = 0x000f;  // 同列
copy(tma_A.with(tma_bitmask_a), gA, sA);
copy(tma_B.with(tma_bitmask_b), gB, sB);

4.2.3 leader CTA + cross-CTA mbarrier

2-CTA UMMA只能从偶数 CTA 发射（leader）。但奇数 CTA 也要等 mma 完成 ，否则不知道 SMEM 何时可被覆盖。机制：

  • 奇数 CTA 的 TMA 用 cta_group::2 qualifier 发起，arrive 到偶数 CTA 的 mbarrier（通过清掉 mbar 地址 bit 24，找到 peer 的 mbar）。
  • umma_arrive_multicast_2x1SM(mma_mbar, mask) 在偶数 CTA 发射后，把完成事件 multicast 到所有依赖该 tile 的 CTA。
  • 所有 CTA wait 同一个 mma_mbar 到达后才能发起下一轮 TMA。

4.2.4 bit-24 mbarrier 地址 trick · 几乎没文档的硬件细节

问题
2-CTA UMMA 时，奇数 CTA（peer）需要 arrive 到偶数 CTA（leader）的 mbarrier——但 mbar 在 SMEM 里，peer CTA 怎么找到 leader 的 SMEM 地址 ？跨 CTA 的 SMEM 在 Hopper 上需要 cluster-scope 寻址（distributed SMEM），软件要显式算 peer SMEM 偏移。
"暴力解" 的代价
如果让 software 维护 cluster 内 SMEM 地址表，每次跨 CTA arrive 都要：(1) 算 peer CTA ID；(2) 查表拿 peer SMEM base addr；(3) 加偏移到 mbar；(4) 发 atomic arrive。每条都吃 SM cycle，对 high-frequency arrive 是性能杀手。
NVIDIA 的硬件 trick
cluster 内的 CTA pair 的 SMEM 物理地址仅差 bit 24 ——NVIDIA 在 cluster network 路由层这样布局。所以 peer CTA 的 mbar 地址 = leader_mbar_addr ^ (1 << 24)（清掉/翻转 bit 24）。一条 XOR 指令就拿到 peer mbar，0 查表、0 算偏移 。硬件 multicast unit 利用这点把一条 atomic 拆成两条同时发到两个 SMEM bank。
实战影响
(1) 软件层完全不感知——CuTe umma_arrive_multicast_2x1SM 内部就是这个 XOR；(2) 但一旦你手写 PTX inline asm ，必须知道这个 trick 才能正确填 mbar 地址；(3) 这个 trick仅在 cluster 大小为 2 时成立 ，更大 cluster 需要不同 routing——所以 cta_group::2 是个独立 PTX 修饰符，不是 cta_group::4/8。
⚠️ 这个 trick 几乎没正式文档 ——主要靠 Colfax 的 GEMM Tutorial 提到一句、加上 SemiAnalysis 的 Blackwell dissection。NVIDIA PTX 文档只说 tcgen05.mma.cta_group::2 会自动 multicast，不解释怎么找 peer 地址。这就是 CuTe 抽象的价值——把这种 hidden hardware quirk 完全包掉 。

DEEP DIVE 为什么 NVIDIA 不直接把单 SM 的 TMEM 扩到 512 KB

看起来"扩 TMEM"比"造 cluster fabric"简单得多。但 TMEM 的物理布局（128 lanes × 512 cols）与 5th-gen Tensor Core 的 spatial array 紧密耦合——TMEM 加宽会影响 Tensor Core 的 spatial array 几何，连锁修改面积爆炸。反过来，cluster + mbarrier 在 Hopper 已经准备好 ，只需要把 mma 指令支持 cta_group::2、再让 TMA 支持 cta_group::2 arrive、扩展 mbarrier 的多播 arrive 即可——用现成的基础设施换更大的 tile 。

4.3 局限

  • cluster shape 必须能被 pair-UMMA 的 AtomThrID 整除（M 维必须 ≥ 2 且偶数）；
  • cluster 仍然必须在同一个 GPC 内 ，最大 size 8（默认）/ 16（opt-in）；
  • 2-CTA UMMA 只支持 M=128 或 M=256 的 atom 形状。

5.2 为什么“全 warp 都是双重身份”成立？

技术上只需要两个条件：

✅ 让 warp 自由切角色的硬件前提
  1. 累加器不在 RF 里 。否则 warp 切角色 = 累加器交接 = 必须 spill 到 SMEM。
  2. 所有异步事件都能挂 mbarrier 。否则跨 warp 协调要写一堆 fence。

条件 1 由 TMEM 提供，条件 2 由 tcgen05 + mbarrier 提供。硬件设计与编程模型设计在这里是同一件事 。

5.3 一个 SM 上 17 个 warp 怎么协调？mbarrier 矩阵

每条箭头都是一个独立 mbarrier。例如 FA4 的实现里，一个 SM 上的 mbarrier 大致是：

裸写 tcgen05.* PTX 是不现实的——descriptor 编码、TMEM 列分配、mbarrier 计数、swizzle 模式…CUTLASS 用 CuTe DSL 把它分层封装成 MMA Atom ：

Layer 6

调度增强 · Preferred Cluster + Dynamic Persistent Scheduler

把过去由编程者手工对齐的事下放到硬件 runtime

Hopper 引入 thread block cluster 时遗留两个调度上的痛点：(1) cluster shape 必须整除 grid，且要装得下 GPC——大 cluster 经常装不下 ；(2) static tile→SM 映射，遇到 SM 抢占就长尾 。Blackwell 一次性补了这两块。

6.1 Dynamic Persistent Scheduler · 长尾消失术

Before · Hopper static schedule
kernel launch 时 tile→SM 静态映射；SM 完成自己的 tile 后退出。
Problem
当某 SM 被其它 kernel 或上下文切换抢占，它的 tile 不会迁移到空闲 SM；148 个 SM 中最慢的一个决定整个 grid 的完成时间。Tensor Core 越快，长尾损失越显著 。
How · Cluster Launch Control
硬件维护一个 tile work-queue。SM 完成当前 tile 后atomic 取下一个 ，由硬件 runtime 全程编排。
After
负载自动均衡，任何空闲 SM 立即接收新 tile。CUTLASS 在 Blackwell 默认开启 dynamic scheduler，无需用户配置。

6.1.1 CLC engine 硬件实现细节 · 从 PTX 到 SMEM broadcast 的全链路

上面 BNA 框给出“硬件维护 tile work-queue”这一句，但具体是哪几块硬件、走什么 PTX、跟谁交互 没说。本节按硬件 → PTX → CUTLASS 三层拆开。

第一层 · 参与的硬件组件

第二层 · CLC 的 PTX 协议（CUDA 12.8 / PTX 8.7 新增两条指令）

// (1) 异步发起一次“我要下一个 tile”查询

                     [smem_addr], [mbar];

//       并 atomic 增加 mbar 的 tx-count 16 byte (= 128 bit)。
//       发起后 CTA 不阻塞，可以去做 epilogue 或下一轮 mma。

// (2) 等 mbar phase 翻转后，从 SMEM 解析 response

                     %canceled, [smem_addr];

                     %ctaid_x, [smem_addr];

                     %ctaid_y, [smem_addr];

128-bit response 字段布局（NVIDIA 未公开完整 bit 定义，但语义清晰）：

bit 127 ............................................. bit 0
[ canceled? | reserved | ctaid_z (~32) | ctaid_y (~32) | ctaid_x (~32) ]
   1 bit

canceled = true 表示“work queue 已耗尽，本 CTA 应退出”；否则 (ctaid_x, ctaid_y, ctaid_z) 是下一个 cluster 的 leading CTA 坐标 ——cluster 内其它 CTA 的坐标由 leading + cluster shape 算出。

第三层 · CLC engine 内部 7 步行为

  1. 接收 query ：CTA 通过 GPU 内部专用 fabric 把 query 路由到 GPC-local CLC engine（避免跨 chip atomic）；
  2. Atomic 取号 ：CLC 内部维护 sub-counter（per-GPC 分布式），原子 increment 拿到 next_idx；
  3. 判断终止 ：if next_idx >= total_clusters → response.canceled = true，跳到第 6 步；
  4. 坐标编码 ：把 next_idx 通过 swizzling pattern（CUTLASS raster order，比如 along-N / along-M / Hilbert）转成 (ctaid_x, ctaid_y, ctaid_z)，打包进 128-bit response；
  5. Cluster broadcast ：response 通过 cluster network 写到cluster 内所有 CTA 的同一个 SMEM 地址 ——而非只发给 query 发起的 CTA；
  6. mbarrier arrive ：CLC engine atomic 增加 mbar 的 tx-count 16 byte，触发 phase 翻转；
  7. CTA 唤醒 ：所有 cluster 内 CTA 的 wait_barrier 同时解锁，读 SMEM 拿到 tile 坐标。

第五层 · CUTLASS 4.x 上的实际调用

// host 端配置
PersistentTileSchedulerSm100<...> scheduler(problem_shape, cluster_shape);

// kernel 内 mainloop
__device__ void mainloop() {
    auto work_tile = scheduler.initial_work();          // 来自 launch 时 blockIdx

    while (work_tile.is_valid) {
        compute_gemm_tile(work_tile.tile_coord);

        // 异步 fetch 下一个 tile（背后是 try_cancel）
        scheduler.fetch_next_work_async(smem_buf, mbar);

        // 这里和 epilogue 重叠！scheduler 开销藏在后面
        do_epilogue();

        // 等 CLC response
        scheduler.wait_for_next(mbar);
        work_tile = scheduler.parse_response(smem_buf);
        // 展开后 = query_cancel.is_canceled + get_first_ctaid
    }
}
✅ 关键 trick: 异步 fetch + epilogue overlap

fetch_next_work_async 是异步 ——CTA 发起 query 后不等 response，先去做 epilogue。CLC engine 在 epilogue 期间已经把下一个 tile 算好并 broadcast 到 SMEM。scheduler 开销被完全 hide 在 epilogue 后面 ，关键路径上只剩 mma + epilogue 本身。

Cluster-scope 妙处一
2-CTA UMMA 模式：只有 leader CTA 发起 query；peer CTA 直接从同一 SMEM 地址读 response。Peer CTA 0 atomic 开销 。
原本的代价
没有 cluster broadcast 的话，2-CTA 中两个 CTA 都要各自 atomic 取号，且必须想办法“取到同一个 tile”——要么各自取再 reduce 协调（额外 ~50 cycle），要么 leader 取了通过 SMEM 通知（写软件 mbarrier）。
大 cluster (4-8 CTA) 的延伸
整个 cluster 共享一个 work item（如 4x1 cluster 处理 256×256 输出）。1 次 query 得到全 cluster 的工作分配；query 次数减少 4-8 倍。
设计哲学
CLC 把scheduling 粒度从 CTA 提到 cluster ，与 cluster network 的 broadcast 能力对齐。这是“硬件 scope 与软件 scope 同构”的范例——TMA multicast 是另一个同构例子（也是 cluster scope broadcast）。
💡 把 CLC 与本文其它机制对应：(1) CLC 之于调度 ，正如 Decompression Engine 之于权重 unpacking 、Tensor Core 之于MMA ——三者都是“曾经吃 SM 算力的事，搬到专用硬件 unit”的范例；(2) CLC 与 mbarrier 的耦合（response 自动 arrive mbar）是“所有异步引擎接同一种通知机制”原则的延伸（见 Layer 1.3）。

6.2 Preferred Cluster · 大 cluster 装不下时怎么办

Hopper 引入 cluster 时，cluster shape 必须整除 grid 且能装进 GPC——一旦 SM 数与 cluster shape 不整除就有 SM 闲置。NVIDIA 官方“四象限”图把这件事讲透：

Before · Hopper
cluster size 4 装不下剩 6 SM？要么浪费这 6 SM，要么放弃 cluster，要么人工拆 grid。
Problem
cluster shape 是编译期或 launch 期写死 的；不同 GPU SKU（B100/B200/RTX5080）SM 数都不同，写一份 kernel 配多个 SKU 几乎不可能。

DEEP DIVE 调度增强是 Blackwell 异步设计的“必然推论”

Tensor Core 越快、TMEM 越大、CTA Pair 让 tile 越大——每个 tile 的执行时间相对越短 。这意味着传统的“长 tile 摊平 launch overhead”策略失灵：launch 一个 cluster 的开销和单个 tile 时间已经接近，必须靠 runtime 持续喂入而不是按 grid 一次性划好 。Cluster Launch Control 是这种 trend 的工程实现：把 grid 描述变成 work-queue，让硬件像 task scheduler 一样运转。

第二层 · DE API 调用模式（CUDA 13.x 引入）

// host 端：注册压缩张量

    /*compressed_addr*/   d_compressed_weight,

    [smem_dst], [tmap, coord_x, coord_y], [mbar];

这里关键是 .with.decompress qualifier — 把 TMA 的 GMEM→SMEM 路径中间插入 DE 处理。API 上和普通 TMA 几乎一样 ，开发者不需要写 decode kernel。

第三层 · DE engine 内部 7 步流水

  1. Tensor map 解析 ：DE 收到 TMA 请求，解析 cudaTensorMapDecompressed_t 拿到压缩格式 + chunk size + base address；
  2. HBM bulk fetch ：HBM3e fetch unit 按 chunk 大小读压缩流到 DE 内部 input buffer；
  3. Format dispatch ：根据 format flag 路由到对应 decoder（Bitcomp/LZ4/Snappy/...）；
  4. Pipelined decode ：decoder 流式产出解压数据；不同 chunk 可以并行解压（取决于 pipeline depth，见 Table III 8-16 ops 并发）；
  5. Output staging ：解压数据先写 DE 内部 output buffer，做 alignment 对齐；
  6. SMEM inject ：通过专用 SMEM injector 写到目标 SM 的 SMEM 地址（绕过 L2，不抢带宽）；
  7. mbar arrive ：DE 状态机检测到 chunk 全部 inject 完成，原子触发 mbar 的 tx-count +chunk_bytes，触发 phase 翻转。

注意 DE 也接 mbar——所有 Blackwell 异步引擎都接同一种 mbar 同步原语 。这意味着一个 mainloop 可以同时等多种异步事件：

// 一个典型的 Blackwell mainloop 可能同时在等三种事件
cp.async.bulk.tensor with.decompress [sA], ..., [tma_mbar_a]; // DE 解压 weight A
cp.async.bulk.tensor              [sB], ..., [tma_mbar_b]; // 普通 TMA load activation B
tcgen05.mma [d_tmem], a, b, idesc;                        // mma
tcgen05.commit [mma_mbar];                                // mma retire

wait_barrier(tma_mbar_a, phase_a);  // 等 DE 完成
wait_barrier(tma_mbar_b, phase_b);  // 等 TMA 完成
wait_barrier(mma_mbar,   phase_c);  // 等 mma 完成
✅ “wired retire 路径”的统一抽象

把 §3.6（TC retire → mbar）、§6.1.1（CLC response → mbar）、§7.2.1（DE chunk done → mbar）三个机制并列看：Blackwell 的每种异步硬件 unit 都有专用 wired 路径直连 mbar engine 。这是“全异步”在硬件层面的架构性原则 ——不是某个 unit 的局部设计，而是所有 unit 都遵循同一种通知协议 。软件因此只需要 wait_barrier 一种 sync 原语。

应用二 · Mixed-Precision / Block-Scaled GEMM

MXFP8 / MXFP6 / MXFP4 / NVFP4 — 让低精度真正可用的硬件原语

这是 Blackwell 最重要的实战场景 ——LLM 推理几乎完全靠它。原始公式：

block-scaled MMA

9.1 Hopper 上为何做不出这种性能

Before · Hopper 软件 block-scaling

9.3 Scale 在 TMEM 中的多播布局

Scale 必须放 TMEM，且必须多播复制 4 份 ——因为每个 sub-partition 的 lane 都要在 mma 内部读 scale。从 GMEM 到 TMEM 的标准路径：

  1. GMEM → SMEM ：用 TMA 加载 scale matrix。但 scale 的 hardware layout 非常严格 (((32,4),REST_M),((16,4),REST_K)) : (((16,4),512*REST_K),((0,1),512))——若量化时是 K-major 紧凑存的，必须先 reshape+transpose 成这个 layout 再 TMA。
  2. SMEM → TMEM ：用 tcgen05.cp.cta_group.32x128b.warpx4。warpx4 qualifier 让一个 warp 的拷贝同时复制到所有 4 个 sub-partition 。
DEEP DIVE NVFP4 vs MXFP4 — 为什么 NVIDIA 自己造一套

OCP 标准的 MXFP4 用 block-32 + UE8M0 scale。NVIDIA 私有的 NVFP4 用 block-16 + UE4M3。差别看似细节，但实测：Mistral-7B 的 4-bit 推理 perplexity 退化，MXFP4 是 9%，NVFP4 是 4% ——粒度更细 + scale 表达更精准让一半的精度损失被挽回。NVIDIA 让硬件同时支持两种 spec，让用户在“标准兼容 vs 极致质量”间自选。这也是"低精度需要硬件帮忙才能可用"这个判断的实证。

9.7 MXFP8 scale layout 实战拆解 · 三个最容易踩的坑

读 §9.3 的 scale TMEM 多播布局图（F16）后，最常见的疑问就是"那 GMEM 端 scale 该怎么排" 。本节用三个独立小问拆透三个最不直观的点：(1) 为什么 scale 必须在 GMEM 提前 swizzle、(2) "M 维 128 个 scale 必须连续" 的精确语义、(3) 128×4 tile layout 是怎么来的。

9.7.1 问题一 · 为什么 scale 必须在 GMEM 端提前 swizzle？

Before · 量化器自然输出的 layout
训练后量化（PTQ）或 calibration 输出 scale 时，最自然的存法是K-major ：scale[m, k_block_id] 沿 K 方向连续。一个 4096×4096 weight 在 block-32 下产生 4096 × 128 = 524,288 个 scale，按 K-major 紧凑存。
Problem · TMA 不是任意 permute 引擎
TMA 的能力边界：能 做 box-shaped tile 拷贝、加 stride、加 swizzle pattern、加 padding（FP4 的 16U4_ALIGN16B）；不能 任意 transpose / interleave / reshape。但 mxfp8 在 TMEM 中要求的硬件 layout 是 4-level nested（(((32,4),REST_M),((16,4),REST_K))），其中 32 lane × 4 col 是 atom，再 outer-product M/K 维。K-major 的 GMEM 直接喂给 TMA 后，SMEM 中得到的不是 这个 nested layout——硬件 mma 读 scale 会读到错位置。
How · 离线一次 swizzle，运行时 TMA 直读
把 scale 在量化时就按 hardware layout 重排（DeepGEMM 的 pack_fp8_scales_for_blackwell.py 等脚本）。原本的 K-major 索引 scale[m, k_block] 通过 reshape + transpose 变成 scale_packed[outer_m, outer_k, lane_in_atom, col_in_atom]，与 TMEM 物理 layout 一一对应。TMA 拿到这个排好的 GMEM 后一次到位 送到 SMEM；tcgen05.cp.warpx4 再把 SMEM 多播到 4 个 sub-partition。
After · 两条路径的 cycle 账
不预 swizzle 的方案：每次 GEMM 启动前要 launch 一个 reshape kernel（额外 SMEM round-trip 200-500 cyc + 一次 atomic counter）。预 swizzle 后每次 GEMM 的 scale 加载路径 = 1 次 TMA + 1 次 tcgen05.cp，0 额外 SM 开销 。LLM 推理一次 forward 调上千次 GEMM——预 swizzle 是必选 。量化是离线一次的成本，推理是反复的，所以要把代价摊到 offline。
💡 实战工具： DeepSeek DeepGEMM 仓库里 tools/pack_fp8_scales_for_blackwell.py 给训好的模型在 deploy 阶段做这次重排；NVIDIA TensorRT-LLM 的量化导出器自带 swizzle 路径；ModelOpt 也有对应 transform pass。用框架自带的 packing ，别手撕——nested layout 的 stride 计算非常容易写错。

9.7.2 问题二 · "M 维 128 个 scale 必须连续" 的精确语义

先把数算清。Tile 设 M=128, N=64, K=32, block_size=32 (mxfp8)：

项计算数量
|A| 数据元素M × K = 128 × 324096 个 FP8
|B| 数据元素N × K = 64 × 322048 个 FP8
|A| scale 数M × ⌈K/32⌉ = 128 × 1128 个 UE8M0
|B| scale 数N × ⌈K/32⌉ = 64 × 164 个 UE8M0
合计 scale128 + 64192 个 UE8M0 = 192 字节

注意 warpgroup 严格定义是 4 warps × 32 thread = 128 thread（不是 256）。所以一个 warpgroup 一次 cooperative tcgen05.ld 刚好覆盖 128 个 lane，每 thread 拿 1 个 M-dim scale。

Before · 假如 scale 不连续摆放
比如 scale 在 SMEM/TMEM 里散布，需要每个 thread 自己算偏移、做 gather load。每条 tcgen05.ld 路径都是不规则访问，bank conflict 拉满。
Problem · TMEM 的物理约束
TMEM 是 4 个 sub-partition，warp K 只能访问 lane 32K..32K+31。"非连续 scale" 意味着同一个 warp 要访问的 32 个 scale 散布在 multiple sub-partition——硬件 access port 直接禁止 ，会编译期报错或运行时无效结果。
How · 沿 lane 维严格对齐
scale 在 TMEM 中按 (lane=row mod 128, col=row/128) 摆。即：

thread 0   ────► lane 0   ────► scale_A[row=0]
thread 1   ────► lane 1   ────► scale_A[row=1]
...
thread 31  ────► lane 31  ────► scale_A[row=31]   ← warp 0
thread 32  ────► lane 32  ────► scale_A[row=32]
...
thread 127 ────► lane 127 ────► scale_A[row=127]  ← warp 3

4 个 warp 的 tcgen05.ld 同时 各取自己 sub-partition 的 32 lane，128 个 scale 一次 read 完成。

After · "连续" 的精确含义
"M 维 128 个 scale 必须连续" 不是说 SMEM 字节连续，而是沿 TMEM lane 维一一对应映射 128 个 row ——每 32 个 row 对应一个 warp 能独占访问的 sub-partition。当 M > 128 时（比如 M=256），第 129–256 行的 scale 占下一列（column），靠 2-bit SFA_ID 索引切换；不会跨 sub-partition borders。

9.7.3 问题三 · 128×4 tile layout 是 TMEM 限制吗？

答案：128 来自 TMEM、4 来自 MMA atom 设计 ，二者结合产生 128×4 几何。

📐 拆开两个维度看
来源动机
128 lanesTMEM 物理硬件TMEM 就是 128 lanes × 512 cols × 32-bit；想覆盖 128-row 累加器，scale 必须沿 lane 维分布。硬件 access port 成本权衡（见 §14.5）。
4 colsMMA atom + SFA_ID 设计当 atom_K=16, block_size=32 时一条 mma 仅消耗 半个 block 的 scale → 2 条 mma 才用完 1 个 block；NVIDIA 把 4 个连续 mma 的 scale 摆成 4 个 sub-column，每条 mma 用 2-bit SFA_ID 选自己那一列 。
Before · 假设是 128×1（不打包）
每条 mma 只读 1 个 scale，每条 mma 要发起一次 TMEM 读。一个 atom row 有 4 条 mma，要读 4 次 TMEM。
Problem · TMEM 读取的固定开销
每次 tcgen05.ld 触发都有 ~10 cycle 的固定 setup（地址解码 + sub-partition routing + bank arbitration）。连续 4 条 mma × 4 次 ld = 40 cycle 纯开销。Tensor Core 单条 mma 才 11 cycle，scale 加载占了 mma 时间的 ~90% 。
How · 4 个 scale 打包到 1 个 32-bit cell
1 个 TMEM cell = 32-bit = 4 byte = 4 个 8-bit UE8M0 scale。NVIDIA 把 4 条连续 mma 的 scale 摆在同一个 cell 的 4 个 sub-column ，硬件读一次 cell 就拿到 4 条 mma 的 scale。SFA_ID 只是选择当前 mma 该用 cell 内的哪个字节 ——不触发新一次 TMEM 读。
After · 几何 = 性能 = 物理对齐
"4" 不是巧合：(a) 32-bit cell ÷ 8-bit scale = 4；(b) 一个 atom row 对应 4 条 mma；(c) 2-bit SFA_ID 刚好编码 4 个 sub-column。三件事在 4 这个数字上对齐 ，scale 加载摊销到 1 个 TMEM 读 / 4 条 mma。每条 mma 的 scale 开销摊到 ~2.5 cycle，不再成为瓶颈 。

NVFP4 (block-16) 让 scale 占用翻 4 倍 到 8 KB（含所有 4 sub-partition 的多播副本是 32 KB）。256 KB TMEM 中本来用一半给累加器（128 KB）、留一半给下一轮 K-tile 流水（128 KB）；现在 scale 要占 32 KB，意味着累加器 tile 必须缩 ，或者放弃 stage=2 流水。这是 NVFP4 实测 % BW 利用率比 MXFP8 低（47.6% vs 67.3%）的非显然根因 之一——不是单纯 dtype 大小问题。

💡 三个问题串起来的一句话总结：
(1) GMEM 必须预 swizzle，因为 TMA 不能任意 permute；
(2) M 维 128 个 scale "连续" 的精确语义是沿 lane 维一一对应 4 个 sub-partition ，不是字节连续；
(3) 128×4 中 128 来自 TMEM 物理 lane 数、4 来自 "4 byte cell ÷ 1 byte scale = 4 条 mma 共享" 的几何对齐。
三件事一起决定了 mxfp8 path 的 fast track。

Layer 10

应用三 · Sub-Byte (FP4 / FP6) GEMM · TMA 顺手做 unpack

GMEM 紧凑、SMEM 16-byte 对齐、TMEM 1-byte/elem — 三种格式之间的转换谁来做

FP4 / FP6 在 GMEM 中以密堆 形式存放（FP4 = 2 elem/byte），但 Tensor Core 要求 SMEM/TMEM 中每个元素占 1 字节（带 padding）。这件事 Blackwell 让 TMA 来做——通过特殊 tensor map 类型。

10.1 三种格式不兼容的根源

Before
三方各有刚需：GMEM 想紧凑（节省 HBM）；Tensor Core 只接受 1-byte/elem（spatial array 寻址简单）；SMEM 必须 16-byte 对齐（swizzle 单位）。
Problem
“在哪一步把 packed 变 padded？” 软件 unpack 占 cycle，Tensor Core 内部 unpack 增加 spatial array 复杂度。
How
让 TMA 顺手做 unpack ——TMA 本来就是带 layout descriptor 的 DMA 引擎，加 unpack 路径几乎无成本。新增 tensor map 类型 CU_TENSOR_MAP_DATA_TYPE_16U4_ALIGN16B。
After
GMEM 紧凑 → TMA 自动加 padding → SMEM 16-byte 对齐 → tcgen05.cp 拷到 TMEM 每 elem 1 byte。整条路径对上层 kernel 透明。

10.3 对齐约束变严了

  • GMEM base address ：32-byte 对齐（其它精度只要 16-byte）；
  • Leading dimension ：必须是 128 elements 的倍数；
  • 实际推荐：FP4 64-byte 对齐，FP6 96-byte 对齐——同时满足上面两条。
⚠️ 实战陷阱： 从 HuggingFace 拉下来的 4-bit 量化模型若按 FP16 layout 排列（每行非 128 倍数），无法直接用 TMA 加载 ，必须 offline 重排。这是 deploying inference 的一个常见 blocker。

10.4 LLM 推理收益

Sub-byte 的真正价值不是 throughput（已经被 Layer 9 覆盖），而是HBM 容量 ：

  • 200 B 模型 × FP16 = 400 GB → 不能放进单 GPU；
  • 200 B × FP8 = 200 GB → 仅 H100 80GB 装不下；
  • 200 B × FP4 = 100 GB → 可装进单 B200 (192 GB)；
  • 200 B × FP4 + Bitcomp 3:1 = 33 GB → 可装进单 GB203 (16 GB) 的两 GPU 上。

这是为什么 DeepSeek-V3 / Qwen 系列都在明确支持 NVFP4 推理 ——硬件能容、能算、能做对，软件就能上。

Layer 11

应用四 · FlashAttention V4 · 为非对称硬件而生

Tensor Core 翻倍但 SFU/SMEM 没动 — 算法-硬件协同设计的范例

Hopper → Blackwell 是非对称升级 ——某些资源翻倍，某些资源完全没动。这种非对称性使得 FA3 在 Blackwell 上不再最优。

11.1 为什么 FA3 不再最优

❌ FA3 在 B200 上的两个瓶颈
  1. Forward 受 SFU 限制 ：硬件 MUFU.EX2 只有 16 ops/cycle/SM。Tensor Core 翻倍后，softmax exp 的相对开销翻倍，变成关键路径 。
  2. Backward 受 SMEM 带宽限制 ：递推 dQ/dK/dV 时反复读写 SMEM，带宽没涨意味着无法跟上 Tensor Core。

11.2 FA4 的四个核心改动

11.2.1 软件 exp · 把 SFU 工作分给 FMA

Before
所有 exp 走硬件 MUFU.EX2，16 ops/cycle/SM。
Problem
Tensor Core 翻倍后 softmax 行权重计算成瓶颈。
How
(1) Cody-Waite range reduction：2^x = 2^n · 2^f；(2) Horner 形式三阶多项式近似 2^f ≈ p₀ + p₁f + p₂f² + p₃f³；(3) 用 FMA 单元算多项式（Tensor Core 跑时 FMA 闲）。
After
软件 exp 总指令数更多但用的是富余的 FMA 算力 ；MUFU 不再瓶颈。

11.2.2 Conditional Rescale · 把 normalize 移出关键路径

Before · 标准 online softmax
每个 tile 都用 exp(diff) rescale 累加器。
Problem
“乘 exp(diff)” 在每个 tile 的 critical path 上。
How
仅当 max 跳变超过阈值 τ 时才 rescale；最终结果靠最后一次全局 normalize 修正。
After
大部分 tile 的 rescale 操作消失；最终精度等价。

11.2.3 Ping-Pong 双 Q tile

每个 CTA 同时处理 Q tile A 和 B，让 tile A 的 softmax 与 tile B 的 PV-GEMM 重叠。softmax 完全藏在 PV-GEMM 时段内 ，关键路径上只剩 GEMM。

11.2.4 TMEM 跨 phase 复用

FA4 里同一块 TMEM 区域要先后承担 S（QKᵀ）、P（softmax(S)）、O 累加器三个角色——靠流水位置切换，不会同时占用。这只有累加器在 TMEM 才做得到 ，RF 切换需要昂贵的 spill。

DEEP DIVE FA4 为什么用 CuTe-DSL 而不是 C++ 模板

FA4 完全用 CUTLASS 的 Python DSL（CuTe-DSL）实现。原因是 C++ 模板版本编译一次要十几分钟——对 FA4 这种"流水阶段、warpgroup 角色、TMEM partition"都需要反复试错的算法是致命的。DSL 编译时间缩短 20–30×，让算法开发者可以像写 Triton 一样迭代，但底层仍然是 PTX 级控制。这也预示了未来 Blackwell 上的 frontier kernel 都会走 DSL 路径——kernel 复杂度已经超过 C++ 模板的人类可维护极限 。

11.4 这是"算法-硬件协同设计"的范例

FA4 的核心 insight 是：瓶颈不是 Tensor Core，是 softmax 。当 Tensor Core 翻倍而 SFU 没翻倍，最优算法必须：

  • 把 exp 往 FMA 上挪（软件 exp）；
  • 把 rescale 移出关键路径（conditional rescale）；
  • 用 ping-pong 把 softmax 完全藏在 PV-GEMM 后面；
  • 用 TMEM 跨 phase 复用累加器（RF 做不到）。

这些都是只看 Hopper 一代是想不到要这么做的 ——必须看到 Blackwell 的非对称硬件升级，才能反推出算法应该往哪改。这就是协同设计的本质 。

13.1.1 这张图的优化点：用 CuTe 抽象屏蔽掉 90% 的 PTX 复杂度

裸写 PTX 的痛苦
没有 CuTe 的话，开发者要手写：(1) 32-bit SMEM descriptor 编码（leading byte offset / stride byte offset / swizzle mode / base offset）；(2) tcgen05 instruction descriptor（dtype / negate / sparsity 等 32 个 bit）；(3) lane → SMEM bank 映射避免 conflict。每个新精度都要重写一遍。
具体痛点
SM100_MMA_F16BF16_SS 的 PTX 字符串实际是 150+ 个字符的 inline asm 模板，包含 5 个寄存器约束 + 字节级 descriptor 编码。错一位就跑出错误结果而不是编译错误（PTX assembler 不会校验 descriptor 语义）。
CuTe 怎么省事
把 PTX 包成 atom：using AtomMma = SM100_MMA_F16BF16_SS<_128, _256, _16, float>; + TiledMMA tiled_mma = make_tiled_mma(AtomMma{});。后面 cute::gemm(tiled_mma, tCrA, tCrB, tCtAcc) 一行展开。descriptor 编码、bank conflict、swizzle 全部由 atom traits 自动算出。
优化收益
(1) 同一个 kernel 模板换精度只需换 atom 名字 (F16BF16_SS → F16_TF32_SS)；(2) descriptor 写错的概率从~30% 降到 0；(3) 代码量从 200+ 行 inline PTX 降到 ~10 行 CuTe 调用。这是"为什么 Blackwell kernel 一律用 CUTLASS / CuTe-DSL"的根本原因 。

13.1.2 这张图的优化点：从 thread-level 到 CTA-level 的 partitioning 转换

SM90 (Hopper) 的做法
MMA 是 CTA-local + thread-level partitioning：auto thr_mma = tiled_mma.get_slice(threadIdx.x); 每个 thread 切自己的 tile 块。要写一堆 boilerplate 处理 thread → MMA tile 的映射。
问题
thread-level partitioning 假设"每个 thread 都参与 MMA"——这在 Hopper wgmma 上成立（128 thread 一组）。但 Blackwell tcgen05.mma 是单线程发射 ，剩 31 个 thread 无事可做；如果继续用 thread-level partition，会让单线程持有"应该所有 thread 共享"的 fragment——逻辑上错。
SM100 (Blackwell) 的解
升一层：MMA 是 Cluster-local + CTA-level partitioning。代码用 cta_tiler 切 problem → 给 CTA；用 mma_tiler 切 problem → 给 MMA atom。thr_mma = tiled_mma.get_slice(0)（peer ID 0 表示本 CTA）而不是 threadIdx.x。两个 evenly_divides 断言强制 problem shape 整除，否则编译期失败。
给开发者的影响
只用改一行：tiled_mma.get_slice(threadIdx.x) → tiled_mma.get_slice(_0{})。但语义上完全变了：从"我这条 thread 该做什么"变成"我这个 CTA 该做什么"。FA4 / DeepGEMM 这类多 warp 角色 kernel 必须用 CTA-level，否则不同角色 warp 算出不同的 tile slice 会乱套。

13.2 Feature 2 · Using TMA · 把 GMEM 搬入交给硬件 DMA

第二步把 cooperative copy 换成 TMA：从 GMEM → SMEM 不再吃 SM 算力。Blackwell 上 TMA 1-D / 2-D atom 沿用 Hopper SM90 的 TMA atom ——这是 CuTe 抽象成功的体现：硬件升了一代，软件 atom 不用换。

13.2.1 这张图的优化点：cooperative copy → TMA 救出三类 SM 资源

cooperative_copy 的代价
(1) 每个 thread 算 GMEM 偏移量（占 INT32 单元）；(2) 每个 thread 发起 LDGSTS（占 LSU bandwidth）；(3) sync warps 等 LDGSTS 完成（warp scheduler 占用）。一个 128×128 tile 的 cooperative copy 大约消耗 32 thread × 16 LDGSTS = 500+ cycle 的 SM 资源。
问题
这 500 cycle 期间 Tensor Core 必须 idle 等数据。Blackwell tcgen05.mma 单条延迟仅 11 cyc——cooperative copy 让 mainloop 节奏被访存而非计算决定 。
TMA 怎么救
TMA 是硬件 DMA ：单线程一条 cp.async.bulk.tensor.2d 让 DMA fabric 自己跑，不占用任何 SM 算力 / LSU / warp scheduler 。Blackwell 沿用 Hopper SM90_TMA_LOAD_5D atom 的好处是同一份 CUTLASS kernel 跨 H100/B200 通用 ——只要换 SM100_MMA atom，TMA 部分不动。
实测收益
把 cooperative copy 换 TMA 后，一个 BF16 GEMM kernel：(1) SM 占用率从 ~50% 提到 ~90%（剩下 50% 之前都在算地址）；(2) Tensor Core 利用率从 ~60% 提到 ~90%（idle gap 消失）；(3) RF 用量降低 ~30%（不再需要 thread-private GMEM 偏移）。

13.2.2 这张图的优化点：用 mbarrier 把 TMA 完成事件挂上 producer-consumer 流水

TMA 的"难点"
TMA 是异步的——发起后不知道何时完成。如果用 polling 检查"数据到齐了吗"，就把 TMA 的优势抹平了。
具体场景
多 stage 流水里，consumer warp 必须知道"stage k 的 SMEM 已经被 TMA 填好"才能发射 mma；同时 TMA warp 必须知道"stage k 的 SMEM 已被 consumer 用完"才能开始填下一轮。
TMA + mbarrier 三步走
(1) auto tma_a = make_tma_atom(...) 定义 atom；(2) 通过 cluster_layout_vmnk + cta_v_in_cluster 算出 multicast mask（决定哪些 CTA 接收数据）；(3) copy(tma_a.with(mbar, mcast_mask), tCgA, tCsA) 发起 TMA 同时把完成事件 atomic 累加到 mbar 的 tx-count 。Consumer 用 wait_barrier(mbar, phase) 等 phase 翻转。
关键优化
(1) 0 行 polling 代码；(2) TMA 完成事件由硬件 fabric arrive，不消耗 SM；(3) multicast mask 让 cluster 内多个 CTA 共享一份 TMA — 例如 4-CTA cluster 共享 B 矩阵时HBM 带宽需求降至 1/4 。这是 Layer 4 中"2-CTA UMMA SMEM 流量砍半"的一般化形式。

13.3 Feature 3 · MMA.2SM 与 TMA.2SM · 跨 SM 协作的"换 atom 即可"

第三步把 1-CTA UMMA 升级到 2-CTA UMMA。CuTe 的做法是换 MMA Atom（从 _SS 换成 _2x1SM_SS），ThrID 从 _1 变 _2，CTA-level partitioning 自动生效。这一步的优化点不在算更多 FLOPs，而在 SMEM 流量砍半 。

13.3.1 这张图的优化点：把"硬件能力切换"压成"换 typedef"

没有 CuTe atom 的话怎么写 2-CTA UMMA
(1) 写 cluster setup（grid dim, cluster dim）；(2) 算 leader CTA 选举；(3) 写 cross-CTA mbarrier setup（要找到 peer CTA 的 mbar 地址，需要清掉 bit 24）；(4) 写 multicast mask；(5) 改 thread → tile 映射逻辑。每个细节都是 PTX-level 陷阱。
陷阱实例
cross-CTA mbarrier 的地址必须在 leader CTA 的 SMEM 内 ——peer CTA arrive 时硬件会自动 routing。如果两个 CTA 各自 init 自己的 mbarrier，arrive 会发到错的地方，kernel hang。
CuTe 把这一切包成 atom
右侧代码就是完整的 atom 定义：(1) MMA Op = tcgen05.mma.cta_group::2.kind::f16 PTX 包装，固定 4 个 inline asm 参数；(2) MMA Traits = using ThrID = Layout<_2>（2 个 CTA 一起）+ ALayout/BLayout/CLayout（描述每个 peer 持有哪一片）+ FrgTypeA = UMMA::smem_desc。开发者只需 typedef 这一个 atom，所有跨 CTA 复杂度被消化 。
优化效果
从 1-CTA 升 2-CTA：代码改动 = 1 行（atom typedef）+ 1 行（cluster_dim）。性能改动：B 操作数 SMEM 流量砍半（multicast 共享）→ arithmetic intensity 翻倍 → BW-bound GEMM 直接获 1.5–2× throughput。

13.3.2 这张图的优化点：为什么沿 M 切而不是 N 切或 K 切

三种切法的对比
沿 M 切：CTA0 算 row 0-127，CTA1 算 row 128-255，B 共享 。沿 N 切：A 共享，B 不共享。沿 K 切：A、B 都不共享，但累加器需要跨 CTA reduce。
K 切的致命问题
K 切意味着 CTA0 算 D += A·B[:, 0:K/2]，CTA1 算 D += A·B[:, K/2:K]，最后必须把两个 CTA 的累加器加起来 ——要么走 SMEM 跨 CTA reduce（巨慢），要么走 atomic（破坏精度）。所以 K 切对 fp16/bf16 + fp32 累加是禁区 。
M 切 vs N 切的选择
都可行。NVIDIA 选 M 切，因为：(1) M 维通常 = batch×seq_len，是最大的维度 ，切 M 让两个 CTA 处理的 row 不重合；(2) B 矩阵是权重（在 LLM 中 K×N 维度都不大），multicast 一份能让两个 CTA 共享，HBM 带宽节约最大。
具体优化数字
对 m=256, n=256, k=16 的 atom：(1) 不切：每 CTA 加载 |A|+|B| = 256×16+16×256 = 8192 elem；(2) M 切 + B multicast：每 CTA 加载 128×16+16×256 = 6144 elem (-25%)；(3) 两个 CTA 总加载 12288 elem vs 16384 elem (-25%)。arithmetic intensity 提升 33% 。

13.3.3 这张图的优化点：CuTe Traits 设计的优雅

对比传统做法
没有 Traits 抽象时，1-CTA → 2-CTA 升级要改：partition_A、partition_B、partition_C 三个函数 + thread→tile 映射 + atomic 行为切换。改动散落在 6+ 个文件。
易错点
partition_A / B / C 必须保持互相一致 ——否则会出现 CTA0 算 row 0-127 但加载 row 128-255 的 A 这种 bug，且不会编译错。
CuTe Traits 设计
把所有 partition 信息收敛到一个类型级 的 Traits 模板里。using ThrID = Layout<_2> 改了之后，partition_A/B/C 自动从 ThrID 推导出新的 layout。开发者只动一处，CuTe 编译期保证全局一致。
真实价值
升级 atom 不再需要"改完跑测试 → 找 bug → 再改 → 再跑"的痛苦循环。这种类型驱动的 invariant maintenance 是 CUTLASS 4.x 比 cuBLAS 灵活的核心原因。

13.3.4 这张图的优化点：三层 elect 嵌套的 throughput 含义

为什么需要三层 elect
2-CTA UMMA 必须由偶数 CTA 中的某一个 thread 发射。三层条件：(1) cta_in_cluster_v == 0（leader CTA）；(2) elect_one_warp()（leader warp，这个 warp 中只有 lane 0 是 active）；(3) elect_one_thread()（lane 0 thread）。
不三层会怎样
(1) 没有 leader_cta 检查 → 两个 CTA 都发射 → undefined behavior（mma instruction 双发）；(2) 没有 elect_one_warp → 4 个 warp 都尝试发射 → 重复发射 4 次；(3) 没有 elect_one_thread → 32 个 thread 都尝试 → 重复 32 次。性能 + 正确性双输 。
elect_one_* 的硬件机制
是 PTX 原生指令，硬件 atomic 选出 warp/lane 的 leader。比手动 if (lane_id == 0) 更安全（编译器知道这是 control divergence 不算分支预测）。
剩余 31 thread + 3 warp 干嘛
这是 Blackwell 编程范式的核心 — 它们不闲着 ，可以做 epilogue 准备（比如读 bias、准备 swizzle 模式）、做 softmax（FA4 case）、做下一轮 TMA load。这就是"全 warp 都是 producer & consumer"在代码层的实现。

13.4 Feature 4 · TMEM Accumulators · 把累加器从 RF 搬出去

第四步把累加器从 RF 搬到 TMEM。这是所有 Blackwell 优化里收益最大的一步 ——单这一步就能让 occupancy 翻倍、让 epilogue 与 mainloop 完全解耦、让 FA4 这种多角色 kernel 成为可能。

13.4.1 这张图的优化点：为什么 TMEM 需要 explicit alloc/dealloc

RF 是怎么管的
编译器自动管：声明一个 fragment，编译器分 register。RF 不够就 spill 到 local memory（变慢）。开发者不需要写 alloc/dealloc。
RF 的根本限制
每 SM 256 KB RF 在 occupancy=2 时每 thread 仅 ~512 byte。一个 128×256×fp32 的累加器占 128 KB，4 个 thread 持有 → 每 thread 32KB → occupancy 必须降到 1 。Latency hiding 完全没了。
TMEM 的解
跟 SMEM 一样 explicit 管理：TMEM::Allocator1Sm tmem_allocator; + tmem_allocator.allocate(num_columns, &shared_storage.tmem_base_ptr);。alloc 单位是column（最小 32，必须 power-of-2）。alloc 之后把 base_ptr 写到 SMEM 让所有 warp 都能看到，然后 tCtAcc.data() = shared_storage.tmem_base_ptr 让 fragment 指向它。
为什么 explicit 反而是优势
(1) RF 用量降到 ~10% — occupancy 可以拉到 4-8；(2) 累加器 ownership 与 warp 解绑 — Mainloop warp 写完即解放，Epilogue WG 用 tcgen05.ld 取出来；(3) 可以跨 phase 复用 同一块 TMEM 区域（FA4 把 S/P/O 累加器放在同一块 TMEM 上轮换），RF 模式做不到（切换需要昂贵的 spill）。

13.4.2 这张图的优化点：让 TMEM 学 SMEM 的"已知良好" pattern

TMEM 是新东西，本可以设计成新 API
NVIDIA 完全可以为 TMEM 发明新的 PTX 模型（比如 mmap-style、或自动 GC）。
但这样有学习成本
开发者需要重新理解一套新的 lifetime / ownership 规则。CUTLASS 团队（CuTe 作者）也要重新设计 atom 抽象。生态成本高。
NVIDIA 的选择：复用 SMEM allocator pattern
TMEM allocator 的 API 与 SMEM allocator几乎对称：alloc / dealloc / 必须 explicit、alloc 后 base_ptr 写 SMEM 让所有 warp 看到。差别只在granularity（column vs byte）和 access constraint（同 sub-partition 32 lane）。
开发者收益
会 SMEM allocator 就会 TMEM allocator。CuTe 作者只需扩展现有 atom 抽象，新增 TMEM_LOAD / TMEM_STORE atom 即可。生态学习成本几乎为 0 。这就是为什么 GTC 25 课程能在 30 分钟讲完 4 个 Feature——所有 API 都遵循已知的 pattern。

13.4.3 这张图的优化点：TMEM↔RF 拷贝为什么必须 4-warp 协作

TMEM 物理限制（见 §2.3）
每个 warp 只能访问同 sub-partition 的 32 lanes 。一个完整的 128×256 累加器横跨 4 个 sub-partition (lane 0-31 / 32-63 / 64-95 / 96-127)。
错误做法
如果只用 1 个 warp 调 tcgen05.ld，只能拿到 1/4 的累加器（仅该 warp 所属 sub-partition 的 32 row）。剩余 96 row 永远拿不到。这是 Blackwell 编程最常见的 bug。
CuTe 的 cooperative copy
把 4 个 warp 的 tcgen05.ld 自动编排：auto tiled_t2r = make_tmem_copy(SM100_TMEM_LOAD_32dp32b1x, tCtAcc); 创建 tiled copy；copy(tiled_t2r, tCtAcc, rC) 在 4-warp warpgroup 范围内自动拆分——每个 warp 取自己 sub-partition 的 32 row 写到本 warp 的 RF fragment。
实际优化
(1) 4-warp 并行 load → 一次 tcgen05.ld 把 128×8 行数据 4-way 并行拉到 RF（vs 顺序 4 次 = 4×延迟）；(2) cooperative pattern 与 SMEM cooperative copy 同构 → 学习成本零；(3) 内部用 wave-quantized layout（atom 名字里的 32dp32b1x = 32 dst-partition × 32-bit × 1 register）。

13.4.4 这张图的优化点：5 步流水让 mainloop 与 epilogue 完全解耦

Hopper epilogue 的耦合
wgmma 累加器在 RF 里 → 必须由同 warpgroup 处理 epilogue → mainloop warpgroup 必须存活到 epilogue 完成 → mainloop 不能自由发射下一轮 wgmma。结果：mainloop 与 epilogue 必须 ping-pong 错开两个 warpgroup，复杂度翻倍。
具体代价
Hopper 实现 (DeepGEMM/FA3) 必须用两个 wgmma warpgroup 交替 才能让 epilogue 与 mma overlap，需要写复杂的 phase 控制 + warp scheduler logic。
Blackwell 5 步
(1) Mainloop warp（leader CTA 单 thread）发 tcgen05.mma 写 TMEM；(2) umma_arrive(mma_mbar) 通知；(3) Epilogue WG（4 warp）等 mbar；(4) 用 tcgen05.ld 拷 TMEM 到 RF + 后处理 + TMA store；(5) tmem dealloc。Mainloop 和 Epilogue 是两组完全独立的 warp 。
三个具体收益
(1) 不需要 ping-pong - 1 个 mainloop WG + 1 个 epilogue WG 就够 ；(2) Mainloop warp 写完 TMEM 立即可以发下一轮 mma，不等 epilogue；(3) Epilogue 可以 fuse 任意复杂的链（bias + activation + quantize + requant + 多分支输出），不挤压 mainloop occupancy。FA4 的 ~17 个 warp 多角色调度本质就是把这 5 步推到极致 ——softmax 是一组 warp，correction 是一组，store 是一组。

13.4.5 这张图的优化点：用类型把 TMEM 包成"普通张量"

裸 TMEM 用起来很硌人
TMEM 地址是 32-bit word-addressed (lane 高 16 bit + col 低 16 bit)，不能 dereference。如果开发者直接处理这种地址，每条 ld/st 都要算 lane × 65536 + col 的偏移。
认知负担
Tensor Core 累加器是 2D 数据（128 row × 256 col），但 TMEM 地址是 1D word，layout 不直观。开发者要手算 lane→row、col→col 的映射。
CuTe 抽象
Tensor tCtAcc = cta_mma.make_fragment_C(tCgC); 一行让 CuTe 在 TMEM 上创建一个带 layout 的张量 。tCtAcc(i, j) 自动转换成 TMEM 地址 (i << 16) | j。后续把 tCtAcc 当普通 CuTe Tensor 用，传给 cute::gemm / cute::copy 都行。
开发者收益
(1) 不必接触 TMEM 32-bit 地址编码；(2) 不必关心 sub-partition 边界（CuTe 在 cooperative_copy 时自动 4-warp 拆分）；(3) 切换 1-CTA / 2-CTA atom 时 fragment 类型自动调整。整个 Layer 2-4 描述的"硬件复杂度"在 CuTe 层面被压成 1 行代码 。
✅ 4 个 Feature 累计的优化收益账
Feature不上的代价上之后的收益实测速比
1 MMA atom裸 PTX，~30% 峰值CuTe 抽象，descriptor 自动1× (baseline)
+2 TMAcooperative copy 占 SM 算力硬件 DMA，0 SM 消耗2.3×
+3 2-CTA UMMAB 加载两份，AI 减半multicast 共享 B3.4×
+4 TMEMRF 累加器吃 50% occupancyoccupancy 翻倍，epilogue 解耦5.6× (96% 峰值)

把 4 个 Feature 全开后一个完整 BF16 GEMM mainloop 大约 40 行 CuTe DSL（vs 等价 Hopper kernel ~120 行 + 复杂的 wgmma fence/commit/wait 嵌套）。硬件越复杂，CuTe-DSL 的相对优势越大 。

13.6 SMEM descriptor 字段编码 + swizzle 模式选择 · 隐藏的性能黑洞

§13.1 一笔带过“CuTe atom 屏蔽 PTX 复杂度”，但SMEM descriptor 究竟编码了什么、swizzle 模式选错会有什么后果 没展开。这是 NVIDIA 论坛 + flashinfer 仓库 issue 中 frequency 仅次于 TMEM 的提问点——错一位就静默算错或 throughput 损失 30%。

13.6.1 64-bit SMEM descriptor 的字段布局

tcgen05.mma 的 A / B operand 都不是直接传 SMEM 指针，而是传一个 64-bit packed descriptor 。它的字段（社区博客 + flashinfer issue 整理）：

bit  63 .................................................. 0
     [ swizzle | base | stride_byte_off | leading_byte_off | smem_addr ]
     [  2 bit  | 3 bit |     14 bit       |      14 bit      |  14 bit  ]
     ↑ bit 53-52        ↑ bit 45-32         ↑ bit 29-16        ↑ bit 13-0
              ↑ bit 51-49

字段含义:
- smem_addr (14-bit):    SMEM 地址 / 16, 即 14-bit 编码 16-byte 对齐地址
- leading_byte_off:      leading dimension byte offset / 16
- stride_byte_off:       stride dimension byte offset / 16
- base (3-bit):          base offset for matrix coordinate
- swizzle (2-bit):       0=NONE, 1=128B, 2=64B, 3=32B
裸手填 descriptor 的痛苦
FlashInfer issue #1078 就是问怎么手填这 64 bit。bit 偏移、字段宽度、是否右移…每一处都容易错，错任何一位 → MMA 静默算错（不报错）。
具体的两种典型 bug
(1) 忘了 smem_addr 要除以 16 ：直接把 SMEM 地址塞进 bit 0–13 → MMA 读到错误位置，结果是垃圾值；(2) swizzle 字段填错 ：本来 SMEM 是 128B swizzle 排的，但 descriptor 写 swizzle=NONE → MMA 按非 swizzle 读 → 数值完全错乱。
正确的 CuTe 用法
UMMA::smem_desc<a_major> traits 类自动生成 descriptor。开发者只需把 SMEM tensor 传给 cta_mma.make_fragment_A(tCsA)，CuTe 内部根据 SMEM layout 推断 swizzle、stride、leading dim，0 行 bit 操作 。
实战教训
除非你在写底层 inline asm 或者复刻 CUTLASS 内部，永远不要手填 64-bit descriptor 。CuTe atom 的存在就是为了消化这个复杂度。

13.6.2 SMEM swizzle 4 种模式 · 选错就是性能黑洞

SMEM swizzle 是为了避免 bank conflict 而设计的逻辑地址 → 物理 bank 的扰乱映射 。Blackwell tcgen05.mma 支持 4 种 swizzle，不同 swizzle 对应不同的最小连续 tile 几何 ：

Swizzle 模式最小连续 tile典型用途对 throughput 的影响
SWIZZLE_NONE每 8×16B tile 必须 SMEM 连续小 K（K=16）GEMM大 K 时 throughput 损失 30%+（bank conflict 拉满）
SWIZZLE_32B每 8×32B tile 连续，内部 16-byte 单位再 swizzleK=32 的 FP16 GEMMbaseline
SWIZZLE_64B每 8×64B tile 连续，内部 swizzleK=64 的 FP16 / K=32 的 BF16baseline
SWIZZLE_128B每 8×128B tile 连续，最大 swizzle 颗粒大 K（K=128+） GEMM、attention大 K 时最优
新手常踩
(1) 跨架构复用 SMEM layout 但忘改 swizzle——例如把 Hopper 的 64B swizzle layout 直接搬到 Blackwell 想跑 FP4，结果 bank conflict 卡到飞起；(2) 选 SWIZZLE_NONE 想“省事”——但 K>16 时 SMEM bank conflict 让 tcgen05.mma throughput 砍 30%+。
为什么影响这么大
Tensor Core spatial array 一次 cycle 要从 SMEM 读整个 tile 的所有 lane（128 lane × 16 byte = 2 KB）。如果 swizzle 错位，多个 lane 撞到同 bank → 串行化 → throughput 砍 N（N = 撞同 bank 的 lane 数）。
怎么选 swizzle
规则：swizzle 字节数 = tile 的 contiguous K 维字节数 。例如 FP16 K=64 → 64×2 = 128 byte → 选 SWIZZLE_128B；FP8 K=32 → 32 byte → 选 SWIZZLE_32B。CuTe 提供 Layout_K_SW128_Atom<TypeA> 这类助手，按 dtype + K 自动配。
实测教训
NVIDIA 论坛上有人 benchmark 4 种 swizzle 跑 fp8 SS-MMA，结果显示 SWIZZLE_128B 比 SWIZZLE_NONE 快 25–35%（取决于 K 维大小）。这就是“选错 swizzle 的代价”。

13.6.3 SMEM tile contiguity 与 swizzle 的对应规则

这是 gau-nernst 博客整理出来的最有用的 cheatsheet：

SwizzleSMEM tile 必须满足违反会怎样
NONE每 8 row × 16 byte 块在 SMEM 连续编译过、运行错值
32B每 8 row × 32 byte 块连续，块内 16-byte 子块再 swizzle同上
64B每 8 row × 64 byte 块连续同上
128B每 8 row × 128 byte 块连续同上

关键 trap：SMEM 物理 layout 必须与 descriptor 中 swizzle 字段一致 。如果你手摆 SMEM layout 是 128B swizzle 但 descriptor 写 NONE → tcgen05.mma 按 NONE 解释地址 → 拿到的 16-byte 块顺序错乱 → 矩阵乘的累加结果完全错。

✅ 用 CuTe 避免所有这些坑

SMEM layout + swizzle + descriptor 三者必须强一致。CuTe 的 atom 系统在编译期就保证三者一致 ——你只需选好 atom（如 SM100_MMA_F16BF16_SS）和 SMEM layout helper（如 tile_to_shape(SW128, ...)），CuTe 自动配 descriptor swizzle 字段。手写 PTX 时三者要靠开发者自己对齐，错任何一处都静默错值 。这是“不要写裸 PTX”原则的最强论据之一。

14.1 PTX 8.7 → 9.2 演进与 tcgen05.mma instruction descriptor

Layer 3 提到 tcgen05 是 PTX 8.7 (CUDA 12.8) 引入的指令族。一年后的 PTX 9.2 又做了几次扩展，理解这条演进可以判断"现在写的 kernel 在哪个 toolkit 才能编"。

PTX 8.7 (CUDA 12.8, 2025-01)
初代 tcgen05 PTX：基本 tcgen05.mma + .kind::{f16,tf32,f8f6f4} + .cta_group::{1,2} + 基本 alloc/dealloc/cp/ld/st。FP4 / FP6 已通过 .kind::f8f6f4 支持，但只能用 1× block scaling。
缺什么
(1) 没有 NVFP4 (UE4M3 scale) — 只能用 OCP MX 风格的 UE8M0；(2) scale_vec 只有 1X、2X — 不能做 block-16 这种细粒度；(3) sparsity 只有基础支持。
PTX 9.2 (CUDA 13.1) 新增
(1) .kind::mxf4nvf4 同时兼容 OCP MXFP4 与 NVIDIA NVFP4 的 atom；(2) .scale_vec::4X qualifier 让 scale 粒度从 32 elem 一段降到 8 elem 一段（atom_K=64 时）；(3) sparsity scaling 扩展。
实战意味
用 NVFP4 推理 LLM 必须 CUDA 13.1+ 。CUDA 12.8 上写的 kernel 只能拿 MXFP4，perplexity 损失会比 NVFP4 高 ~5 个百分点（见 Layer 9.4 实测表）。

关于 instruction descriptor 的具体编码：这是一个 32-bit 整数 ，运行时 构造（不是 PTX 编译期常量），包含：

  • a_format / b_format ：integer 编码 ABI 数据类型（FP4=0, FP6_E2M3=1, FP6_E3M2=2, FP8_E4M3=3, FP8_E5M2=4 等）；
  • a_neg / b_neg ：1-bit 取负 flag；
  • a_major / b_major ：transpose 信息；
  • sparsity ：稀疏 metadata；
  • scale_vec ：scale 粒度选择。

这就是为什么 GTC 25 课程反复强调 "instruction descriptor is a runtime argument" — 这让同一份 kernel 二进制可以处理多种 mixed-input 组合 ，descriptor 在 host 端配好就行，不需要为每种 (a_dtype, b_dtype) 重编一次。

14.2 NVFP4 vs MXFP4 · NVIDIA 官方博客深度对照

Layer 9 简要说了两者差异，NVIDIA 在 2025 年发的官方博客 Introducing NVFP4 for Efficient and Accurate Low-Precision Inference 给出了更细的设计动机。

Block size32 elem16 elem更细粒度 → 局部动态范围适应力↑ 2×
Scale typeUE8M0（仅 2 的幂）UE4M3 (1 sign + 4 exp + 3 mantissa)scale 表达精度↑ → 量化误差↓
Scale 最大值2^127448 (E4M3 max)NVFP4 元素最大 = 6 × 448 = 2688

NVIDIA 官方实测—vs FP16 内存 -3.5×、vs FP8 -1.8×、accuracy 退化 < 1%—
✅ NVIDIA 选 UE4M3 scale 的设计理由

UE8M0 (MXFP4) 的 scale 只能是 2 的幂——比如 1, 2, 4, 8, 16，缺少 1.5、3、5 这样的中间值 。当一个 32-elem block 内最大值是 5.7 时，UE8M0 必须选 8（向上取）或 4（向下取），都引入大误差。UE4M3 能精准表达 5.75（最接近 5.7 的可表达值），rounding error 减半 。这就是 NVIDIA 实测 perplexity 退化 +4% (NVFP4) vs +9% (MXFP4) 的根因。

⚠️ 跨厂商部署注意： NVFP4 的 .safetensors 文件无法直接在 AMD MI300 / Intel Gaudi 上加载 ——它们只支持 OCP MXFP4。如果模型要 portable 部署，必须 quantize 两份（NVFP4 + MXFP4），或选 MXFP4 接受精度损失。

14.3 DeepGEMM Blackwell SM100 · DeepSeek 实战实现

DeepGEMM 是 DeepSeek 2025 开源的 FP8/FP4 GEMM kernel 库，是 Blackwell 上最被广泛部署的 LLM kernel 实现 。其 Blackwell SM100 路径设计与本文 Layer 8 描述高度对齐，但有几个工程细节值得单独说。

SM90 (Hopper) 上的 DeepGEMM
(1) 仅支持 NT memory layout (row-major × col-major)；(2) scale factors 必须 FP32；(3) 仅 normal GEMM + grouped GEMM (M-axis 分组)。
面对 Blackwell 怎么改
(1) NT-only 限制让某些 attention 路径需要额外 transpose；(2) FP32 scale 占内存 4× 于 UE8M0；(3) MoE expert mixed precision (FP4 weight + FP8 activation) 没法表达。
SM100 (Blackwell) 路径新增
(1) 所有 4 种 layout 都支持 ：NT, TN, NN, TT；(2) scale 用 packed UE8M0 ：4 个 UE8M0 (8-bit each) 打包到一个 int32，节省 4× 内存；(3) 新增 FP8×FP4 mixed precision GEMM ；(4) Mega MoE with overlapped communication；(5) JIT 编译，安装时 0 CUDA 编译。
给框架开发者的影响
vLLM / SGLang / TensorRT-LLM 等可以不重写 attention 层 就支持 Blackwell——layout 灵活性补上了。Mega MoE 让 256-expert MoE inference 在 NVL72 域内完成 expert all-to-all 通信，省下显式 NCCL 调用。
💡 分组 GEMM 的"DeepGEMM 风格"： 不像 CUTLASS 的传统 grouped GEMM 那样 M/N/K 都可分组，DeepGEMM 仅沿 M 分组、N 和 K 必须固定 。这正好匹配 MoE 中"所有 expert 共享 weight shape (K×N)、每个 expert 处理不同 token 数 (M)"的场景。看起来限制更多，但实际是 MoE 推理的常态——失去的灵活性换来 30%+ 的 throughput 提升。

14.4 FA4 polynomial approximation · Cody-Waite + Sollya 数学细节

Layer 11.2.1 提到 FA4 用三阶 Horner 多项式近似 2^x 的方法，但没说怎么挑系数 。Tri Dao 与 Colfax 的官方博客补了关键数学：

📐 三步走数学结构
  1. Range reduction（Cody-Waite 风格） ：把 2^x = 2^n · 2^f 拆成整数部分 n（直接 bit-shift exponent field）和小数部分 f ∈ [0, 1)。
  2. Polynomial approximation ：用三阶 Horner 形式 2^f ≈ p₀ + f(p₁ + f(p₂ + f·p₃))。3 次乘 + 3 次加，全程走 FMA。
  3. 系数选取 ：用 Sollya 数值优化软件 挑 (p₀, p₁, p₂, p₃)，目标函数是 minimize relative approximation error 。Sollya 是 INRIA 出的专业工具，能在给定多项式阶数下找最优 minimax 近似。
不用 Sollya 自己挑系数
常见做法：用 Taylor 展开（2^f ≈ 1 + f·ln2 + f²·ln²2/2 + f³·ln³2/6）。简单，但 minimax 误差不优。
Taylor 的代价
三阶 Taylor 在 [0, 1] 区间最大 relative error ~3×10⁻³，给 softmax 带来明显数值偏移。LLM 推理 ROUGE/BLEU 退化几个百分点。
Sollya minimax
三阶 Sollya 多项式在同一区间最大 relative error ~1×10⁻⁵，差 300 倍 。多项式形式不变（仍 3 次乘 + 3 次加），系数差别让精度跨数量级。
实战收益
FA4 在 BF16 forward 实测 1605 TFLOPS / 71% utilization，且数值精度与硬件 EX2 不可区分（因为 BF16 mantissa 仅 7 bit，~10⁻⁵ relative error 远低于量化噪声）。
💡 实战可以借鉴： 任何用 SFU 算 exp/log/sin 的 kernel，如果 SFU 是瓶颈，都可以套这个"Cody-Waite + Sollya minimax 三阶多项式"模板把工作分给 FMA。Triton 也能写。

14.5 TMEM per-warp access 限制 · 硬件成本视角

Layer 2.3 说 TMEM "每个 warp 只能访问同 sub-partition 的 32 lanes"，但没解释为什么 NVIDIA 这样设计 。NVIDIA Developer Forums 上有官方工程师的回答（CUDA Programming and Performance 板块）：

❌ 假如 TMEM 让每个 warp 都能访问全部 lanes

那硬件需要的是 4× 更多的 access port ——任意 warp 在任意时刻可以读写任意 lane 的 cell。Port 数量直接乘以 SRAM 面积：每个 port 需要独立的 wordline / bitline / sense amplifier。256 KB TMEM 估计要从单 port 设计的 ~3 mm² 涨到 4-port 的 ~10 mm²。

✅ NVIDIA 的妥协：4 sub-partition × 1-port each

把 TMEM 切成 4 个独立 sub-partition，每个 sub-partition 只有 1 个 access port，由对应 warp 独占 。总面积保持单 port 设计的 ~3 mm²，但需要 4 个 warp（一个 warpgroup）才能 cover 全部 lanes。

这是经典的"硬件复杂度换软件约束" 取舍：硬件省了 7 mm² 面积、几瓦功耗；软件多写一行 cooperative copy 把 4 个 warp 的 tcgen05.ld 串起来——CuTe 的 make_tmem_copy + cooperative copy pattern 把这个复杂度也封装掉了。

💡 同样的设计哲学还出现在 ：(1) Hopper SMEM 的 32 banks（避免 32-port 设计）；(2) Distributed SMEM 仅在 cluster 内可见（避免全局 SMEM crossbar）；(3) Tensor Core 单条指令处理 spatial array（避免 MAC unit 全连接）。NVIDIA 的硬件设计逻辑是"用 access pattern 限制换 area 与 power 节约" 。

14.6 Cluster Launch Control (CLC) · 跨 CTA 一致 query 机制

Layer 6 提到 Dynamic Persistent Scheduler 用 CLC 实现，但没说 CLC 在 PTX / 硬件层面具体怎么工作。NVIDIA 官方 CUTLASS 文档给出关键机制：

传统 work-queue 实现
用 SMEM 全局 atomic counter：每个 CTA 完成后 atomic_add(counter, 1) 取下一个 tile。问题：counter 是全局热点，atomic 串行化。
具体瓶颈
148 SM × 数十次 atomic / kernel 执行 = 数千次全局 atomic。每次 atomic 需要跨 GPC 一致性同步，单次 ~50 cyc。CLC 之前 Hopper 上动态调度有 5-10% throughput 损失。
CLC 的硬件机制
(1) CLC query ：CTA 完成 tile 后发起一条 CLC PTX query；(2) broadcast response ：CLC engine 把下一个 tile 的 ID 写到整个 cluster 内所有 CTA 的同一个 SMEM 地址 ；(3) distributed counter ：CLC 内部维护多个 sub-counters 分散热点；(4) cluster-scoped 一致性 ：cluster 内 CTA 共享 SMEM，response 一次广播到所有 CTA 不需要额外 atomic。
实测优势
(1) Atomic 开销从 ~50 cyc 降到 ~5 cyc（CLC engine 内部并行）；(2) 不再串行化：多个 CLC query 可同时处理；(3) 实测动态调度损失从 5-10% 降到 < 1%。这是为什么 CUTLASS 4.x 在 Blackwell 上敢把 dynamic scheduler 设为默认开启 。

CLC 是 Blackwell 第一次把"work scheduler" 做成专用硬件 unit。之前所有 CUDA 调度都是"软件 work queue + 全局 atomic"模式，现在 NVIDIA 把它升级到了"专用引擎 + cluster-scoped broadcast"。这与 Decompression Engine 类似——把曾经吃 SM 算力的事搬到专用硬件。

# 2. 启动 cuda-gdb，设置 hang 后中断
cuda-gdb ./my_kernel
(cuda-gdb) set cuda break_on_launch application
(cuda-gdb) run
# 当 kernel hang 时按 Ctrl-C

# 3. 列出所有 warp 状态，找 stalled 的
(cuda-gdb) info cuda warps
# 输出列出每个 warp 的 PC / state，blocked 的 warp 标 BAR/MEMBAR/etc

# 4. 切到 stalled warp 看它在等什么
(cuda-gdb) cuda warp 5 lane 0
(cuda-gdb) disas $pc-32, $pc+8
# 看到例如  BAR.SYNC.DEFER_BLOCKING  → 死在 cluster barrier
#       或  NANOSLEEP                → 死在 mbar wait

# 5. 看 mbar 的 phase 字段
(cuda-gdb) print *(uint64_t*)0x10  # SMEM 中 mbar 地址
# arrive_cnt / tx_cnt / phase 三个字段
# 如果 expected_arrive_cnt 不等于 actual → 哪个 producer 漏 arrive
# 如果 phase 没翻 → consumer 等的是上一个 phase
常见 cuda-gdb 看到的死锁模式
(1) BAR.SYNC.DEFER_BLOCKING ：cluster barrier 卡住，一定是某个 CTA 漏 arrive；(2) NANOSLEEP ：mbar wait 卡住，要么 producer 漏 commit、要么 phase 用错；(3) S2R + 大量 stall ：warp 在等 SMEM，但 SMEM 写入方还没完成（TMA / cooperative copy 没发起或没 wait）；(4) elect_one_sync 卡住 ：单 thread 进入了应整 warp 进入的函数（陷阱 ③）。

A.3.2 TMEM 内容查看的特殊步骤

TMEM 不是普通 memory——cuda-gdb 不能 print *(float*)tmem_addr。要看 TMEM 内容必须：

  1. 在 TMEM 写入完成后插入一段调试 kernel；
  2. 这段 kernel 用 tcgen05.ld 把 TMEM 内容加载到 RF；
  3. 再 st.global 写到一块 GMEM 调试 buffer；
  4. cuda-gdb 在 host 端 print 这块 GMEM。

这个限制在 §2.3 已经埋了伏笔——TMEM 是 word-addressed 非线性寻址、不能 dereference 。所以调试体验比 SMEM/RF 差很多，第一次写 Blackwell kernel 强烈建议先在 RF 上把算法跑通，再迁到 TMEM 。

A.3.3 nsight-compute 看 5th-gen TC 利用率

判断 kernel 是不是真在用 Blackwell 路径（不是 Hopper 兼容路径），nsight-compute 上看：

ncu --section=tensor --section=memory_workload     --metrics sm__inst_executed_pipe_tensor_op_hmma_cycles_active.pct_of_peak_sustained_active     --metrics sm__inst_executed_pipe_tensor_op_qmma_cycles_active.pct_of_peak_sustained_active     --metrics sm__inst_executed_pipe_tensor_op_omma_cycles_active.pct_of_peak_sustained_active     ./my_kernel

三个 metric 分别对应 HMMA（FP16/BF16/TF32）、QMMA（FP8/FP6）、OMMA（FP4）的活跃度。正常的 Blackwell GEMM 应该看到对应精度的 metric > 90% ；如果出现 HGMMA / QGMMA / IGMMA 相关 metric → 走的是 Hopper 兼容路径，没拿到 5th-gen 收益（参考 §3.4 wgmma deprecated 警告 + §3.7 陷阱 ⑤）。

A.4 PTX-level microbench 方法论（论文怎么测出来的）

本文中所有的"实测"数字，背后都是 PTX-level microbenchmark 。理解方法论才能理解数字的真正含义。

A.4.1 时序测量：%clock64

NVIDIA GPU 提供 %clock64 寄存器，硬件维护的纳秒级 cycle counter。Jarmusch 等的标准做法：

.reg .u64 %start, %end;
ld.param.u64 %time_ptr, [param_time];
mov.u64  %start, %clock64;     // 取起始 cycle
mad.lo.s32 r1, r1, r2, r3;     // 被测指令
mov.u64  %end,   %clock64;     // 取结束 cycle
// (%end − %start) − 自身 mov 开销 = 真实 cycle 数

注意一个细节：在 GB203 上，两条 mov.u64 %clock64 之间不插任何指令时差值是 1；在 GH100 上是 2。这意味着论文报告 cycle 数时已经做了平台相关的扣减 。

A.4.2 真延迟 (True Latency) vs 完成延迟 (Completion Latency)

指标定义测法含义
True Latency单条依赖指令出结果到下一条能用之间的 cycle串行 dep chain，N 大、cycle 取均值反映 pipeline 深度
Completion LatencyN 条独立 指令总 cycle ÷ N并行 issue 多个独立指令反映 throughput / 发射宽度

这两件事拆开测才能区分"流水深"和"发射宽"两个维度。Layer 2.2 中"统一 INT/FP 让混合真延迟砍半"就是True Latency 视角；Layer 8.3 中"持续 Throughput"是Completion Latency 视角。

A.4.3 ILP 扫描 + warp count 扫描

论文同时扫两个维度：每个 thread 的 instruction-level parallelism (ILP = 1, 2, 4, 8…) 和活跃 warp 数（1, 2, 4, …, 32）。结果：

  • Hopper ：在 ILP=5、29 active warps 时跑满 throughput；
  • Blackwell ：在 ILP=6、25 active warps 时跑满 throughput；
  • 含义：Blackwell 的 warp scheduler 对 intra-warp ILP 利用率更高 ，能在更少 warp 时拿到峰值——意味着 occupancy-constrained 的 kernel（如 attention 长 seq）受益最大。

A.4.4 嵌入式 PTX vs inline PTX 的"陷阱"

Jarmusch 等明确说明：他们用独立 PTX 文件 + ptxas 编译为 cubin ，而不是 CUDA C++ 中的 inline asm——目的是防止 nvcc 优化器对被测指令做指令重排或常量折叠 。每条 microbench 的生成的 SASS 都通过 cuobjdump 验证过，确保实际执行的就是声明的指令。

⚠️ 读论文时常见陷阱： 如果一篇 paper 报告 "Tensor Core 实测 50% 峰值"，先看它是用 cuBLAS 还是 PTX microbench——前者反映的是软件成熟度 ，后者才是硬件能力上限。两个数字差 20% 都不奇怪。
