方法与数据清洗

测量方法只做一件事：8 卡 EP=8，每 rank N tokens，跑一次 forward 的 wall-clock。 每个 rank 在 cuda.synchronize() 前后用 perf_counter 计时，5 轮 warmup 后取 20 轮均值，dist.barrier() 确保同步，TRT-LLM 走 NVLinkOneSided alltoall，不是 raw kernel。

精度对齐是这次对比的核心前提。DeepGEMM 走的是 m_grouped_fp8_fp4_gemm（W=NVFP4 + A=FP8）；与它精度对等的是 TRT-LLM 的 W4A8MXFP4-FP8，而不是 NVFP4（A=NVFP4，激活精度更低）。 早期版本拿不同精度的两者直接比较，结论不可信。本文在引用跨精度数据时会明确标注。

③ kernel breakdown — 4.7 ms 差距来自哪里？

4.7 ms 的差距在 wall-clock 层面已经可见，但原因不明显。用 nsys 在 rank 0 上采集 25 次 forward，把「每 forward 摊到的 GPU-busy 时间」（avg_us × calls / 25）逐 kernel 列出来，就能看清每条路径在 GPU 上实际消耗了多少算力。

DeepSeek-V4-Pro · BS=8192 · EP=8 · nsys rank 0 · cutlass GEMM 每 forward 调 2 次（gate+up + down），其余 1 次

CAVEAT · 这张图展示的是每个 kernel 各自占用 GPU 的时间。TRT-LLM 走 multi-stream，dispatch / combine / AllReduce 跟 GEMM 在不同 stream 上能 overlap，所以柱子总长 ≥ 上面 wall-clock 表里的实测值。柱长可以看「这条 path 在 GPU 上一共干了多少活」，但不是「forward 实际耗时」——后者以 ① / ② 的 wall-clock 表为准。

FINDING · W4A8 慢，根因是 A2A dispatch 5403 us（NVFP4 才 251 us，21× 差距）——TRT-LLM 1.3.0rc9 的 W4A8+EP+alltoall 没走 NVFP4 那条 fast path。即便靠 multi-stream overlap 把 5.1 ms 藏掉，wall-clock 还是 7.74 ms。DeepGEMM 一发 fused kernel 3.08 ms，不需要靠 overlap 救场。

结论

  • 同精度对比（W=NVFP4 A=FP8），DeepGEMM 领先 2.5×——DSV4 BS=8192：3.07 ms vs 7.74 ms；Qwen3.5 因 expert 规模较小，绝对差距收窄，倍数降至 1.54×。
  • 即便 TRT-LLM 用更激进的 A=NVFP4，DeepGEMM 仍快 2.03×，说明差距不来自精度选择。
  • 根因定位：TRT-LLM W4A8+EP+alltoall 的 dispatch kernel 耗时 5403 us，而 NVFP4 路径仅 251 us，差了 21×。1.3.0rc9 的这条路径存在明显未优化。
  • DeepGEMM 的工程路线：将 dispatch + L1 GEMM + SwiGLU + L2 GEMM + combine 融入一个 CUDA kernel，利用 NVLink 对称内存省去 host-side 协调；TRT-LLM 拆成 8 个 kernel 靠 multi-stream overlap 部分掩盖通信开销，overlap 有效时可以追近，但掩盖失效时差距就暴露出来。
