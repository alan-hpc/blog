Shared Memory Swizzle 正确性深析：从一行 XOR 到 Hopper / Blackwell 的硬件 descriptor

Swizzle 改变了元素在 shared memory 的物理地址，却不改变逻辑矩阵——因为写入端与读取端引用同一个地址函数。本文用一个带真实数值的 8×8 例子，把这条不变量从一行 XOR 一直追到 CUTLASS 的 GMMA / UMMA descriptor。

Overview · 一个看似矛盾的现象

在 Tensor Core 数据通路里，shared memory 几乎总要做 swizzle。随之而来一个让人困惑的现象：swizzle 改变了元素在 shared memory 中的物理存放位置，ldmatrix（或 wgmma）又按这个被改变的布局去访问，两处看起来都「错位」了，但 Tensor Core 最终算出的 $C = A \cdot B$ 完全正确。

这并非巧合，而是一条精心设计的不变量：只要写入与读取使用同一个地址映射，逻辑元素的身份就不会丢失。 swizzle 换的是物理座位，不是逻辑身份；而换座位这件事在写入和读取两端同步发生。

本文不停留在比喻层面，而是落到可验证的实现：先用一行 addr ^= ... 给出 swizzle 的本体，再用真实数值跑通一次 store / load 往返，然后进入 ldmatrix 的具体语义，最后到 CUTLASS 的 SM80 GEMM 主循环、Hopper GMMA descriptor 与 Blackwell UMMA descriptor，逐处确认这条不变量如何被代码强制保证。

实验设置 · 一个可手算的 8×8

为了让每一步都能手算核对，全文使用同一个最小例子。

矩阵      A，8×8，元素值约定 A[r][c] = 10·r + c     （行号×10 + 列号，便于辨认身份）
存储      bf16，每元素 2 字节，行主序，行跨度 8 元素 = 16 字节

按列取数正是 ldmatrix / wgmma 取一整列操作数的形态，也是 bank conflict 的高发场景。逻辑真值表（后续所有正确性以它为准）：

r0        0   1   2   3   4   5   6   7
r1       10  11  12  13  14  15  16  17
r2       20  21  22  23  24  25  26  27
r3       30  31  32  33  34  35  36  37
r4       40  41  42  43  44  45  46  47

1. Swizzle 是什么 · 作用在 shared 地址上的一次 XOR

本节回答 Q1。在 Ampere / Ada 上，swizzle 不依赖任何专用硬件：它就是计算 shared memory 地址时多写的一行 addr ^= ...。写入写这一行，读取也写同一行。

元素 $(r,c)$ 的朴素字节地址是 $\text{off} = 16r + 2c$，只占 7 个 bit：

swizzle 函数本体即两个地址函数之差：

using bf16 = __nv_bfloat16;
constexpr int W = 8;                       // 每行 8 个元素

// 朴素字节地址:(r,c) -> r*16 + c*2
__device__ inline uint32_t addr_plain(int r, int c) {
    return (r * W + c) * sizeof(bf16);     // = r*16 + c*2
}

// swizzle 字节地址:仅多一行 XOR
__device__ inline uint32_t addr_swz(int r, int c) {
    uint32_t off = (r * W + c) * sizeof(bf16);
    off ^= ((off >> 4) & 0x7) << 1;        // <-- swizzle 全部内容
    return off;                            // (off>>4)&7 = r;  <<1 对齐到 c 的位段
}

逐位验证 $(r,c)=(1,3)$，即值 $13$ 该落到何处：

朴素   off = 1*16 + 3*2 = 22 = 0b001 011 0     (行=001, 列=011, 字节=0)
取行   (22 >> 4) & 7 = 1 ;  对齐 1 << 1 = 2 = 0b000 001 0
XOR    22 ^ 2 = 0b001 011 0 ^ 0b000 001 0 = 0b001 001 0 = 20
结果   字节 20 = 元素索引 10 = p2  (列位从 011=3 变为 001=2，正是 3⊕1)

swizzle 的可逆性来自 XOR 自身

$x \mapsto x \oplus f(x)$ 在每个 swizzle 周期内是一个置换：没有两个逻辑地址映到同一物理地址，也没有物理地址无人认领。元素既不丢失、不重叠，也不会凭空多出，只是更换物理位置。这是后文「往返闭合」与「FP8 宽度约束」共同的前提。

2. store 与 load 如何实现 · 同一个地址函数的往返

本节回答 Q2。「数值为什么不变」的答案不在于两种方案碰巧得到同值，而在于每种方案内部，读地址等于写地址——写入位置 X、读取位置 X，取回的必然是写入的那个值。

extern __shared__ bf16 smem[];            // 8*8 = 64 个 bf16

// 写入 A[r][c]
bf16 v = A_global[r * W + c];

// 无 swizzle:写到 r*16 + c*2
st_shared(smem, addr_plain(r, c), v);
// 有 swizzle:写到 XOR 后的地址
st_shared(smem, addr_swz(r, c), v);

// 读回 A[r][c]——读取调用同一个地址函数
bf16 a = ld_shared(smem, addr_plain(r, c));   // 无 swizzle
bf16 b = ld_shared(smem, addr_swz(r, c));     // 有 swizzle,用同一个 addr_swz

写入与读取都调用同一个 addr_swz。把 $13$ 写入 addr_swz(1,3)=20，再从 addr_swz(1,3)=20 读取，取回的必然是 $13$。

形式化地，记写入为 $\text{smem}[\sigma(r,c)] = A[r][c]$，读取为 $\text{reg} = \text{smem}[\sigma(r,c)]$。两端的 $\sigma$ 是同一个函数，于是

$$\text{reg} = \text{smem}[\sigma(r,c)] = A[r][c]$$

与 $\sigma$ 的具体形式、与它是否「错位」无关。结果恒等于无 swizzle 时的真值。

切换有 / 无 swizzle，等价于把 `addr_swz` 替换为 `addr_plain`，kernel 其余部分一字不改。这就是 swizzle 在 Ampere 路径上的完整实现——一个作用于 shared 地址的纯函数，写读两端共用。

3. 数值守恒、冲突消失 · 带真实 bank 号的对照

本节回答 Q3。两种地址函数存入后的物理布局不同，但存入的真值集合完全相同，区别只是摆放顺序。下表中每格写的是该物理槽实际存放的逻辑列对应的值；颜色无关，看 bank 列即可。

无 swizzle，物理槽 = c，bank = c（每行摆法一致）：

行＼槽  p0    p1    p2    p3    p4    p5    p6    p7
r0       0     1     2     3     4     5     6     7
r1      10    11    12    13    14    15    16    17
r2      20    21    22    23    24    25    26    27
...     每一竖列同 bank —— 同一逻辑列的 8 个元素全部落在同一个 bank

有 swizzle，物理槽 = c⊕r，bank = c⊕r（每行额外旋转 r 位）：

行＼槽  p0    p1    p2    p3    p4    p5    p6    p7
r0       0     1     2     3     4     5     6     7      (⊕0)
r1      11    10    13    12    15    14    17    16      (⊕1)
r2      22    23    20    21    26    27    24    25      (⊕2)
r3      33    32    31    30    37    36    35    34      (⊕3)
...     每一竖列覆盖全部 8 个 bank —— 同一逻辑列被摊到 8 个不同 bank

注意 r1 行：值 $10,11,12,\dots$ 全部仍在，只是 $10\leftrightarrow11$、$12\leftrightarrow13$ 互换了位置（$\oplus 1$ 的效果）。没有任何数值改变。

现在两种布局都来取列 c3，期望值 $3,13,23,33,43,53,63,73$：

                无 swizzle (槽=3)          有 swizzle (槽=3⊕r)
线程 r  期望值   读物理槽  bank            读物理槽  bank
0        3        p3        3               p3        3
1       13        p3        3               p2        2
2       23        p3        3               p1        1
3       33        p3        3               p0        0
4       43        p3        3               p7        7
5       53        p3        3               p6        6
6       63        p3        3               p5        5
7       73        p3        3               p4        4
冲突              8 路冲突(全撞 bank 3)      0 冲突(8 个 bank 各一)

两列「期望值」完全相同——这就是送进 Tensor Core 的寄存器内容。唯一不同的是「bank」列：无 swizzle 全部撞向 bank 3，硬件串行化、吞吐降到 1/8；有 swizzle 摊满 8 个 bank，单周期取完。

「物理错位」与「逻辑错位」是两件事

swizzle 路径下，线程 1 实际访问的是物理槽 p2 而非直觉的 p3——物理访问位置确实错位，且这是有意为之（为分散 bank）。但读取公式 $3\oplus r$ 精确抵消了写入时同样的偏移，逻辑数值不错位。「访问错位」不等于「读到错误数据」。

4. ldmatrix 的具体配合 · 交地址 + 固定发牌

本节回答 Q4。前面的「读一列」是简化模型；ldmatrix.sync.aligned.m8n8.x1 的真实语义分两步。

第一步（交地址）：warp 的前 8 个线程（t = 0..7）各交出一行的起始地址——线程 t 交第 t 行地址。一行 8 个 bf16 = 16 字节，是一次访问的整块。

第二步（发牌）：硬件把读到的 8×8 = 64 个值，按写死的 lane → 寄存器映射分给 32 个线程，每人 2 个（一个 32-bit 寄存器）。映射规则为 $t \to$ 第 $\lfloor t/4 \rfloor$ 行、第 $(t\bmod 4)\cdot 2$ 与 $+1$ 两列：

线程 t   行 r=t/4   列      寄存器拿到的值
0        0          0,1     0, 1
1        0          2,3     2, 3
2        0          4,5     4, 5
3        0          6,7     6, 7
4        1          0,1     10, 11
5        1          2,3     12, 13
...
28       7          0,1     70, 71
31       7          6,7     76, 77

这张发牌表由硬件固定，与 swizzle 无关；它定义了「算完之后每个值落在谁的寄存器」，恰好对齐 mma 期望的 fragment 布局。

swizzle 在 ldmatrix 里防的是第一步的冲突：8 个线程同时交出 8 个行起始地址。典型布局（行跨度为 128 字节的倍数）下，这 8 个行首周期性撞进同一个 bank-group。swizzle 把第 r 行整块挪到 bank-group $0\oplus r = r$：

                无 swizzle              有 swizzle
线程 t  取第几行  行内容    bank-group   bank-group (0⊕r)
0        r0       0..7      G0           G0
1        r1       10..17    G0           G1
2        r2       20..27    G0           G2
...                         全撞 G0       散到 G0..G7
冲突              8 路冲突                0 冲突

swizzle 的粒度等于访问宽度

ldmatrix 每线程从其地址连续抓 8 个元素送进寄存器。若 swizzle 打乱块内字节，这 8 个元素就不再是连续逻辑元素，fragment 即失效。实际的 swizzle（如下文 Swizzle<3,3,3> 的 base bits）只在「16 字节块之间」洗牌，块内顺序原封不动，因此每线程那 8 个连续元素始终正确对齐 MMA 期望布局。

因为行地址用 $\sigma$ 计算、读到的是正确行，硬件固定的发牌映射就把每个值送进正确寄存器。上表「寄存器拿到的值」在有无 swizzle 下完全一致，区别仅在于有 swizzle 时第一步快约 8 倍。

5. 真实 GEMM 里 swizzle 作用几次 · CUTLASS SM80 实证

本节回答 Q5。真实 GEMM 的数据流是 Global → Shared → Register，swizzle 在两个触碰 shared memory 的端点各被求值一次：G2S 写入与 S2R 读取。下面以 CUTLASS 官方教程 kernel 为据。

swizzle 只定义一次——它是 shared memory tile 的 layout 属性，见 sgemm_sm80.cu:358-363：

auto swizzle_atom = composition(Swizzle<3,3,3>{},
                                Layout<Shape <_8,Shape <_8, _8>>,
                                       Stride<_8,Stride<_1,_64>>>{});
auto sA = tile_to_shape(swizzle_atom, make_shape(bM, bK, bP));   // 带 swizzle 的 smem 布局
auto sB = tile_to_shape(swizzle_atom, make_shape(bN, bK, bP));

随后这个同一个 sA 被两个 copy 引用。G2S 写入端，partition_D（写目标）见 sgemm_sm80.cu:120，copy 见 L145-146：

Tensor tAsA = thr_copy_a.partition_D(sA);            // 写目标 = swizzled sA
// ...
copy(copy_a, tAgA(_,_,_,k_tile_next), tAsA(_,_,_,k_pipe));   // cp.async 落到 swizzled 位置

S2R 读取端，make_tiled_copy_A 见 L176，partition_S（读源）见 L178，copy（ldmatrix）见 L244-245：

TiledCopy s2r_copy_a = make_tiled_copy_A(s2r_atom_a, mma);
Tensor tXsA = s2r_thr_copy_a.partition_S(sA);        // 读源 = 同一个 swizzled sA
// ...
copy(s2r_atom_a, tXsA_p(_,_,Int<0>{}), tXrA(_,_,Int<0>{}));   // ldmatrix 从 swizzled 位置读

tAsA（G2S 写目标）与 tXsA（S2R 读源）都是同一个 swizzled sA 的 partition，因此两次都套用同一个 Swizzle<3,3,3>。

数据位置 是否 swizzle 依据
Global memory 否，自然 layout tAgA 来自 global tensor
Shared 写入地址（G2S） 是，Swizzle<3,3,3> partition_D(sA)
Shared 读取地址（S2R） 是，同一个 Swizzle<3,3,3> partition_S(sA)
Register（fragment） 否，MMA fragment 布局 tXrA
「swizzle 作用两次」的精确表述：一个 swizzle，在 G2S 写、S2R 读两个触碰 shared memory 的端点各被求值一次；Global 端与 Register 端不参与。两端引用同一个 layout 对象，写读一致由类型系统保证。

6. Hopper 与 Blackwell · descriptor 如何保证直读正确

本节回答 Q6。Hopper 的 wgmma 与 Blackwell 的 tcgen05.mma 常直接从 shared memory 读操作数，不经 ldmatrix。此时反 swizzle 由硬件完成，靠一个 descriptor 把 swizzle 模式声明给取数单元。

6.1 Hopper · GMMA descriptor

CUTLASS mma_sm90_desc.hpp:48-52 的 swizzle 枚举与 L80 起 的 64-bit descriptor：

enum class LayoutType : uint8_t {
  INTERLEAVE = 0,
  B128 = 1,    // SWIZZLE_128B
  B64  = 2,    // SWIZZLE_64B
  B32  = 3,    // SWIZZLE_32B
};

union GmmaDescriptor {
  struct {
    uint16_t start_address_      : 14, : 2;   // bit[0,14), 低 4 bit 不含 -> 16B 对齐
    uint16_t leading_byte_offset_: 14, : 2;   // LBO
    uint16_t stride_byte_offset_ : 14, : 2;   // SBO
    uint8_t  : 1, base_offset_   : 3,  : 4;   // bit[49,52), 仅 128B/64B 有效
    uint8_t  : 6, layout_type_   : 2;         // bit[62,64): swizzle 模式
  } bitfield;
};

start_address_ 在 L110，base_offset_ 在 L122，layout_type_ 在 L125。

保证机制：G2S 由 TMA 按 swizzled smem layout（Swizzle<3,3,3> = 128B）写入；S → Tensor Core 由 wgmma 拿 descriptor 直接取数，layout_type_ = B128 这 2 个 bit 告知张量核心「该 smem 用 128B swizzle 存放」，硬件结合 base_offset_（相位）与 LBO / SBO（走位）自行反推地址。这两端的 swizzle 模式并非各自填写——CuTe 的 make_gmma_desc 从同一个 smem layout 推导 layout_type_，写入用的 swizzle 与 descriptor 声明的 swizzle 同源（源码逐行勘验见 §7.3）。

wgmma 只接受 4 种 canonical swizzle

descriptor 仅能编码 NONE / 32B / 64B / 128B 四种模式，因此 smem 必须使用 GMMA::Layout_K_SW128_Atom 这类 canonical layout，不能用任意 XOR。这把「可表达的 swizzle」限制为「硬件可识别的 swizzle」，从源头排除写读不一致。

6.2 Blackwell · UMMA descriptor 与 TMEM

CUTLASS mma_sm100_desc.hpp:79-91 的 SM100（tcgen05 / UMMA）枚举多了一档，且 layout_type_ 扩到 3 bit（见 L98 起 的 SmemDescriptor 与 L111-112 的位段注释）：

enum class LayoutType : uint8_t {
  SWIZZLE_NONE         = 0,
  SWIZZLE_128B_BASE32B = 1,   // Blackwell 新增
  SWIZZLE_128B         = 2,
  SWIZZLE_64B          = 4,
  SWIZZLE_32B          = 6,
};
// SmemDescriptor: layout_type_ 占 bit[61,64) —— 3 bit(Hopper 为 2 bit @[62,64))
//                 另新增 lbo_mode_ @bit[52,53)

此外 SM100 多出一个独立的 32-bit InstrDescriptor（构建函数见 mma_sm100_desc.hpp:477 起），携带 a_format_ / b_format_ / c_format_、m_dim_ / n_dim_、a_major_ / b_major_、sparse_flag_ 等；其 InstrDescriptorBlockScaled 变体带独立 scale-factor ID，供 FP4 / FP8 block-scale 使用。

最大的架构差异是输出位置：tcgen05.mma 的累加器在 Tensor Memory（TMEM），操作数 A 可来自 TMEM 或 SMEM、B 来自 SMEM。因此对来自 SMEM 的操作数，swizzle 保证与 Hopper 一致（SmemDescriptor.layout_type_ 声明、CuTe 从 smem layout 推导、硬件按它反推）；对来自 TMEM 的操作数则不经 shared，也就不涉及 swizzle，由 tcgen05 的 TMEM layout 自行管理。

6.3 三代对照

架构 反 swizzle 由谁执行 swizzle 模式存放 MMA 输出落点
Ampere / Ada 软件（ldmatrix 地址 = addr_swz） kernel 代码中的 XOR 寄存器
Hopper 硬件（wgmma 取数单元） GMMA descriptor layout_type_（2 bit） 寄存器（累加器）
Blackwell 硬件（tcgen05 取数单元） UMMA SmemDescriptor.layout_type_（3 bit，多 128B_BASE32B） TMEM
贯穿三代的不变量：swizzle 模式始终是 shared memory 这一层的单一声明，由 CuTe 从 smem layout 推导，写入端（TMA / cp.async）与读取端（ldmatrix / wgmma / tcgen05）引用同一份，因此写读永远一致。代际间变化的只是「谁执行反 swizzle」（软件 → 硬件）与「操作数 / 输出落点」（寄存器 → TMEM）。

7. CUTLASS 源码勘验 · swizzle 的单一真相如何被代码强制

前文的论断——「swizzle 是 shared memory 层的单一声明，写读两端引用同一份」——不是设计意图的转述，而是可在 CUTLASS 源码里逐行验证的事实。本节落到 include/cute 的真实实现。

7.1 Swizzle 本体就是一行 XOR

Swizzle<B,M,S> 的核心方法 apply 见 swizzle.hpp:55-78，与 §1 手写的 addr_swz 完全同构：

template <int BBits, int MBase, int SShift = MBase>
struct Swizzle {
  using bit_msk = cute::constant<int, (1 << BBits) - 1>;
  using yyy_msk = cute::constant<int, bit_msk{} << (MBase + max(0, SShift))>;
  using msk_sft = cute::constant<int, SShift>;

  template <class Offset>
  CUTE_HOST_DEVICE constexpr static auto apply(Offset const& offset) {
    return offset ^ shiftr(offset & yyy_msk{}, msk_sft{});   // ZZZ ^= YYY
  }
};

三个模板参数的语义：B = 参与 XOR 的位数（决定 swizzle 强度），M = base，即最低被保护、永不参与 XOR 的位数（正是 §4 的「访问粒度」），S = 选择位段与目标位段的距离。apply 取高位段（yyy_msk）右移 S 后异或进低位段——一次 XOR，与架构无关。

7.2 GMMA canonical 布局 atom：swizzle 强度即模板参数

Hopper GMMA 不接受任意 swizzle，只接受四种 canonical 布局，定义见 mma_traits_sm90_gmma.hpp:78-104：

// 位级布局（element 级 atom 由 upcast<sizeof_bits<T>> 得到）
using Layout_K_INTER_Atom_Bits = ComposedLayout<Swizzle<0,4,3>, smem_ptr_flag, Layout<Shape<_8, _128>,Stride< _128,_1>>>;
using Layout_K_SW32_Atom_Bits  = ComposedLayout<Swizzle<1,4,3>, smem_ptr_flag, Layout<Shape<_8, _256>,Stride< _256,_1>>>;
using Layout_K_SW64_Atom_Bits  = ComposedLayout<Swizzle<2,4,3>, smem_ptr_flag, Layout<Shape<_8, _512>,Stride< _512,_1>>>;
using Layout_K_SW128_Atom_Bits = ComposedLayout<Swizzle<3,4,3>, smem_ptr_flag, Layout<Shape<_8,_1024>,Stride<_1024,_1>>>;

swizzle 强度完全由 Swizzle 的第一个参数 B 表达，M=4、S=3 固定不变：

canonical atom swizzle B 字节周期 含义
Layout_K_INTER_Atom Swizzle<0,4,3> 0 无 不 swizzle（interleave）
Layout_K_SW32_Atom Swizzle<1,4,3> 1 32 B 4 路残余冲突
Layout_K_SW64_Atom Swizzle<2,4,3> 2 64 B 2 路残余冲突
Layout_K_SW128_Atom Swizzle<3,4,3> 3 128 B 0 冲突

§5 教程 kernel 用的 Swizzle<3,3,3> 是对 4 字节 float 在 element 级的等价写法；GMMA atom 在 bit 级统一为 Swizzle<3,4,3>，M=4 对应 16-bit 的 base 保护。两者描述同一个 128 B swizzle，仅计量单位不同。

7.3 单一真相的机械证明：layout_type() + make_gmma_desc

「写端 swizzle 与读端 descriptor 同源」并非约定，而是 make_gmma_desc 从 smem layout 反读 swizzle 写进 descriptor 的直接结果。

第一步，layout_type() 把布局里的 Swizzle<B,4,3> 解出强度 B，见 mma_traits_sm90_gmma.hpp:130-152：

using Swizzle = get_swizzle_t<Engine>;          // 从 layout 取出 swizzle
constexpr int B = Swizzle::num_bits;
static_assert(M == 4, "Unsupported layout swizzle");
static_assert(S == 3, "Unsupported layout swizzle");
switch (B) {
  case 0: return LayoutType::INTERLEAVE;
  case 1: return LayoutType::B32;
  case 2: return LayoutType::B64;
  case 3: return LayoutType::B128;
}

第二步，make_gmma_desc 把这个 LayoutType 直接写进 descriptor 的 layout_type_ 位段，见 mma_traits_sm90_gmma.hpp:199-300：

GmmaDescriptor desc;
constexpr LayoutType LAYOUT_TYPE = layout_type(u128_tensor);   // 从同一个 smem tensor 解出
desc.bitfield.layout_type_ = uint8_t(LAYOUT_TYPE);             // 写进 descriptor
desc.bitfield.start_address_ = start_address >> 4;
// stride/leading byte offset 也从该 layout 的 canonical strides 推出:
desc.bitfield.stride_byte_offset_  = stride_01;
desc.bitfield.leading_byte_offset_ = stride_10;

函数内的注释给出了被强制的 canonical 形态，例如 K-major B128 为 Swizzle<3,4,3> o smem_ptr o ((8,n),2):((8,SBO),1)——descriptor 的 LBO / SBO 即由此 layout 的 stride 推出，不是手填。

为什么这就排除了写读不一致

G2S（TMA）与 S→TensorCore（wgmma）拿到的是同一个 sA tensor。前者的 TMA descriptor swizzle 字段、后者的 GMMA descriptor layout_type_，都由 make_*_desc 从这同一个 layout 的 Swizzle<B,4,3> 反读得到。两端不存在「各自填写 swizzle」的机会，因此结构上无法不一致——static_assert(M==4 && S==3) 还在编译期拒绝任何非 canonical swizzle。

7.4 Blackwell · make_umma_desc 是同一套机制

SM100（tcgen05 / UMMA）沿用完全相同的「layout 反读 → 写 descriptor」结构，见 mma_traits_sm100.hpp:206-225：

SmemDescriptor desc;
desc.version_  = 1;     // Blackwell 版本标记
desc.lbo_mode_ = 0;     // 默认 legacy LBO 模式
constexpr UMMA::LayoutType LAYOUT_TYPE = UMMA::layout_type(u128_tensor);   // 同样从 layout 解出
desc.layout_type_ = uint8_t(LAYOUT_TYPE);                                  // 写进 3-bit 位段

差异仅在枚举多一档，且新增 Layout_MN_SW128_32B_Atom = Swizzle<2,5,2>（即 SWIZZLE_128B_BASE32B），见 mma_traits_sm100.hpp:72 与 L175 的 canonical 注释 128B_BASE32B : Swizzle<2,5,2> o smem_ptr o ((T,8,m),(4,k))。机制不变：smem 操作数的 swizzle 由 layout 单一决定、由 make_umma_desc 写进 descriptor；TMEM 操作数不经 shared，不涉及 swizzle。

源码勘验结论：`Swizzle::apply` 是一行 XOR；GMMA / UMMA 只接受 `Swizzle`（SM100 另加 `Swizzle<2,5,2>`）四到五种 canonical 强度；`make_gmma_desc` / `make_umma_desc` 从同一个 smem layout 反读 `B` 写进 descriptor 的 `layout_type_`。「写读同源」由这条 layout→descriptor 通路在编译期机械保证，而非运行期约定。

附 · 为什么 FP8 强制 tile 宽度不小于 32 元素

ThunderKittens 的 FP8 实现要求最小 tile 宽度为 32 元素。原因正是 §1 不变量的对齐前提：swizzle 按固定字节周期（32 / 64 / 128 B）异或，$\sigma$ 要成为 tile 内的合法置换，一行须正好铺满整数个 swizzle 周期。FP8 每元素 1 字节，128 B 周期等于 128 个元素，是 bf16（2 字节 → 64 元素）的两倍。若沿用 bf16 的 16 元素窄 tile，FP8 下一行仅 16 字节，装不下完整周期，$\sigma$ 在 tile 内不再是置换——会把元素搬出边界或令两元素撞同槽，不变量随之失效。将最小宽度提到 32，即是守护「$\sigma$ 写读一致且合法」这一唯一前提。
